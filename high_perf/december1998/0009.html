So this is about loads. I will try to find it in the manual, too. How
about stores ?? "Scheduling and issuing rules" in the 21164 manual only
gives some *static* rules here, for ex. that loads cannot be issued
together with stores or on certain cycles relative to other
instructions, however, the section on *dynamic* rules simply says:
"stores have no result", hence no latency. This clearly cannot be true
in real life. If the memory subsystem is busy, at some point the
execution must stop. Clearly, there will be some "write buffers" helping
here, but how many ??? Anybody knows that ?? And is there a fast bypass
if loads to the same datum follow shortly after the respective store ? I
aske because I think to have read about that in the Alpha-Linux FAQ
(about the different types of Alpha): As the Alphas < 21264 have no
direct fp to integer register move, all those move have to go through
memory (You will know that). However, the above FAQ has a note in it
that there is/mght be a special bypass to speed up such moves. Maybe it
simply the cache coming in here, so you must be able to read back the
stored value from write buffer or fast L1 cache, even if L2/L3 or DRAM
is still busy and unavailable.
