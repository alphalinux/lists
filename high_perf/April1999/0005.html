Thanks Joe! I don't know how much difference it would make for most apps,
but as far as RPM features go, ease of installation of is only the
beginning... We could put up other RPMs on ftp.alphalinux.org that depend on
fastmath, such as perhaps spreadsheets- or Mesa, where I imagine faster sqrt,
sin and cos would help a lot. OTOH, CPML is slightly faster, and might be
the preferred dependency there. But then again, vectorized sqrt, sin and cos
could make a big difference in Mesa- we'd just have to provide patches to
them, and have them added to the official Mesa tree. [Of course, programs
using Mesa would see no advantage unless they link against ffm- and would be
required to link against it if Mesa included ffm-specific calls. Libtool
seems to be able to handle this though- see the info page, on inter-library
dependencies.] Does anyone have some free time on their hands to hack Mesa?

To fantasize a little further, there could be a standard cross-platform
fastmath API whose implementation uses libffm on Alpha, 3D-Now! on AMD,
Katmai on newer Intel, and it would be a dependency for a lot of different
packages- sort of like the BLAS/LAPACK are a cross-platform matrix API with
different implementations on different architectures. But this type of
proposal may exist already, and if not, is just a little bit beyond the scope
of this list.

For now, since it seems it will work for 21064/21066, we can probably put up
a binary fastmath RPM and see what people do with it...

One pointer: Joe, are you familiar with the GNU shared library versioning
conventions? If not, it might be worth looking into the libtool info page,
in the section on "Versioning". The convention is somewhat different from
other software, and is important because it clearly indicates changes in
library APIs. For example, although the gnome-libs package is at version
1.0.8, libgnome is at 32.3.2, libgnomeui at 32.9.0, libgnorba at 27.1.5, etc.
so an older software package can link with the the most recent older library
version that has the correct API, or otherwise fail to run if it can't find
any with the right API. That was a convoluted explanation, but I think it
might be worth numbering the shared library object using this convention
(e.g. libffm.so.4.7.3), even if it differs from the package version number
(e.g. 0.30).
