On Mon, 6 Dec 1999, Paul B. Brown wrote:

>
> > 1: The new PIII (copperbug) does run faster (and show a lot better
> > SPECint and SPECfp), but when translated to realworld applications
> > (atleast ours) the difference isn´t that significant (since we can't
> > get a hold of Intels internal compiler among other things).
>
> Hummm . . . So again, if code is developed with the Alpha in mind, the
> application will be faster than the same application coded specifically
> for the Intel. OK . . . .

Usually, the compilers make a big difference, if you're benchmarking an
application be sure to check out the Compaq Tru64 Compilers that are being
ported over to Linux. In certain cases these can have a large effect on
performance (the compilers and the libraries together).
 
> > 6: Limited memory bandwith on Intel hardware (at least compared to
> > Alphas)
>
> Is this the 32-bit -vs- 64-bit architecture argument?

No, this is memory bandwidth of the chipset. If you look at something like
a DP264 (DS20) it has a theoretical memory bandwidth of 5.2Gig/second
(accross 2 memory buses), I don't think Intel has anything close to that.
It's more a function of the memory bus being 256Bits wide.

> > something that was a totaly braindead idea.
>
> I agree with this one. I thought Alpha took a header for a short while

> > 8: Alphas have advanced quite extensivly since the takeover with
> > Proc´s (EV6, EV67 and the upcoming EV68) and tech (wildfire), this
> > coupled with the API push downward (price and spec´s).

> EV68, huh? I'll have to look into this one.

microns, EV67 is .25 and EV68 should be .18 (if I remember everything
correctly)
