> I have a PC164 with a flaky floppy controller. I'm using a Buslogic
> scsi card, which the ARC console firmware can't drive, so I must boot
> MILO from floppy.
> [ ... ]
>
> So, with considerable trepidation, I built a fmu.gz with the same

> kernel vanilla 2.0.30 plus alpha-patches from gatekeeper). I flashed
> MILO. There did not seem to be a debug monitor there -- block 0 is
> empty, WNT starting at block 1, MILO starting at block 7 (I think).

> I set bootopt to 130, for the second flash image. It booted to

I went through the same process of trying to flash milo using
fmu.gz. It doesn't work, because:

On the pc164, the SROM (initial boot rom) at boot time does not
look for a "bootopt" number so milo located in the second or
third position cannot be run.

One might expect milo written over WinNT ARC monitor would run in
position 2 when the jumper is set to boot WinNT ARC, however I
tried this and it didn't work. Either the SROM looks
specifically for "WinNT ARC" in the rom image that it's trying to
boot or there's something about milo that prevents it from
running correctly in place of ARC. So I'm still using WinNT ARC
monitor to boot milo from a small DOS partition. Evidently DEC
made a marketing decision not to have SROM check "bootopt" for
the PC164. But booting the third image through "bootopt" works
fine with the EB164...

Also, note that though it is fairly easy to restore an
overwritten ARC console monitor starting at block 1 (using the
"fail safe boot loader" and floppy disk), block 0 is not empty
and contains the "fail safe boot loader", and if you erase this
block with MILO it is much more difficult to get it back (need
software from DEC and special cable).

It would be nice to have a MILO that would boot in place of ARC,
(i.e. ignoring bootopt) but so far this hasn't worked for me.
