Thanks to everyone for thier help and advice on using the Zip
drive for a rescue disk. I have it working wonderfully for my Linux/Intel
boxes, off of my Parallel Port drive with a kernel floppy.
I also got the Zip drive working with my UDB. The newest driver
for ppa that I grabbed off the net autodetected the zip drive even on the
alpha! Though that driver is quite a bit newer than the ones found in the
2.0.30 kenerl.
Though when I went to install RedHat AXP onto the zip disk from
the Alpha, I ended up hitting a snag. First of all, in order to get the
installation system to recongize the ppa zip disk, I had to compile a new
alpha kernel, place it on a floppy and use it as the boot disk in place of
RedHat's boot disk. Then I used RedHat's root disk as usual. Everything
seemed to go just fine, the install process never realized, or at least
didn't care, I had switched kernels on it. I was able to see the zip drive
as a SCSI disk, and the setup program recongized it as such.
The problem occured when I tried to fdisk the zip drive from the
Alpha. When I tried to print the parition table, it prinited the first
parition on the zip disk (the Linux/Intel rescue part), but there it
froze, and so did the entire system. After rebooting, and loading my
standard installation off of the hard disk as normal, I tried to run fdisk
on the zip disk again. Same behavior as last time, only it really locked
the machine up. The only response I could get from it at the console or
across the network was pings. It was dead as a doornail otherwise. After
rebooting again, I tried running fdisk and printing the parition table for
the internal 340mb SCSI drive, and it printed my initial FAT parition, the
swap parition, and then once again a solid lock up, with the main linux
parition still to print.
I also tried in the setup program to not fdisk any disks, and
proceed with them as they were (after I had them set up right on the
zip disk off of my Linux/Intel box). But then on the part of "Initializing
filesystems" I think was the name, where one chooses what will be mounted
where, and what parition is the root filesystem, the entire system
rebooted, with no warning!
It seems there is something amiss with fdisk. The only thing I can
trace this to, as it was working just fine recently, is to a new kernel I
just installed on the system, and then used in the setup process. It is a
stock 2.0.30 kernel, with the alpha-patches-2.0.30-0.1 applied that I
pulled off azstarnet. After that I have nothing strange compiled in the
kernel, except for the new drivers for the ppa that I pulled off the net.
Otherwise the system has been running fine, with fdisk as the only
problem that I have hit yet.
Sorry for the long post, but this has got me stumped. Does any one
know how I managed to break fdisk here? Is it due to the alpha-patches or
due to the ppa drivers, since those are the only two new variables
introduced since the last time fdisk worked. Thanks for any help!
