Oren Teich wrote:
> I've compiled my own copy of milo ( we want to hack on it, but first I
> figured I should get the plain vanilla copy running ), version 2.0.30
> from decs site. Compiled fine against the 2.0.30 kernel, with the
> patches from the dec sight applied. During boot, however, it freezes
> after
>
> Swapping to PALcode at 0x0 ...

I don't know what you mean by the elipses,
but if that is address zero (0x00000) [all zeros],
you need to check that the PALCODE_AT variable in
the milo is being used. (and make sure the PALcode
is actually put there)

It would also help to know what architecture you're on.
Myself, I've only been working on a pyxis-based motherboard,
so my value for PALCODE_AT = 0xfffffc0000200000

The 0xfffffc part just orders pyxis to do kernel-direct address
mapping to physical memory. So the palcode would be located
in physical memory at 0x00200000.

I've found that MILO hacking is a royal pain.
Our head software engineer had to write a utility
to manually copy osfpal into memory
(notice, not the compressed version; which micropal then decompresses)

like I said, a royal pain.
but fun hacking, nevertheless!
