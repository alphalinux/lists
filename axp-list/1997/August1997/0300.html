Hello Robert

> We just purchased an Aspen, Durango system (PC164 though
>/proc/cpuinfo says EB164) 433MhZ. Thus far, installing RedHat on it has
>been quite an fest. Just now, I got two kernel unaligned accesses so I
>decided it was time to post here and see what you guys think. I am
>almost beginning to wonder if I have a bad motherboard.
>
>Setup:
> Aspen Durango, 433MhZ EV5, 1MB L3, 64MB RAM (128 bit access).
>The motherboard should be a PC164 but the info in /proc/cpuinfo says
>EB164. ARC Firmware 4.49-4. Redhat 4.2 Ftp'd from RedHat. Updated
>RPMs as of Aug 8. Whole installation sitting on another Alpha running
>D.U. for a NFS based install. Two Quantum Fireball EIDE disks (6GB). A
>#9 GXE based video card. Installation tried with the PC164 kernel and
>milo. Name info placed on my bootp server.

The ARC firmware should be 4.49-7 thats the latest and greatest. All the
hardware looks good with one exception, the IDE drives.

>Problems:
>
>1. Milo would boot and then hang writing garbage to the screen. This
>was with loadlin 1.5 and the stock milo. Swapping the video card with
>an ATI Mach 64 as per info on the mailing list archives worked.
>
>2. The RedHat install couldn't do the DNS resolution to find the NFS
>server by name. I used the IP address instead. Petty annoyance.
>
>3. (This one was REALLY annoying) The RedHat setup could only mount one
>partition of less than 128MB. My original plan was to have a 10MB milo
>FAT for booting (hda1), 48MB / (hda2), 128MB swap (hda3), 48MB /var
>(hda5), 512MB /usr (hda6) and the rest partitioned into 2GB and 4GB
>filesystems. Trial and error till I found out it was able to deal with
>128MB / (hda2) and 128MB swap (hda3). I tried the EB164 milo/kernel
>disks with worse results. I installed a minimal OS and made the /var
>and /usr partitions by hand later. After merging them in, I installed
>the rest of the RPMs I was interested in. After some heavy tweaking I
>had a somewhat usable Linux box.
>
>4. Time to build milo and the kernel. The kernel compile from the RPM
>sources fails with several missing header files. This is most likely
>pilot error and missing RPMs as a result of #3.
>
>5. Kernel unaligned accesses on swap requests after a certain point.
>The system was swapping beforehand and this started in the middle of the
>failed kernel compile. Typing "sync" was a sure method to get the error
>after it started. I ran badblocks to make the swap so it is unlikely
>that there is a bad section of the swap area.
>
>6. Petty annoyance, sometimes milo hangs on reboots.
>
>7. Kernel is panicky!

My best guess is that its the IDE drives. We've tried 2Gb Quantums
Fireballs and larger drives with some success. However, its a major pain in
finding a partition table that works solid. My tech spent quite some time
trying to get it to run for a customer that was insistent on using his IDE
drives. It can be done but it takes some magor tweaking on the partition table.

One point I know we had is we couldn't go over 2Gb in a partition or you
experinced what you mentioned above. The bottom line it wasn't reliable. My
only suggestion is move to SCSI.

Hope this helps.
