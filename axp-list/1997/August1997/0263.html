We recently bought an Alpha 21164 based machine at the German
computer discounter Vobis (`Highscreen Alpha 5000').

We replaced the preinstalled Windows NT by Redhat Linux 4.2, but
encountered several problems with this system.

Hardware:
21164/500 MHz Alpha chip,
DEC PC164 motherboard,
256 MByte RAM,
4 GByte EIDE harddisk,
Winner 2000 AVI/3D graphics card (S3 Virge VX R2 chip).

OS:
Redhat Linux 4.2,
Kernel 2.0.31 (pre-patch 3)
XFree86 3.3 (SVGA/S3V)

Problems:

1) XFree86 3.3
The XF86_S3V server is very unstable and causes system crashes.
The XF86_SVGA server is nearly stable but dies sometimes, when
switching between virtual console and X.
Additionaly, the SVGA server doesn't start up in most cases
(screen remains black, it is possible to kill the server with
Ctrl-Alt-Backspace).
A fix for this problem is to start XF86_S3V first for a short time.
Afterwards the graphics card is initialized and XF86_SVGA
can be started and runs nearly stable.

2) f2c causes spurious floating point errors during translation
This happens with f2c provided by Redhat 4.2, or with a self-compiled
current version of f2c.
A workaround for this problem is to catch the SIGFPE signal,
the produced C code seems to be O.K.
Additionally, we changed the `long int' types in `f2c.h' to `int'.
Without these changes, compiled FORTRAN programs cause also spurious
floating point errors.

This is not only a problem of f2c, we observed it also with
gnuplot and fig2dev.
Maybe the reason for this behaviour is the assignment of 64 bit
pointers to 32bit integers (can you comment on this?)

3) `clock' command shifts system time by 20 years
After booting, the system date is shifted by 20 years (to the year 2017)
time, day of month and month are O.K. in most cases.
Sometimes even time and date are totally wrong.
This seems to be caused by a bug in the `clock' command.

Can you comment on these problems?
Are there bug fixes (especially for the S3 Virge server)?

Thanks,

Stefan
