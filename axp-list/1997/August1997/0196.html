we are having difficulty with our new alpha based system which is
intended to run linux. following is the system configuration:

DEC Alpha 21164A-500MHz CPU
PCI bus
IBM 4GB Ultra Wide SCSI 34330
Diamond Stealth64 S3-chip Trio64 PCI-2MB video card
Redhat linux 4.2
*NCR scsi controller model identified as ncr53c810

*indicates what we believe to be the cause of our problems

following description of problem:
we received the machine and started the linux os in the console mode.
then as any new user might attempt we tried to start the x environment
while logged on as root.
#startx - this is what we typed
it locked up the system and we felt forced to hit the reset button.
this turned out to be a more serious problem. while rebooting the
system noted that the filesystem had not been unmounted cleanly and
forced a file system check. when this happened we got something like:
"
oscript cmd=89030000
oreg.o da 10 80 13 47 08 00 1f 03 01 80 21 00 01 71 00
ncr53c810-o: have to clear fifos
ncr53c810-o: restart (fatal error)
ncr53c810-o - <target 0, lun 0>:
command failed (9ff)
@fffc000068a820
"
****then this sequence is repeated forever or until we got sick!!! of
looking at it***

our filesystem looked something like:
/dev/sda1 dos partition holding necessary files
/dev/sda2 root partition i.e. / ***this is where we got the failure
/dev/sda3 swap partition i.e. linux swap
/dev/sda4 /usr/users partition i.e. linux native like root

we tried the following:
1. left with no other choice we tried to reinstall linux from cd media
provided by the mfg.
2. at first few attempts this failed because it could not format the
partitions or from what we could see just hung there
3. we finally did get the os to install once for an unknown reason (yes
we really do mean "unknown" since the only thing we did differently was
to turn the machine off for a brief i.e. 5min period and this doesn't
seem relevant)
4. when we overcame our state of joy and excitement we once again tried
to get x env. to start up under a root login and it hung up. after many
explicatives uttered by one of the chinese students and myself we once
again pressed the reset button and the same process began over again

** when i say the same process i am referring to the sequence of lines i
typed earlier which included the "ncr53c810-o" reference**

finally a summary question:
we are wondering if all of our trouble stems from the use of this
particular ncr controller. we would not have any problem approaching
our supplier for another card is this can be identified as the problem
as power outages may very well lead to this same situation, and needless
to say we can't afford to reinstall the machine each time. that is if
we can even get the thing to install consistently for us. has anyone
else experienced problems with this scsi controller and alpha
motherboard with linux installed?

we did check the web and this list also for past problems and others
have reported strange problems when it came to the ncr scsi controllers.

thanks for any assistance,
