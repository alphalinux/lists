On Sun, 16 Mar 1997, David Mosberger-Tang wrote:

> Some of you reported problems with profiling under RH4.1. Here is a
> uuencoded file of three fixed object files. The md5sum output is:
> [...]
> After extracting these files, replace them in your libc_p.a. E.g.:
>
> ar rv /usr/lib/libc_p.a gmon.po ieee_{s,g}et_fp_control.po
>
> and then profiling should work again. Note that the binutils-2.7
> gprof has a bug that sometimes causes a SIGFPE. See my earlier mail
> for a patch that fixes this.
>
> Finallly, there is still a problem with mixing profiling and dynamic
> linking. E.g., with dynamic linking, adding -lm will cause a binary
> that segfaults. Adding the -static flag should fix this problem for
> now (until a real solution is found).

Thank you for your effort to make profiling work on the old glibc.
However, I can't get profiling to work. With your fixes the binary won't
segfault anymore when started, instead it goes into some eternal loop
mode. This happends before the binary gets to main() (break main; run in
gdb doesn't get to the breakpoint. Also gdb segfaults if I try to
interrupt the process :( ). I belive I have pretty much tried the
standard 4.1 rpms when doing this. I normally use a newer binutils.

Here is what doesn't work for me:

--------------hello.c-----------------
int main()
{
printf ("Hello world!\n");
}
-------------------------------------
gcc -static -o hello -pg hello.c
./hello

I would really like profiling to work, I feel somewhat handicapped without
it!

I would appreciate any suggestions.
