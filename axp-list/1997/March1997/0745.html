*sigh*

As reported earlier by someone, I now have a reproducable way to make
my alpha kill our entire ethernet.

I just upgraded my alpha (UDB/Multia/linux 2.0.18).
It was running 4.1, and I've ftped anbd rpm'ed everything from the
updates directory. To my great disappointment, this has only made
matters worse :(

1)... reboot/halt/init 6 doesn't work anymore. It claims to do
reboot or shutdown, but never does. You can only turn off the
machine, resulting in filesystem errors :(
2)... turning on the networking still doesn't work as expected. The
ethernet gets configured as 0.0.0.0. No routes are configured.
3)... When I set the eth0 (class C network, with default router
on the eth0 segment) manually it works, however when I start
anything that results in a namelookup, i can see the LED of our
router go berserk, and the entire network gets overloaded.
I can't even run tcpdump on the remote (nameserver) host anymore,
because it cannot handle the truckload of packets. (Or the packets
are corrupt).
4)... When I removed the ethernet (BNC) connector to safe our office
network, and typed 'sync' on the alpha, it locked up solid.
This is probably due to having my external scsi cabinet at
home, and having two moint points listed in /etc/fstab. (ie:
/dev/sdb[12] isn't there while it is in the fstab).
5)... killall -9 amd doesn't even nuke amd when it's running.
6)... There is no modutils package installed. This might have been
stupidity from my site by forgetting to install the rpm containing
it. But without these, you cannot build a kernel.

If RedHat 4.0 was still on the mirror sites, I'd probably install it
instead of 4.1. 4.1 is really not stable for me at all. At least with
4.0 I just had a few scsi kernel panics :(

Has anyone found out the problems here? Is it just a matter of
nuking amd? I've heard people blame amd for lots of networking problems,
but is that relaly the culprit?

Paul
