Hi Franklin,

Thanks for your earlier reply to my question about the flash usage on
the PC164. It would still be convenient to have a way to software select
the flash image to load. For example, I have the fail safe loader, followed
by the minidebugger, followed by milo. These all easily fit, but the
motherboard jumper allows only the selection of the failsafe loader or the
minidebugger. It turns out the fail safe loader counts as one of the
images. So, what I do is boot the minidebugger, then romload milo, then
jump to milo. But all this is not the point --- the minidebugger doesn't
seem to work well on the PC164. My new question is: is there a later
version than what's on the Ver 2.1 CDROM? Anyway, here are the problems
I'm observing with the minidebugger.

1) fldir works OK to view the contents of a diskette, but flload to read
a file always fails with an I/O error. I have the same behaviour with many
different diskettes, and my floppy drive always works under linux.

2) The PCI devices are not initialized properly. I get lots of invalid
PCI configuration messages when I boot milo through the minidebugger. These
messages never happened when I booted through the NT ARC loader. These
messages
look like:

bios32 WARNING: slot 1, function 0 requests -2147483520B of contiguous
address
space---don't use sparse memory accesses on this device!!
bios32 WARNING: slot 0, function 0 requests -2147483520B of contiguous
address
space---don't use sparse memory accesses on this device!!
bios32 WARNING: slot 0, function 0 requests -2147483520B of contiguous
address
space---don't use sparse memory accesses on this device!!
bios32 WARNING: slot 0, function 0 requests -2147483520B of contiguous
address
space---don't use sparse memory accesses on this device!!
bios32 WARNING: slot 0, function 0 requests -2147483520B of contiguous
address
space---don't use sparse memory accesses on this device!!

I also get the message:

PPA unable to initialize controller at 0x378.

Again, none of this happens when I use ARC to load milo.

3) After I romload milo from the flash into ram, then I execute jtopal to
the load address, the system usually hangs. However, I have discovered that
if I disassemble the first few instructions of the milo image before the
jtopal, it usually works. Perhaps this is because some processor caches
are being flushed as the disassemble command is being executed. To make this
explicit, the following hangs 9 out of 10 times:

romload linux ; loads milo into address 0x200000
jtopal 200000 ; start milo

But the following usually works:

romload linux
dis 200000 ; disassemble a few instructions
jtopal 200000

My system: PC 164 433 MHz, SROM version 2.1
EBSDK info: Version 2.1, built on Digital Unix Aug. 23, 1996
I think the BIOS emulation code prints out: V2.02ebsdk

Thanks,
