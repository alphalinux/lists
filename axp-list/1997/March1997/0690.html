>At 10:33 AM 03-24-97 -0500, you wrote:
>> I came in to the UDB game kind of late and would like a pointer to
>>the 2017 problem/solution. I am running RH4.0 and it seems that the
>>boot causes the machine to initialize to 2017. Sorry to ask for a
>>solution, but I didn't see it in the FAQ.
>>
>This is a standard Red Hat install problem. Unfortunately the things needed
>to read the clock from the ARC firmware aren't initialized until _after_
>the install, leaving tons of files laying around with 2017 dates.
>
I am running 4.1 on a 2/400 and you cannot leave the firmware with a 1997
date on the real time clock. You get a message saying that Windows/NT wont
boot with the time you have on the clock and it is a mandatory fix. Anyway
I dont have NT so I figure no problem but to the formware all operating
systems are apparently NT so I have to boot with 2017 then do a date
command to fix things up.

The solution I am implementing is to edit out the calls to the ' clock '
command in the rc files for bootup. This will eliminate the reference to
the real time clock completely. Of course this will also probably make the
system use the time from when the root file system was unmounted as the
'correct' time.

Then I am going to get the time off the newtork with ntp or timed to use in
place of the real time clock.

There is supposed to be an offset in the settings file for the clock
program. Has anyone tried setting this to -631238400 which is the number of
seconds that the real time clock is fast. And I was never satisfied with
quartz accuracy of less than 1 second a day, now this.

Maybe this is the NT fix to the year 2000 problem. Just pop the machine
into next century and put it off for 70 more years...
