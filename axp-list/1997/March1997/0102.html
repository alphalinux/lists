Well, here's a data point for you (not a PC164, but feels similar):

XLT-300, TGA, Qlogic SCSI, TULIP, running RH 4.1 plus 2.0.29+axp-diffs-2.0.21.
(I'll try to remember to include the /proc/pci at the end). MILO 2.0.18, so
no LATENCY mods.

Boots up fine (at the moment), login as root, "startx"; then I get Oopses
from any number of sources, but mostly kernel data sensitive places, like:

free_pages
get_hash_table /* very popular :-( */

Sometimes, I've even seen it caused by get_sysinfo, trying to get the percpu
info, when the processor_offset field appears to be bogus, so the resulting
address (hwrpb+hwrpb->processor_offset) causes (sometimes first an unaligned)
the Oops.

Somebody somewhere is scribbling where they ought not... :-(

I've got this instrumented with KGDB, and can do whatever is appropriate. I
don't have a clue as to where to start, as it's not predictable what will go
wrong first, and as you have said, setting the breakpoint within an
interrupt can sometimes cause heartburn... :-)

Thanks for *any* suggestions where to start looking...
