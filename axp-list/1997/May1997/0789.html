On May 27, David Dawes wrote:

> 3.2Z is available.
>
> This version identifies itself as 3.3, and should be used for building
> the binary distributions, but it is tagged as 3.2Z to allow for minor
> source updates before the actual release is made publicly available.
> For a change to be accepted at this point, it must be either a very
> serious bug, or a platform-specific build problem.

I know you will hate me, but...

the last think I did before leaving home was to use a ViRGE/VX
card with Linux/AXP which showed two problems with packed 24bpp:

- it's *awfully* slow in s3im.c because all this 24<->32 bpp copying
is completely unaligned. just starting X11 with give ~1.5M unaligned
access traps which results in many seconds needed for root weave
or for display save/restore when switching consoles.

- using 1024x768x24 with 1280x1024 (or similar) virtual there is
a problem with panning (s3Adjust()):
when panning the display horizonally one step, colors are wrong.
I think this is a result of a displayed framebuffer starting address
not being a multiple of 3.

I guess everyone with a ViRGE/VX card should be able to see/fix this ?!
could anyone test/check/fix this one ??
are there no problems with non-/VX cards with panning?
might be a result of different HorizontalScale values
(or whatever they are called; no source online here...)

I'll back at home somewhen after 11pm (MESZ) and I try to write
a faster 24->32 bit and 32->24 bit copying routines (if not anyone
else is faster, hint hint;)

since this this alignment stuff is specific to Linux/AXP and XF86_S3V
I hope we can add this fix (if I manage to fix it) and just build
this one binary later or again...

Harald
