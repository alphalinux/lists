Ben Scott writes:

Gave SRM commands "set boot_defdev dka100.1.0.6.0", "set boot_file
1/vmlinux.gz", "set boot_osflags root=/dev/sdb1", and finally, "boot dka1".
The SRM found a valid boot block, loaded it, and booted it. aboot put up
a small banner and loaded the kernel. The kernel started to boot.

That is where things got funky. The kernel boot process seems much slower
then when under MILO. It tends to stop and think for a couple seconds on
the serial driver init, for example. Under MILO it was nearly instant.
Then, when the kernel reaches the SCSI driver (BSD NCR SCSI port, 1.18f),
it blows up completely. The driver keeps giving "Aborted command due to
timeout" errors and something about "Test Unit Ready".

Does anyone have any idea what might be wrong?

Yes, I have done this myself. When you configured the kernel I bet you
answered NO to the question about SRM support, as you would normally do
if you were using MILO to boot. Run 'make config' again, and be sure to
tell it you want to boot from SRM.

If this works, you can thank David M-T who answered almost exactly this
same question for me about 6 months ago :-)

Jim
