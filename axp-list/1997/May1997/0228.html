A> jestabro@amt.tay1.dec.com wrote:
>>
>>>>> Stephan Schroepfer said:
>>>> After installation of RHL4.1 on a PC164-based machine 'date'
>>>> reports 'year 2017' (...)
>>
>> (...) Change /etc/sysconfig/clock to: CLOCKMODE="ARC".
>>
>>>> ... Calling clock (...) from the shell (...) I get a message
>>>> like 'missing I/O port address'.
>>
>> If it says: "clock: unable to get I/O port access", its probably
>> because you're not running as "root". If it says something else,
>> check that the output of "cat /proc/cpuinfo" has EB164 for the
>> value of "system type".
>> ...
>> Also, you could try the work-aound by doing:
>> ln -s EB164 /etc/alpha_systype
>> which should allow "clock" to do the right thing.
>> ...
>> --Jay++

A> Hello Jay and the others out there:

A> Oh yes, I think there is a problem with 'clock' which cannot
A> be solved with Jay's 'ln -s' workaround alone.

A> Summary:
A> - Patching /etc/sysconfig/clock according to
A> Jay A Eastbrook's tip seemed to solve the problem
A> of having a wrong date only temporary (until
A> next reboot (from milo))
A> - 'ln -s EB164 /etc/alpha_systype' - workaround
A> (according to Jay A Eastbrook) seemed to get date
A> right until next to next reboot (had to use reset
A> button and after that things got even more weird)

A> Following text is a rewrite of some sort of commented manuscript
A> I made during my (somewhat frustrating) experiences with the
A> 'Wrong date with RHL4.1 on PC164' - problem

A> # logging in as root (I thought first settting the right date
A> # and then allow users to log in could be no error)
A> date : Sat Apr 29 19:46:51 2017
A> # (maybe Sat Apr 29 2017 will be really a saturday)
A> # '...2017' may serve as an alias in the following text as
A> # '...1997' will serve for the right date, I think '...2035'
A> # needs no further explanation
A> clock : ...2017
A> clock -r -A : ...1997
A> grep type /proc/cpuinfo : system type : EB164

I've got a similar problem: setting the system clock of my Jensen with "clock
-r -A" does only work as long as I don't reboot. The ARC clock obviously is not
affected by the clock command.

I have to reset system time from the ARC console which is a bit annoying, I
have to shutdown, change OS type to NT, switch the machine off and back on, set
the clock, change the OS type back to OSF, switch off and on, and do this one
time more if I failed to calculate the right offset between ARC time and CET or
CEST which I occasionally do.

Is there any solution for my problem?

Viele Grüße, Christoph
