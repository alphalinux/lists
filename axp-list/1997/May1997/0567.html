Hi,

I often use an old Archive Viper 150 (SCSI-I tape) to move
files around. This old dinosaur works flawlessly on all kinds of machines
(Linux-i386, SysV on m68k, Suns) except my UDB. The firmware finds it on
boot and so does Linux; /proc/scsi/scsi shows it as

Host: scsi0 Channel: 00 Id: 05 Lun: 00
Vendor: ARCHIVE Model: VIPER 150 20000 Rev: -000
Type: Sequential-Access ANSI SCSI revision: 01

However, any attempt to read or write to the tape results in a
hung process. The tape light blinks for a second, and the process is
stuck in state DW until reboot, keeping the load avg at 1. Nothing is
actually read or written. "mt status" shows the following before hanging:

# mt status
SCSI 1 tape drive:
File number=-1, block number=-1.
Tape block size 512 bytes. Density code 0x0 (default).
Soft error count since last status=0
General status bits on (1010000):
ONLINE IM_REP_EN

With the BSD-ported driver, the system keeps on working once a
process is stuck on the tape; with the 53c7,8xx driver the whole system
hangs. The tape is properly terminated (passive, but works on all other
systems including those with an NCR53810 based controller), and there's
nothing else on the SCSI bus except the internal Toshiba HD. The UDB runs
kernel 2.0.27 with no modules; I'll try 2.0.30 and 2.1.38 to see if they
make any difference.

Any ideas?

Tnx in advance.

Regards,

Miguel
