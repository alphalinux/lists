Stephan Kanthak <kanthak@gazelle.informatik.rwth-aachen.de> writes:
> >Try '-funroll-loops'. This often helps, and it isn't turned on by any
> >-O<x> level. Also avoid unnecessary float/integer conversions.
> I remember Linus saying that '-funroll-all-loops' makes Alphas much slower
> because small loops do no longer fit into L1/L2 cache. Is this true for
> '-funroll-loops', too, or does GCC unroll loops then with respect to L1/L2
> cache size?

'don't know about the cache issue, but citing the gcc info pages:

`-funroll-loops'
Perform the optimization of loop unrolling. This is only done for
loops whose number of iterations can be determined at compile time
or run time. `-funroll-loop' implies both `-fstrength-reduce' and
`-frerun-cse-after-loop'.

`-funroll-all-loops'
Perform the optimization of loop unrolling. This is done for all
loops and usually makes programs run more slowly.
`-funroll-all-loops' implies `-fstrength-reduce' as well as
`-frerun-cse-after-loop'.

I indeed never got an performance increase with
'-funroll-all-loops'. However, I measured an about 10% performace
increase (21064/Jensen) by using '-funroll-loops' on povray.

And that's just one example. Depending how tight the inner loops are,
it can be much better. But it might be a good idea to use profiling
and only to apply this option on certain files.
