Something is truely hosed in this log.

> May 1 18:59:06 asgard kernel:
> X(14480): unaligned trap at 0000000000000024: 00000001206c11ce 28 1

The X server should not be running code at pc 0x24.

> May 1 18:59:24 asgard kernel:
> kernel: unaligned trap at 000000012007bb14: 000000011ffff791 29 16

The kernel should *definitely* not be running code at 0x12007bb14 --
everything should be up in the 0xfffffc0000310000 range as earlier in
the log.

> May 1 18:59:24 asgard kernel: X(14480): Instruction fault 4
> May 1 18:59:24 asgard kernel: pc = [<12057cf14>] ps = 0000
> May 1 18:59:24 asgard kernel: Code: 2057c000 00000001 005100c6
> <00cc013a> 2047d400 00000001 015a3140 00000000 20563c00

And finally, we kick off trying to execute bits that aren't code,
in kernel mode, located somewhere in user space. What fun!

I really can't imagine what lead to this sorid state of events. Well,
actually, I can imagine -- stack corruption by an errant driver --
but not which driver or under what circumstances.

But it definitely isn't the X server's fault.

I'd be interested to know if this sort of thing happens again. If
so, I can only suggest getting rid of some of the less common bits
like appletalk and see if the problem goes away. But at present I
have no particularly constructive suggestions.
