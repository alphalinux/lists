>
> I installed Redhat 4.2 on a PC164, because many directories and symbolic
> links with Redhat 4.1 had a date with the year 2017.

Ugh! Not again!

Arc console has a different idea how to read hardware clock
then your system (which follows the same conventions as SRM).
Unfortunately install procedure does not correct that before
starting write to a disk.

> Unfortunately, the
> kernel used to install Redhat 4.2 also thinks it is 2017 and installs the
> directories and links with the wrong date.

If this still happens that means that you configured your system
wrong by not telling it that you use ARC clock. There is an extra
flag, -A, used by 'clock' on Alpha to "synchronize watches". It is
used automatically by startup scripts IF your configuration is proper.
See 'man clock' for more details.

> The wrong dates wouldn't bother me, but I think they confuse RPM when
> upgrading packages.

After you set your clock and date properly:

touch /tmp/now_stamp
find <list_of_directories_to_update_here> -newer /tmp/now_stamp | xargs touch
rm /tmp/now_stamp

This will not change dates on symbolic links, and such operation
requires a bit more of care, but this should be a smaller worry.
Also all dates in the future will be changed to the current time
and not to original dates from archives.
