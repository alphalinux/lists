Yes, it's true, the event you've *all* been waiting for has *finally* happened!
(Well, at least *some* of you have been waiting for it, and it *is* actually
*two* events, but forgive my hyperbole :-)

Digital Semiconductor's AMT Group Linux Project Team is proud to announce
that you may now find on gatekeeper.dec.com:

/pub/DEC/Linux-Alpha/Kernels/alpha-patches-2.0.30-0.1.gz
/pub/DEC/Linux-Alpha/Miniloader/v2.0/src/milo-2.0.30.tar.gz

Highlights
==========

alpha-patches-2.0.30-0.1.gz

includes "axp-diffs-2.0.30.gz" and em86 patches
support for new platforms
can now build BOOTP-able images
many enhancements/fixups/cleanups/work-arounds

milo-2.0.30.tar.gz

builds under ELF (RH 4.2 at least) with the DEC BIOS emulator
allows booting of large, 2.1.xx kernels
eliminates (?) the dreaded PC164 128Mb memory problems (you tell me)
supports MIATA aka PWSa(u) 433/500/600

Details
=======

alpha-patches-2.0.30-0.1.gz
---------------------------

This file supercedes "axp-diffs-2.0.30.gz" as The Ultimate Patch File (tm).
Included are *all* of the former's wondrous innovations, now accompanied by:

1. support for new (at least to LINUX :-) Alpha platforms:

NORITAKE aka AlphaServer 1000A
SABLE aka AlphaServer 2100
MIATA aka PWSa(u) 433/500/600
AlphaBook1 aka the HOT laptop :-)

2. miscellaneous cleanups, fixups, work-arounds, new stuff, include:

XL 233/266 aka the EV4 XLs (esp. for MIllennium-based)
MIKASA aka AlphaServer 1000 (EV4 and EV5 CPUs)

em86 beta-1 original patches
BOOTP images (for booting directly from SRM console)
multiple installed SCSI controllers
Myrinet card configuration
kernel build option for SRM direct boot
for previously unsupported platforms

As before, correct installation of patches is accomplished by (with the
appropriate files in the current directory):

tar xzf linux-2.0.30.tar.gz
cd linux
zcat ../alpha-patches-2.0.30-0.1.gz | patch -p1 -E >& ../patch.log

There are quite a few new Alpha-related configuration choices, so I *highly*
recommend careful reading of the options (at least) the first time through
"make config" (I have no idea whether "make menuconfig" or "make xconfig"
will work, but suspect they may not).

milo-2.0.30.tar.gz
-------------------

This file supercedes former files of the same name. It is the full source
tree needed to build MILO for the supported platforms. It includes a new
ELF version of the DEC BIOS emulator library so that MILO may easily be
built under ELF-based distributions, especially Red Hat 4.2. Also included
are many cleanups and such, and new memory management routines that will
allow booting of *large* kernels (JavaOS is >4Mb :-) like the 2.1.xx series,
as well as (hopefully) eliminate the problems PC164 owners have had with
large (128Mb and up) memory configurations.

The MILO HOWTOs are, as always, the final source of directions for building.

A new set of MILO binaries is *not* included at this time, but will appear
Real Soon Now (tm).

However, there is a PC164 MILO on gatekeeper in that Kernels/ directory,
named "milo-2030-pc164.test", IIRC, that *does* include most if not all of
the kernel and MILO patches, for those who wish to "take it out for a spin".

Some VERY IMPORTANT Notes
====================

1. many of the alpha-patches-2.0.30-0.1.gz patches were included in the
Red Hat 4.2 kernel sources/headers RPMs, but far from all. If you decide
to use alpha-patches-2.0.30-0.1.gz, *please* do so using the pristine
linux-2.0.30.tar.gz sources, and *not* the Red Hat RPMs, as you will
undoubtedly get *many* complaints from "patch".

2. milo-2.0.30-tar.gz will work best if built against a kernel updated with
alpha-patches-2.0.30-0.1.gz

3. kernels and MILOs constructed during testing of these patches were
primarily (only?) built under Red Hat 4.2 plus upgrade RPMs; this does
*not* include the "alphabits" ones, though I would hope there'd be no
problems with them.

4. as always, if you're *not* experiencing problems with your kernel or MILO,
you don't have to adopt any of the above.

5. there was a problem *building* the EB164 PALcode from source; since this
is not a necessary part of building *any* of the many MILOs based on that
code, simply take the default ("no") for that MILO configuration option.

6. BOOTP images have been tested successfully *only* from SRM console,
and, of course, any kernel to be so booted *must* be built to
"Use SRM as bootloader".

7. these patches do NOT, repeat NOT, contain SMP support.

8. these patches do NOT contain specific PC164-LX support; the LX will
run stock PC164 kernels and milo.dd but ONLY from SRM console.

9. the Sturgeon General warns that these patches may be addictive... :-)

Now, I'm sure I've forgotten some details about all that's in there, so if
you have any specific questions please feel free to mail me directly at:

jestabro@amt.tay1.dec.com

Also, PLEASE send information about any problems you run into to me as well.

Well, that's all, folks; have fun with it!
