See the following hints which I received regarding what seems to be
exactly the same problem -- Thanks to Michal Jaegermann, the following
is a shortened version:

----------------------------------------------------------
Michal Jaegermann in reply to some questions of mine:

...

Most likely you run into a known (this is not Alpha specific) memory
management bug which may lead to a page starvation even if there is
a plenty of free memory on your system. For some reasons this bug
is more likely to show up with 2.0.30 than with earlier kernels
(2.0.29 including).

The bug is very actively worked upon but problems are not so simple.
It seems that a resolution is in sight - we will see soon. As a
workaround for now you may try, for now, for memory intensive jobs to
hike up values which show up in your /proc/sys/vm/freepages by echoing
a new triplet there. This has to be done from root account.
"Downgrading" to 2.0.29 also may help (but does not have to).
>
> Unfortunately, this /proc/sys/vm/freepages
> stuff is slightly above me. I realize that I can operate with cat
> on /proc to obtain info (at least as root), but what do I have
> to do to change one of these settings?

echo "100 200 400" > /proc/sys/vm/freepages

The above is only an example.

> Also, what do those
> three numbers mean,

They affect swapping policies by setting minimum, low and high
values for a number of free pages. 2.1.x seriers of kernels
actually has some write-up on this in linux/Documenation. Otherwise
"Use the source, Luke". :-)

> and what are reasonable values to increase
> them

Depends on how much memory you have, how much you can spare and what
will prevent what YOU are doing from triggering this bug. Start with
doubling, or quadrupling, values which are shown by 'cat'.

Attemtps to set file systems on big partitions (like many gigabytes)
are known to run into this bug, in particular. You may want these
numbers really high when you are doing that. OTOH this eats up into
your "work" memory so it is not likely that you want these high
for a "normal" situation.

The above is only a workaround and not a fix. The later is still
"in works".
