At 09:38 AM 7/22/97 -0400, Steven Kokinos wrote:
>
>Since we upgraded to 4.2 we have seen a few odd problems crop up. The
>first is that the 2017 date problem is back again, which we had managed
>to eradicate with one of the patches in 4.1. In addition to that,
>however, cron fails to run any of our crontabs. The daemon appears to be
>running, but never actually executes any of the scripts. Could this
>somehow be related to the date problem, or possibly something else?

WRT to the date; does it work correctly with some combination of args
from the command line? Meaning, if you boot via MILO, does "clock -a -A"
set the right date; if booting directly from SRM, "clock -a"? If so, then
checkout /etc/sysconfig/clock and /etc/rc.d/rc.sysinit for what is expected
and performed.

And yes, I think it might adversely affect "cron".
