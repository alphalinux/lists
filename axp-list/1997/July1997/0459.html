> On the subject of Milo, I have spent two weeks off and on attempting to
> convince Milo 2.0.29 (or any earlier ones I've tried) to "see" my second
> SCSI drive at boot time. Even after wiping the entire disk, partitioning
> with NT 4.0 arcinst.exe and rebuilding the Linux partitions and file
> systems, it _still_ reports "Unknown partition table" at boot.
>

Is this an FDISK problem? Are you saying that MILO fails to read the
partition table but that Linux does not?

> The only way I'm able to start Linux on the second drive is create a small
> ext2 partition after NT on the first drive, place the kernel image on it,
> and start it from Milo with "root=/dev/sdb5". The image can see and mount
> the second disk without incident.
>
> The machine is a UDB FX-40 w/ 40Meg of memory and RedHat 4.2 (although RH
> 4.0 did the same thing..). Now that a (re)buildable source for Milo is
> available, I'll take a look at it. Before wading in, are there any
> obvious differences between it and a "real" kernel regarding how they
> handle the disk partition maps?
>

There's no difference, they use the same code: if the kernel can see the
partition table then MILO should be able to. I should be building all
new MILO images next week (including the UDB/Noname)...
