I have a PC164 with a flaky floppy controller. I'm using a Buslogic
scsi card, which the ARC console firmware can't drive, so I must boot
MILO from floppy.

The very first time I tried it, it didn't work ("A problem has been
encountered. Press any key to continue") but trying again worked.
Since then, the percentage of failures has been going steadily up.

I decided I need MILO in flash, then I can forget the stupid floppy
forever. As long as the flash MILO can boot a hard disk MILO, I'm
fine, even if someday I need a newer MILO.

So, with considerable trepidation, I built a fmu.gz with the same
MILO I'm using on floppy (latest 2.0.30 from gatekeeper, with kernel
vanilla 2.0.30 plus alpha-patches from gatekeeper). I flashed MILO.
There did not seem to be a debug monitor there -- block 0 is empty,
WNT starting at block 1, MILO starting at block 7 (I think).

I set bootopt to 130, for the second flash image. It booted to
the ARC console. I set it to 131. It booted to the ARC console.

So now what? At the moment, the machine is down, having failed
20 tries in a row to boot off floppy. (I've tried several different
drives, and two cables, so it really looks like the controller.)

(All I need is a $10 floppy card, plus perhaps disabling the motherboard
floppy, but the Microsoft(R) Windows(tm) Nt POWERSoftware ENTERPRISEBios
cannot disable the motherboard floppy. Or do much of anything else.
Or boot anything. But I digress.)

So assuming I get perhaps one more chance to load a program off floppy
before this motherboard turns into a wall ornament, what should I do?
Can SRM see Buslogic cards? Does SRM obey bootopt? If I wipe the ARC
console from flash and load MILO starting at block 1, will it get loaded?
Is there a debug monitor in there somewhere?
