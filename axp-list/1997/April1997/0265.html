On Mon, 7 Apr 1997, Daniel Gruner wrote:
> Hi
>
> Can anyone enlighten me as to why sys/socket.h on linux/alpha defines
> functions like accept() and getsockname() using a "size_t *" 3rd
> argument, while the rest of the world, including linux/intel use "int
> *"? This causes all sorts of portability problems. Here is the
> definition from linux/alpha (RH 4.0, kernel 2.0.27 + axp-diffs):
>
> extern int accept __P ((int __fd, __SOCKADDR_ARG __addr,
> size_t *__addr_len));
>
> while on intel it is
>
> int accept __P ((int __sockfd, __const struct sockaddr *__peer,
> int *__paddrlen));
>
size_t is an address type. or a long.

> Mind you, the compiler should automatically cast one to the other (after
> all the two are only pointers), but it somehow doesn't and dies with

you are asking system to write a long(64bit) result into a int(32bit)
location. The value that you get will be correct but it is lilkly to
overwrite the next variable.

>
> ksock.cpp:210: passing `int *' as argument 3 of `accept(int, sockaddr *, long un
> signed int *)'
