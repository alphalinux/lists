This machine is at long last installed. (Peter Braam and I have sent a
couple messages over the last 24 hours on the subject.)

For the benefit of others, here are the lessons:

The machine:
AlphaStation 255 with SRM console, 1gig SCSI disk, SCSI CD-ROM

We added a floppy drive to allow us to load the ramdisk in the install
process.

We upgraded the SRM console to version 6.4 by obtaining the firmware
update program from gatekeeper.dec.com and setting up another Linux box to
run bootpd/tftpd to serve it.

We ran the install program by typing
SRM> boot dka400 -fl i
aboot> kernels/avanti-s.gz root=/dev/fd0 load_ramdisk=1
and then putting the ramdisk floppy in the floppy drive and pressing
return when requested. Here 'dka400' is the CD-ROM.

We used BSD disklabels on the disk, with
partition a ext2 /
partition b swap
partition c ext2 /home

[Note: kernel partition support seems weird: when we tried setting up a
BSD disklabel in a little more customary way--with an unused partition 'c'
spanning the whole disk--everything went straight to hell. While fdisk's
secret BSD annex saw the partitions correctly, the Linux kernel reported
Partition check:
sda: sda1 sda2 sda3
and the boot process fell over. This took place even after rebooting to
make absolutely sure the new partition table was read.]

Key point: we left a little room on the disk before partition 'a'. This
space is used by the 'aboot' bootstrap program.

Once the install is done, it was possible to boot the hard disk system via
the CD-ROM:
SRM> boot dka400 -fl i
aboot> kernels/avanti-s.gz root=/dev/sda1

>From there, we installed the aboot RPM and did
swriteboot /dev/sda /boot/bootx

Then, in the SRM console:
SRM> set boot_file vmlinux.gz
SRM> set boot_osflags root=/dev/sda1

Now the machine comes up into Linux just fine.

Phew!

Michael and Peter
