Jim Paradis <jrp@jrp.tiac.net> writes:
> geerten kuiper wrote:
> > FX!32 not only emulates the x86 environment, but also can do a binary
> > translation, resulting in much better performance for future runs
> > of the same program. Is this functionality planned for EM86 as well ?
>
> No, it is not.
>
> Frankly, I debated back and forth as to whether to mention FX!32 at
> all. EM86 is based on the x86 execution engine from FX!32 (which
> is written entirely in assembler and is VERY carefully hand-tuned
> to achieve optimum performance on Alpha),

Well, that means that there will be no improvement in "raw" speed? I
noticed a factor of 25 between emulated and native codes.

> but I do not intend to
> implement the incremental translator (unless someone pays me VERY
> well do to so 8-) ). I believe that a combination of the FX!32
> interpreter with calling native shared library routines will achieve
> adequate performance for most applications.

While this is true for the average application, it won't be good for
most applications you would like to use with em86: those are mostly
commercial applications, that can't be recompiled. But because there
are so many different Linux distributions and environments, vendors
tend to avoid risks and link most of the stuff static. Take netscape:
Only libc and X11 libraries are dynamically linked, the jpeg library
is instead included, thus we won't see an improvement on the stuff
that's really slow.

Anyway, netscape runs emulated even faster than grail (at least on
text pages), so there's already some benefit to me :-)

Thanks, Martin

(Bugs noticed so far: ldd runs the binary instead of examining it;
some signals are wrong, e.g. ^Z in a shell will yield signal 18.)
