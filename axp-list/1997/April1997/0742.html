I have been trying to use smbfs to mount PC disks over the net on my UDB.

The stock RH4.1 does not have the smbmount/smbumount commands as delivered.
I used the smbfs-0.8 srpm to compile these two. (I also used smbfs-2.0.0
from sunsite and it behaves the same way). It seemed as though they function
well enough except that in a mounted directory, it couldn't find the files.
If I used the full pathname to a file, it worked fine. Example: (mounting a
WfWg 3.11 share drive)

>smbmount //puter/putersd /mnt/pcdisk -n
>cd /mnt/pcdisk/temp
> pwd
/mnt/pcdisk/temp
> ls (works fine, and finds file below)
..
foobar
..
> vi foobar
<new file>
> vi /mnt/pcdisk/temp
<file found ok>

Just for grins, I went into xv and went to browse for a file. It had thrown
an extra couple of characters into the pathname: /mnt/pcdisk/bu/temp.

When I tried mounting a Win95 shared disk, using ls would not list all the
files in the directory. If I typed in the complete pathname, it would use it
fine. Otherwise, the file was invisible.

None of the above behavior was observed using the RH4.1/Intel version
linking to the same place. This leads me to believe it may be something
alpha specific.
I didn't see anything obvious in smbmount.c such as 64/32 bit-isms, but then
I'm no expert on the subject. Also, a goodly portion of the smbfs stuff is
in the kernel tree itself (a place I definitely do not wish to dink around
with).

If someone more knowledgable than I about such things (which would be most
everybody) could take a look at the kernel smbfs code for alpha unfriendly
code, it might turn up something easily fixed.
