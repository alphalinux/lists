> gcc itself (2.7.2) is converting FLT_MIN (1.17549435e-38f)
> into 1.17549435082228750797e-38 (which I believe to be correct)
> The assembler (gas) seems to be screwing up and converting that
> number into 0x007fffff instead of 0x00800000. I'm investigating
> a fix.

There was a bit in

ftp://ftp.redhat.com/pub/alphabits/rth-gcc-2.7.2-961121.diff.gz

that attempted to address that problem. This fix was included in the
RH 4.1 release, but I don't recall if that was before or after RH 4.0.

Anyway, the solution I used was to round the printed constant to one
more decimal than is actually present in the number instead of doing
garbage out to twenty digits. So you'd know if it was present if
gcc's output was of the form ".s_floating %.7e".

This worked at least for the test case

float foo = FLT_MIN;

If you've got something else that it still fails on... well, we'll
either fix gas, output hex from gcc, or both.
