Funny you should mention this! I am in the process of porting the NAG
F90 compiler to Alpha Linux. I have run into a problem with gcc. I have
been emailing with Rick Gorton of DEC and he suggested I post the problem to
this list.
Here is some excerpts from our email conversations that explain the
problem:

----- Begin Included Message -----

What I have found is that there is a serious bug in the C compiler (GNU C
version 2.7.2). When passing the smallest "float" as an argument, it passes
instead a value smaller than the smallest "double".

There is, to my knowledge, a 2.7.2.1 version which apparently contains some
bug fixes. Worth checking, but I do not hold any high hopes.

-------------

Hi Barry,

I've been able to recreate the pass-by-value of MIN_FLT problem
with gcc. I'm looking for the root cause now - I'll keep you posted.

Rick

>From amt.tay1.dec.com!gorton Fri Mar 21 13:27:11 1997
From: Richard Gorton <gorton@amt.tay1.dec.com>
To: caplin@nag.com

Here's what I've found so far:

gcc itself (2.7.2) is converting FLT_MIN (1.17549435e-38f)
into 1.17549435082228750797e-38 (which I believe to be correct)
The assembler (gas) seems to be screwing up and converting that
number into 0x007fffff instead of 0x00800000. I'm investigating
a fix.

FYI, this is a fairly old binutils problem:
Thu Sep 23 16:20:34 1993 Ken Raeburn (raeburn@cambridge.cygnus.com)
* gas/ieee.fp/x930509a.exp: Currently expected to fail always.
# Reported 93/05/09 by Jim Wilson: IEEE single-precision FLT_MIN value gets
# assembled incorrectly. (Off by one ulp.)

Regards,

Rick
