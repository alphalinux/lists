> > I have seen several articles about cache size in intel world,

> > 256K.
> >
> > I think the difference in RISC vs. CISC is the main reason,
> > but the way the executable is laid-out in memory might play
> > into this as well, I don't know - it is just one possibility.
>
> It all depends on the "FEED ME" memory attitude of each processor type,
> and design. At 200 mhz, that a clock of 5ns, most dram is still at 70ns.
> At 500mhz the clock is 2ns. Risc has very simple instruction set, and
> can execute each instruction in a cupple of cycles, and therefor need to
> be "FEED" quite often. Cisc, has a more complex instruction set, for
> which some instructions take a whole lot more cycles to complete, and
> therfor not need to access memory as fast.
> Faster cache, or more cache means that the processor can be timely feed
> when demand calls for it.
>
> Cache is also ( suppose ) to be segmented, much like that of virtual
> memory architectures. Even if the program branched all over the place,
> your executable would be somewhere in cache, ready to feed the
> processor. Therfor if u could fit most of the current executable thread
> into cache, you could just about fly at the speed of the processor.
>
> As for benchmarks, there indicative of something, like statistics. Your
> application(s) should be the final arbitor of any speed questions.
