Could you not compile a kernel to recognize the DU (berkeley UFS)
file system, and SRM boot the DU partition directly for DU and
use a MILO diskette to boot the Linux partition?

Why use two hard drives? Two / partitions should do it, keep the
shared partitions in a commonly recognizable file format. Standard
UFS partitioning divides the disk according to a standard template,
but careful reading of the disklabel man page (or playing with the
emulatably friendly DU 4.0 disk partitioning GUI) will reveal that
there is as much flexibility to UFS partitioning as there is under
FDISK.

Just talking theory here, haven't done such a thing.

You couldn't use ADVFS partitions, since that's proprietary, or was
last I checked, although they (digital) do make some noise in the ADVFS
docs about wanting ADVFS to be some kind of "new standard" like NFS is a
standard, but they haven't (AFAIK) backed that up with releasing a
ADVFS for Linux-AXP kernel module or something like that. Personally
I'd be pleased to have a working set of ADVFS DU 3.2C shared libraries:
the ones here give all sorts of boot time "unresolved symbol" error
messages and occasionally (like last night: we suspect daylight savings
time, no one knows for sure, no one knows what to do with a vmcore file
except erase the thing) freeze the news server and frustrate the
university's news reading community.

Someone please correct me if I'm wrong. (about linux being able to mount
the UFS file system as implemented by Digital.)
