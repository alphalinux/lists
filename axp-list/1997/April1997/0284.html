pete, terry (and children), others

(A)
I've finally managed to run the exactly same command on both Intel and Alpha:

time povray +a0.3 +w640 +h480 -ikaleid5.pov +d +SP32
(640x480 because 800x600 crashes, +d +SP32 to be able to see something
(adds only a few seconds))

results:

21164 Pentium
366 MHz 90 MHz
time for parse 0:00:03 0:00:10
time for trace 4:46:53 16:08:45

user 17013.03 57423.00
system 4.99 11.99
elapsed 4:46:56 16:08:15
CPU 98% 24%

So the main conclusion is that "time" is broken on Intel: my calculator
makes that 99%.

The second conclusion is that the Alpha at 366 is 3.4 times as fast as the
Pent at 90. This is not as good as I hoped (hence Subject:). I think a
better compiler should improve this. Larger cache might also be useful
(fixed at 1M on PC164), and maybe memory bandwith. Please feel free to
verify this by running the same test (a) with DU-compiled povray, (b) on an
EB164, or one of those newer boxes, with more cache, or (c) on a PC164 with
8 SIMMs installed.

(B)
Some other experiments: editing the .pov file to find out where that time
is being spent:
removing the fancy parameters on some of the light sources --> much, much
faster
turning them off altogether --> even better
replacing the BITMAPnn textures with WOOD1 --> pretty too (and faster)
All of those together --> 1:13 (no, not 1h13m: 1m13s. Would you believe it?)
For these variations the speedfactor between Alpha and Pentium varies
between 3.6 and 4.4.

I guess , to meet her deadline, your friend should have rendered the
picture without the extra params on the lightsources. I imagine pretty much
the same effect could have been achieved by simply adjusting the
RGB-values. This variation took 16:14 on the Alpha, 58:23 on Pentium, and
would certainly have completed within a couple of hours on your 486.

(C)
I started out doing those tests on my X-desktop, with top running in an
xterm. Then I realized that top effects the povray in two ways. It takes
away CPU time, so it will take longer. OK. This is easily accounted for. No
harm done. But.., on top of that, every time after top has run, and talked
to xterm, which has talked to the X-server, and then povray gets back to
the CPU, and finds (part of) it's cache flushed, so for a while it will run
slower than it could have.
I tested this by running one rendering with top, without top, without some
other things, and finally from single user mode. There's a difference of
about 2% in the CPU-time taken by povray (as reported by time). So this is
not too important after all. I'm just telling you to show how busy i've
been, and maybe to show how easy it is, in performance work, to get
involved in minor details and lose track of the main issues.

(D)
Avoiding the povray 3.01 bug by running the Intel version from the EM86
environment (I think Pieter came up with this..?) In my opinion you
wouldn't run into the bug, but we'll never be certain because I doubt
anyone will have the patience to run this tests. Those CPU intensive
programs usually show very bad performance, even under the best of emulators.

(E)
Now I'll render the picture one more time, at a very high resolution, just
to look at it. And that will have been the last time.
