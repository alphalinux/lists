> With help form Jay I have been succesfully installing an Alpha Server 2000.
> It uses the SRM console and aboot to boot, and on power-up, it automatically
> boots.
> However, if I reboot the machine, with either 'reboot' or 'shutdown -r now'
> it gets back to the SRM console and I get the message:
>
> halted CPU
> halt code = 5
> HALT Instruction executed
> PC = fffffc00003131f0
> boot failure
> P00>>>
>
> If I type boot, it just boots nicely, again.
>
> Can someone enlight me where to start looking to get this behaviour right?
> Should I look in the kernel-code, in the aboot-code, in the SRM console
> settings?

Unfortunately, I think the kernel does not gracefully re-enter SRM on
reboot. I believe that SRM expects a certain environment that the LINUX
kernel does not return, such as PCI device addresses and IRQs, virtual
memory setup, and perhaps others. This makes auto-reboot non-functional
when booting under SRM console on most if not all Alpha platforms.

As a start, one could rebuild the kernel specifying in to the
arch/alpha/kernel/bios32.c code that PCI_MODIFY was *NOT* to take place.
This would at least leave the PCI devices as SRM expects to find them.
However, other changes by the LINUX kernel may still prevent SRM from
successfully restarting; if so, I'm not sure how to proceed from there.

Sorry... :-(
