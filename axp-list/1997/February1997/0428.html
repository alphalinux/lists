Terran D Lane <terran@ecn.purdue.edu> wrote:
> one or more "Oops 0" and "Unable to handle kernel paging request"
> error messages [...]
> Initially I experienced this with kernel 2.1.13 and the BSD NCR
> driver. Since I hadn't used the BSD driver before, I attributed the
> errors to that, and re-compiled with the other driver. Same problems.
> I upgraded to 2.0.28 and thought things were normal again, but the
> system died like this a couple of days ago.

I haven't tried removing the internal disk, but my experience these errors
starting when I tried to run 2.0.x kernels.

I was running RH/AXP 3.0.3 with a 1.3.90 kernel, and everything was fine.
I built my own 2.0.0 kernel, and started getting the errors.
I went back to 1.3.90, and everything was fine.
Later I tried 2.0.10. Errors again.
Went back to 1.3.90. No problems.

Then RH 4.0 came out, with a 2.0.18 kernel, and I figured that whatever
the problem was, it had to be fixed or Red Hat wouldn't have released
it. Upgraded to RH 4.0. Got the Oops and Unable to handle kernel paging
request errors back.

Unfortunately sinch RH 4.0 is ELF based, it is now really difficult for me
to go back to 1.3.90. I keep thinking that I'll do a binary search of the
kernels between 1.3.90 and 2.0.0 to find out exactly where this problem was
introduced, but so far I haven't found the time to do it.

People have suggested that all sorts of things are wrong with my machine,
from bad SIMMs and disk drives to bad SCSI cables and termination to
overheating. I don't believe any of it, since 1.3.90 and Windows NT both
run flawlessly on the exact same system.

Some people have suggested that the BSD driver would solve my problems, but
it doesn't.

I think some sort of race condition crept into newer kernels. Maybe this
weekend I'll finally get around to reinstalling RH 3.0.3 so I can try to
track this down.

Cheers,
Eric
