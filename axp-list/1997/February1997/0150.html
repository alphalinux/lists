I have a DEC Alpha XLT-366 (an XL-266 that's been upgraded by DEC to a 366).
I just upgraded to 256 meg (in 2 banks of 4 32MB simms). If I only populate
one bank (ie, 128 meg), it boots and loads Linux. However, if I populate
both banks, the system hangs during the load with the message

Swapping to PALcode at 0x8000 ...

More details:

I've upgraded the AlphaBIOS to 5.24 (the most current from DEC).
AlphaBIOS 5.13 exhibited the same behavior. I'm using the current
linload.exe off of gatekeeper.dec.com, and the XLT Milo from kernel
version 2.0.27, also from gatekeeper.dec.com. The same thing happens
if I'm loading linload and milo from a floppy and if I'm loading them
from a hard drive partition.

The full boot messages are (for an attempt to load from floppy):

----------
Linux Loader V1.3

Copyright(c) 1994,1995 Digital Equipment Corporation

System Partition is OSLOADPARTITION=scsi(0)disk(0)rdisk(0)partition(1)
Load File Name is OSLOADFILENAME=\
Load Path is multi(0)disk(0)fdisk(0)\
OSLOADOPTIONS=

Opening file multi(0)disk(0)fdisk(0)\milo ...
Loading file at 0x80080000 ...

Setting firmware parameters ...

...Memory size is 0x10000000 bytes

Setting up argmuments for Milo @ 0x801FFC00

Setting up memory size for Milo @ 0x801FFE00

Swapping to PALcode at 0x80000 ...
----------

and then the hang. ("argmuments" is really what it says)

It also booted fine with 4 16MB simms (the old configuration). The
memory test and setup both find all 256MB. WindowsNT also boots and
runs with 256MB for what that's worth.

Any suggestions?

-Dave
