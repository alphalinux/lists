I've got a UDB that I'm wanting to upgrade from RH 3.0.3 to (at this
point) 4.0. I suppose 'twould be nice to move to 4.1; I may get the
CD at some point. I suspect that my problems would still stand.

I've got the CD in CDROM (a Sony 2x SCSI CD-ROM, for what it's worth)
and am trying to boot off of it order to upgrade the box. I'm liable
to blow away existing configuration; I'll likely run the process a
couple of times in any case.

I'm at the MILO prompt, with the Yggdrasil Archives version of RH4.0
in the CD drive. It's there; I can do
MILO> ls -t iso9660 scd0:
and get a nice directory listing.

I use:
MILO> boot -t iso9660 scd0:kernels/noname.gz root=/dev/scd0

to try to boot the Red Hat initialization process. So far, so
good. It starts detecting disk drives, and starts loading the kernel.
It does the "A write error to b-cache occurred at address 18090d0"
"Attempted to access nonexistent memory." thing. Looks OK.

It starts loading the kernel.
###################################################################################################################done.
Bad filehdr magic number 0x457f should be 0x183
MILO: Failed to load the kernel
MILO>

And there I sit.

I had a similar problem when I tried to upgrade to a newer kernel; I
happily had not replaced the old 1.3.90 version with my 2.0.whatever,
but rather experimented with /vmlinuxnew.gz. I get similar results.

Is this a problem with MILO? My first guess is that I've got an old
version of MILO that uses a different "magic number" calculation than
does some newer version. Or is this completely off-base? It *looks*
like the problem is with the kernel; I could imagine it being with the
root file system. But it surprises me that it would be true both for
my "real" root which *does* boot successfully, with the older kernel,
as well as with a CD-ROM file system.
