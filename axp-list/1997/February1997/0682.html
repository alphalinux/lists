On Feb 18, 11:14am, Jerry Feldman wrote:
> Subject: Re: fclose() (fwd)
>
> > I just checked and Linux and Digital Unix both do the same thing. Heres
> > the program:
> Not quite. On Digital Unix, I change %x to %lx. Pointers are 64 bits.
> > #include <stdio.h>
> > #include <stdlib.h>
> >
> > int main(void)
> > {
> > int i;
> > for(i=0; i<3; i++) {
> > FILE *f = fopen("/dev/null","r");
> > printf("f = %lx\n", f);
> > fclose(f);
> > }
> > return 0;
> > }
> >
> > and here is the Digital Unix output:
> sauron:assem [1528%] a.out
> f = 3ffc0080348
> f = 3ffc0080380
> f = 3ffc0080380
> --

That's incredibly stupid. Is this a /dev/null trick ("read all you want,
there's plenty of nothing for you"). Let's say I open /etc/passwd three times
and start reading records from each (but not necessarily synchronized; f1 reads
20 bytes,
f2 reads 30 bytes, f3 lseeks to the end of file - 50 bytes) There's no way
these three pointers can be to the same address; there's no excuse for
fclose()ing
one automagically closing another (esp. if you fclose it twice, because, well,
you're a fool, but you have a license to be a fool on the wall)
