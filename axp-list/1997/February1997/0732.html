Richard Henderson writes:

> Would the person having the sbrk problem try this out? I think
> this will be close enough to satisfy both OSF and current Linux libc.

...
Changes to kernel source listed, adding new system call to the kernel...
...

(By the way, sbrk and brk are called by malloc, and so are
crucial to many programs.)

Right now I'm afraid to try this, because I just got my machine
(preconfigured from SWT) and I haven't recompiled the kernel
myself yet, at this point. Even though the change looks simple,
I might screw it up and then I'd be in big trouble...

However, it seems to me that this doesn't address the main
problem. Regardless of the way the "out of memory" error is reported,
the OSF program isn't able to get as much memory as the same
program compiled with gcc on the linux machine itself.
When I compile with gcc on linux and run on linux, I get 30
successful calls to sbrk before it runs out of memory (600 Mb). When I
compile with cc on OSF and run on OSF, I get 10 successful calls
(200 Mb). Both of these are behaving correctly. But when I
compile with cc on OSF and run on linux, I only get 5 successful
calls to sbrk--100 Mb when the machine has 256 Mb of RAM.
(limits are datasize=614400 kbytes, stacksize=8192 kbytes).

I don't really care about how it dies if it runs out of memory--
I just want to be able to run big programs. I purchased the OSF
c++ compiler cxx for my OSF machine, and it optimizes much better
than gcc.
