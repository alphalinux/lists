> The same symptom appears with ssh 1.2.17 (running on 2.0.27, RedHat 4.0).
> Interestingly, similar symptoms occur with a standard xterm on SunOS 4.1.3.

> I would guess that it's a problem in the tty code or something related to

Well, tty sounds closer - a new trial (telnet in, cat> /dev/null)
gives the same control result, but if I enter returns into the text,
the problem doesn't show up.

The only thing that makes me think it's telnetd is the following results:

1. cat > /dev/null from the console doesn't exhibit the problem.
2. rlogin from console to localhost doesn't exhibit the problem.
3. rlogin from a remote machine doesn't exhibit the problem.
4. telnet from remote machine exhibits the problem.
5. telnet from console to localhost exhibits the problem.

1 would seem to rule out a drive/tty problem,
2 would rule out the network driver in general,
3 possibly rules out the remote machine being funny,
4 starts to indicate telnetd, and
5 starts waving flags in my head.

Perhaps it's really just how long a string can be in telnetd before
it fails to be written out? Control is never handed back to the
telnet client - the connection is locked pretty stiff. If it were the
length limitation seen in write, you'd be able to BS out of the
problem, but nothing works at this point.

Or am I babbling? :)
