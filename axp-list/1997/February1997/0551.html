> In professional world, I've seen far too many programs that close all
> handles at termination, regardless of whether if known that handle is
> open or not. It WOULD be nice to legislate good programming, I agree...

*sigh* Well, there is a solution when you can't fix the source
and that is use a different malloc that can handle multiple frees.
This can be done by either explicitly linking with one when you are
in control of such things or using LD_PRELOAD when you aren't.

But the vast majority of the programs run under Linux *can* have
their sources fixed, and most often it's just a silly little thing
that only takes a few minutes, and so we should.

> But legacy code rules, especially when practice is "sort of" backed up
> by API specs.

There is no API spec that allows "fclose(foo); fclose(foo);".
As always, the effect is undefined/unpredictable.
