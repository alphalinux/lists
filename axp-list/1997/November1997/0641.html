>
> I was also disappointed to see that XF86Setup does not work on alpha,

Actually it does - to an extent. :-) XF86Setup works either
a) when you have X already running but you are doing reconfiguration
b) when VGA16 X server is present

There is no VGA16 on alpha but one can cheat and link that one to
a Mono server which does exist on alpha and XF86Setup, even when run
from a console, will start provided you overriden Red Hat and installed
XF86Setup despite of failed dependencies and Mono server only without
all other associated files (you do not want to clobber rpm information
for your **real** server). This would be fine and dandy if not for
the fact that tcl, on which XF86Setup relies, is and always was broken
(its original code breaks C Standard and it shows). The net effect is
that when you will try to write out your configuration efforts to a file
then whole thing will lock up and that will be about it.

Same tcl bugs affect other thing, like grail - for example - and various
Red Hat GUI administration utilities (they are Python based, but Python
relies on tcl). In the last case it is a bit strange that nobody bothered
to look for causes; glint on alpha can go quite often into a "weird" mode.
I stopped pay attention and I am using rpm directly from a command
line, which is fine with me, but others seem to like pictures.

This brings back a remark by Erik Troan that he is not going to apply
blindly various patches which "float around". Experience shows that
these patches are mostly correct and there is really no other place to
put them than on this list. It is not a very great incentive to spent
time and effort to fix tcl knowing that the next RH release will likely
propagate the same old bugs once again. If Red Hat would have some
formal bug reporting and tracking system, in particularly for alpha,
they would likely not ship from "always" amd configuration files with
a missing colon. The last thing is far from deadly but annoying and
filling out log files with a nasty messages "for free".
