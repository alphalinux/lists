On Tue, 18 Nov 1997, Thomas Bogendoerfer wrote:

> I've asked a couple of days ago wether someone has used cdrecord on
> Linux/Alpha successfully. As nobody had success with it, I debugged
> cdrecord and found the usally problems (100 instead of HZ, casts
> from pointers to ints and two functions which assume sizeof(long)==4).

Hello,

Sorry for my ignorance, but we got an Alpha UDB as an entry-level system
to learn some of the peculiarities of porting to a 64 it system. We
haven't finished our code yet (doh!) to start going 64-bit clean; what is
"HZ" and its relationship to 100 (or not, in the Linux/Alpha case)?

Also, if you don't mind me asking, what can I do about this? Our code is
into two distinct parts: a UI portion, and a DLL portion (yeah, it's an NT
thing), which does all the needed calculations and memory operations. The
API is hidden behind 'wrapper' functions, such as:
MyAPI_Get( MyAPIptr, API_TYPE, "an.api.property" ); (MT-Unsafe)
MyAPI_Set( MyAPIptr, API_TYPE, "an.api.property", void *Value );

This was halfheartedly designed in the XView paradigm (typically a SunOS
shop) where all return values are strings and the "void *Value" bit can
be anything, which is typcast as needed by the property. But I have a
problem: For one specific property, I want to pass the address of a
function (callback handler). GCC complains to no ends about it.
Naturally, the Microsoft compiler thinks it's great. :)

I guess my question is will this work on an Alpha? Will a function
pointer fit within a void pointer? Does anyone know about the UltraSPARC?
Or a 'proper' workaround that adds no complexity to the user API?
