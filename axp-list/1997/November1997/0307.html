Michal Jaegermann wrote:
>
> >
> > Some tests I did indicate any Alpha divide (float/int) is slow.
> > The table lookup for sqrt() is probably the fastest technique for
> > an approximation.
>
> Actually on machines like Alpha a table lookup is NOT a good technique
> in general. It tends to generate page faults and kills caches.
> For a fast RISC a short localized computation can be vastly preferable
> over a table lookup. Things were different in the past.

That's true if the table is large. Not true if the table is "reasonable"
Depends on page size but a 4K page gives you 1024 floats for a table,
sufficient for lots of types of calculations. In fact, certain RISC
vendor high performance libraries use this technique plus a twiddle.

>
> If 1/sqrt() is on a critical path then a specialized code which computes
> this function instead of calling sqrt and dividing 1.0 by results
> could be a much better answer.

I agree. For mis-behaved functions, this is the only way to go.
