On Nov 22, Maurice W. Hilarius wrote:

> Oh, I don't know if CPU is all of the story. From the amount of HD I/O you
> produce I think disk subsystem performance is pretty important to the
> figures, as well.

I don't see any evedence for this. do a `du /usr/src/linux/' with
a compiled kernel tree (~28M for 2.0.x ix86, no alpha tree arond right now)
and distribute this amount of disk I/O on several minutes kernel compile time
(ok, header files will be read many times but they'll stay in bugger cache).

also check the time of `time make book' and compare user time with
system time and the difference of real time and user time.
both sys and real-user would have be _much_ larger if kernel compilation
is disk bound.

with enough memory (64+ M, for x86 even 32M) compare the compile times
of the first `make boot' with cold/empty buffer cache and the 2nd compile
after `make clean' with warm cache...

> Amount of memory can have some effect, too.

only to the extend that gcc isn't swapping and most of the kernel tree
can be kept in buffer cache. then, only memory bandwith might give
additional advantage.

> All in all this is a good test of a system, as a whole, not inclding video
> performance.

esp. if you plan to use this system for software development;)
but e.g. of course not for FP performace...
