As Armin Ollig pointed out in his reply, this way of benchmarking does not
really give you a direct comparison between alpha & intel, simply because
its a completely different program you're running.
He suggests the bytemark, but running then under NT is not a good idea
either (can runnung anything under NT ever be a good idea ?), because NT -
even on alpha - is only 32 bits, so it does not use the full power of the CPU.
When you want to use the kernel build time for a benchmark, keep in mind
what it is you're measuring. For instance, on the older alpha's (notably
the UDB) a kernel build will take a disappointingly long time, due to the
limited memory bandwith, masking the powers of this system in other areas,
which make it still a nice little box for uses where memory bandwith is
less crucial.

Now, for what it's worth:
I usually time only the "make boot" part (do not use make zImage, which is
only good for intel)
kernels 2.0.29 - 2.0.31
PC164, 366 MHz, 128 bit wide memory path: about 7:30 minutes
PC164, 366 MHz, 256 bit wide memory path: about 7:00 minutes
Pentium, 90 MHz, similar configuration: about 20:00 minutes

Note: building a kernel is very much CPU bound, therefore a faster clock
should give a better result (as far the memory bandwith allows: if the
board you're considering is (similar to) the PC164, 8x16Mb will be better
than 4x32Mb. But...)
... Caveat: surprisingly, other programs I ran as a benchmark (e.g.
rendering a POVray image) actually got slower after I put four more SIMMs
in the PC164. (Does anyone care to offer an explanation ?)

Note: building a kernel is very much CPU bound, therefore using make -j
will not do you much good, unless you have multiple CPUs. Linux SMP is not
supported on alpha (yet).

And again, even if you are going to do a lot of compiling on your system,
this is ot the right benchmark. You should care about the speed of the
resulting program, not the speed of the build.

Good luck though,..
