To add to the EB164/NCR SCSI discussion...

Like Christian Czezatke in the previous post I have an EB164 (a Screamer with
an NCR53C810 controller) that has had a persistent problem under Linux. The
machine froze up continually under the earlier RH2.1, a trait I attributed
originally to the TGA server but later it froze several times on console only -
so that was not it. The advice I had from Jay++ at the time was to upgrade to
RH4.0, which I just did.

The RH4.0 2.0.18 kernel had some ethernet problems on boot-up so I generated a
newer stable 2.0.27 kernel last night. Or I finally did, the machine would not
run long enough to compile the kernel when X was running! I did eventually get
a compile to complete under the VGA console and the new kernel runs OK but the
machine still freezes up when I do a kernel compile in an X window.

Note that this machine runs flawlessly under OSF (first under 3.2d now under
4.0) and has been used for some big number crunching. Different firmware but
the machine has had no problems under OSF.

If this is a SCSI driver problem,

1) how do I tell what is freezing up the machine? Are there kernel messages
logged somewhere?

2) If it is a SCSI problem, what options should I use for the NCR driver? Note
that both 2.0.18 and 2.0.27 use versions of the FreeBSD NCR driver so are there
any recent driver changes that would fix things? The latest driver in the tsx
archives is ncrBsd2Linux-1_14c-src.tar.gz, should I use this?

Any help would be appreciated.
