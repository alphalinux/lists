Lev D Gelb <gelb@cheme.cornell.edu> wrote:

> I have found similar problems between f2c+gcc and DEC
> f77, that DEC f77 is 2x as fast as the converted code.
> I believe that there is something terribly slow in the
> libf77 that comes with f2c, because people in our group
> have benchmarked gcc vs DEC cc on numerical C-codes, and
> found only a 10-20% slowdown with gcc.
>
> Eagerly awaiting g77 version 0.5.20, which should have
> Alpha support...

Hmm, haven't seen anything by Craig yet, so lets jump in:

The top entries of the plan file (finger -l fortran@gnu.ai.mit.edu):

Currently planned for 0.5.20, intended to be a "development"
version for some new features/workarounds:

- Support DEC Alpha (AXP) chip (running Linux, primarily).
(Done.)

- Fix bugs in gcc back end (AXP support; loop unrolling crash).
Package in g77 as patches to gcc-2.7.2 and 2.7.2.1. Submit for
inclusion in gcc-2.8.0. (Done.)

Besides, to say more about Claus's code example, we need a fully
compilable program module; just a few lines out of context won't
prove anything. Note, for instance, that the Digital compiler might
throw the loop away, because (apparently) nothing is done with the
result (gam) calculated inside .... If this is not the reason (:-)
the speed difference might be caused by the fact that at the default
optimisation level the Digital Fortran compiler unrolls loops,
whereas g77/gcc won't do that (because this is a CPU architecture
dependent optimisation).

Hope this helps,
