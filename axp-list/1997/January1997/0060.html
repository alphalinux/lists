The equipment:

Multia (VX40) running Linux 2.0.27 and RedHat 4.0.
This system uses ARC console time.

The problem:

I've just upgraded initscripts (as per the Errata), and the clock
command for ARC console time is /etc/rc.d/rc.sysinit is wrong. If my
system executes the script as it was originally "clock -A -a -u", the
reported time and year will be off by some magnitude (like the year
will be 2017). Changing the clock command to "clock -A -a" fixes
it...the reported time (with the date command, or on the FVWM95 task
bar) is correct, anyway.

Here's the (relevant part of the) original rc.sysinit script:

if [ -f /etc/sysconfig/clock ]; then
. /etc/sysconfig/clock
if [ ${CLOCKMODE} = "GMT" ]; then
clock -u -a
else
if [ ${CLOCKMODE} = "ARC" ]; then
clock -A -a -u
else
clock -a
fi
fi
else
clock -a
fi

Changing it to the following fixed it right up:

if [ -f /etc/sysconfig/clock ]; then
. /etc/sysconfig/clock
if [ ${CLOCKMODE} = "GMT" ]; then
clock -u -a
else
if [ ${CLOCKMODE} = "ARC" ]; then
clock -A -a # originally -A -a -u
else
clock -a
fi
fi
else
clock -a
fi

Did I find a bug, or am I being dense? (P.S. I had to fix this in the
original rc.sysinit, too, but I hadn't heard anyone complain about the
date not working.)
