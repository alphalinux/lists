On Thu, 9 Jan 1997, Kevin Osowski wrote:

> I have had this problem with a 2.5 gig drive. It seem that there is a
> problem with fdisk. I took a look at the code for fdisk and found a
> #define for the man size of sectors and it was 63, and you need 109, I
> needed 225 or something. So there is a problem with fdisk. I have been
> told that you can make this work some how, but I was unable. It had
> something to do with giving it a larger heads count so the total disk
> size came out to the right size. I instead fdisked it on another machine
> and them put it on my alpha. It was the wimpy way, but time was an issue.

I had the same problem with my 2.0 GB HDD. When partitioning with fdisk I
used heads=64 and sectors=32 even though my HDD only has 4 heads. The
reason, from the Large Disk mini-HOWTO:

If you are not using DOS or so, then avoid all extended translation
settings, and just use 64 heads, 32 sectors per track (for a nice,
convenient 1 MB per cylinder), if possible, so that no problems arise when
you move the disk from one controller to another.

...

What is the real geometry? The easiest answer is that there is no
such thing. And if there were, you wouldn't want to know, and certainly
NEVER, EVER tell fdisk or LILO or the kernel about it. It is strictly a
business between the SCSI controller and the disk. Let me repeat that:
only silly people tell fdisk/LILO/kernel about the true SCSI disk
geometry.
