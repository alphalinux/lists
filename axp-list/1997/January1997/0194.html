On Tue, 7 Jan 1997, Steven N. Hirsch wrote:

> > > while doing make, i get the following error when it is in the sunrpc
> > > directory.
> > > LD_LIBRARY_PATH=..:../elf:../nss ../elf/ld.so.1 ./rpcgen -c
> > > rpcsvc/bootparam.x -o xbootparam.cT
> > > ld.so.1: memory violation at pc=00004cf4 rp=000044d4 (bad address =
> > > 00000000)
> > > make: *** [xbootparam.c] Segmentation fault
> > >
>
> > Don't do this. If you _do_ manage to compile glibc like this, your system
> > will be unusable when you install it. Ld.so _must_ be working for you to
> > be able to execute any binaries.
>
> Interesting, seeing as how I've been running with the shared libraries
> that resulted from the build. Exactly _what_ is the command which fails
> trying to do? It's something like:
>
> $ LD_LIBRARY_PATH=<path_to_what_we_built> \
> <name of ld.so.1 we just built> \
> rpcgen args...

Yes. It's trying to run rpcgen, using the new ld.so and libc, without
having installed them in /lib first. If it doesn't work then either the
new ld.so or the new libc.so is broken, and things will go very wrong when
you install them.

> When the host system being used for the build already _has_ the gnu
> loader available in /lib, why do I need the one that's just been
> compiled?

Because not all versions of ld.so will work with all versions of libc.
You may get away with it some or all of the time, but you may not.

> I'm not saying that I have this figured out, but what IS going on with
> the command as originally written, and why might it be failing on an
> otherwise stable and correctly-installed system?

glibc 1.99 provokes a linker bug, and ld.so doesn't get built properly.
