Hi!

Sorry to re-iterate that once again. -- But I've been asked to upgrade an
EB164 based machine running RH 3.0.3 to RH 4.0. The box has 2x2GB HDs and
a SCSI-CDROM. All this is connected to an NCR53C810 controller.

The box was quite happily running RH 3.0.3 (with about 1 crash once a
week under heavy load). -- Installing RH 4.0 succeeded (after I've got the
new ramdisk.img from Redhat...), but SCSI is rather flaky now so that the
machine is rather unusable. :-(

AFAIK using the "new" (aka, 53C8XX) driver is strongly recommended. The
original RH kernel for EB164 causes a kernel panic whenever there is heavy
traffic on the SCSI bus (i.e. conmpiling the kernel while copying a CDROM
to another harddisk). -- Recompiling with most of the driver's featuers
disabled (forcing normal IO, no disconnect, no tagged command queuing,
force asynchronous transfer) does not really seem to cure the problem. :-(

Maybe someone could help me on the following points:

1) Is there a known Problem with EB164+NCR53C810+"new" NCR driver?
2) Does upgrading the kernel (stock RH 2.0.18 -> 2.0.27) have any
chance to fix the problem?
3) Is there a newer version of the NCR driver available somewhere on
the Net that might do the trick?

Concerning the other hardware: The box also has one of those "DEC Tulip"
Ethernet cards. Milo (2.0.18) also gives me the infamous "NCR dead"
message every now and then.

Here is an error log of what I get when the problem occurs:
--------------------------------------------------------------
SCSI disk error : host 0 channel 0 ide 0 lun 0 return code = 2
scsidisk I/O error: dev 08:02, sector 327780
Kernel panic: EXT2-fs panic (device 08:02):
ext2_read_inode: unable to read i-node block - inode = 41312, block =
163890

scsi: aborting command due to timeout: pid 18343, scsi 0, channel 0, id 0,
lun 0 write (6) 00 2e 82 04 00

ncr53c8xx_abort: abort call
--------------------------------------------------------------

BTW: Any chance that the "PCI_BRIDGE_OPTIMISATION" option does somehow
influence that problem?

Oh yes, concerning SCSI termination: It is one of those large server
cases. I've not bothered to open it but I've been told that termination is
ok on that machine (and it has worked quite well under RH 3.0.3).

Once more, sorry for bringing that topic up again, but I thought that
"this will never happen to me" -- Until someone came along and asked me to
fix that problem for them... ;-)

thanks for your input

Christian
