To implement Richard Henderson's suggestion (which sounds
reasonable to me), change the program as follows:

PROGRAM EULER
DOUBLE PRECISION GAM, FAC
GAM = 1.0d0
FAC = 1.0d0
NMAX=100000000
DO I=2,NMAX
FAC=FAC+1.0d0
GAM=GAM+1.0d0/FAC
ENDDO
GAM = GAM - DLOG(dble(NMAX))
WRITE(*,*) GAM
WRITE(*,*) 'Exact: 0,57721566490153286060'
END

This is indeed a transformation that's not done by g77 or gcc (nor
by f2c in the translation from Fortran to C). I seem to remember
that SGI's Fortran compiler has an explicit option to _prohibit_
such an optimisation. I belief, though, that in most ordinary
Fortran code few conversions from INTEGER to REAL occur.

> Tests were done with -O3 on both gcc and f77. I got on

This is actually not fair with respect to f77; its highest
optimisation level is -O5; on the other hand, gcc -O3 -funroll-loops
might result in a better figure for the gcc timing.
