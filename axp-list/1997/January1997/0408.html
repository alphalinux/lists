I got the funny feeling I'm gonna feel like I have egg all over
my face when I hear the answer to this one, but I got an Enorex 500mhz
that I'm trying to install Red Hat 4.0 on, and it just doesn't seem
close to happening. A quick recap of what I've gone through so
far this week:

Pulled down a full redhat axp distribution another campus machine,
which got it from ftp.redhat.com. Created the milo bootdisk, plus the
pc164 kernal image, plus the ramdisk, using images under ...axp/images.
(let me just insert this isn't my first linux install, been around
linux for 4 years, so one could say I have a clue of what's up, at
least with x86 stuff)

Anyways, load up milo on the machine, insert kernel disk, that loads
up no problem. (using boot fd0:vmlinux.gz root=/dev/fd0 load_ramdisk=1)
Kernel boots, asks for the ramdisk image, I toss that in, ramdisk loads,
redhat install procedure starts. Asks if I'm using color or b/w,
answer color. Tell it I want to do a full install over NFS install,
enter the various numbers correctly.(I know they're right, 'cause it
*is* mounting stuff..more on that later)

The install program then asks me if I have any SCSI stuff. I say yeah
man, and tell it to autoprobe for an Adaptec 2940. It comes back with
"I cannot run /bin/insmod: No such file or directory." Well now, that
doesn't sound good. I pop over to the second console, find that the
ramdisk seems to be fairly bare. Luckily I have a friend on campus
with an alpha running linux, so I get an account on his machine, ftp
insmod over to my machine, copy it into the axp directory that's being
NFS mounted. Then over on the install target, I copy
/tmp/rhimage/insmod to /bin, and tell it to try again. Now I get
"I can't find the device anywhere on your system!" Turns out some
of the libraries that insmod need are missing, so I copy those over.
Then there's nothing in /modules. Compile up a 2.0.18 kernel and get
the aic7xxx.o module, copy that over, doesn't seem to fit with kernel.
So I compile a whole new kernel, and try to replace vmlinux.gz on the
kernel disk made from the pc164.img image. Now that disk won't boot.

Sooo, cut to the chase...if anybody could tell me why my ramdisk images
are screwed up, or if you could tell me how to get a new kernel onto
that kernel disk so I can boot with the 2940 support, I'd be
veryveryvery greatful.

On a sidenote(I should probably ask this on the SMP list...) has anybody
got linux running on a dual alpha box yet??
