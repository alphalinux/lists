Actually, I *might*, but first a few questions:

Any chance you have a real old version of the ARC console?

Do these messages occur under MILO as well as the kernel?

See, on the PC164, the PCI probing code *MUST* depend on taking a machine
check when probing a PCI "slot" where no device lives, otherwise the data
it gets back *may* be erroneously interpreted as valid. Thus, the messages
indicate it thought a device existed and was asking for a *bunch* of memory
space on the bus.

If the core logic chipset (CIA-2 on PC164) is incorrectly set up, it will
*not* machine check when it should, leading to the results you've seen.

Now, the above analysis was the result of my seeing just such messages
when attempting to boot a PC164 kernel directly from SRM console, ie not
via MILO. I've never seen them on the PC164 when booting from MILO, but I
may have always had a relative recent ARC console version (which *does*
set up the core logic correctly), thence my first question about how old
yours is...

And, because MILO uses the same code and depends similarly on machine check
for validating devices on the PCI bus, I'd expect the same thing to happen
there.

Now, there may be something else going on, but this seems a plausible
explanation. If you *do* have old firmware, think about updating to the
latest available, which you should be able to find somewhere on the Digital
Semiconductor webpages (sorry, I'm at home and my "bookmarks" are not... :-).

Good luck, let me know how you make out.
