Indeed. The REAL problem is most likely in so called ISA-PCI
bridge chip that is on those boards, altough also core-logic
chipsets might yield problems of their own.

The problems at installations occur every time people are using
ISA-bus interfaced devices to DMA data in, right ? If they had
floppies/CDROMs/disks only at PCI-bus resident SCSI interface,
they would not have these problem (except perhaps with more than
1GB of memory..)

The real problem ? The ISA-PCI chip supports DMA into 64 MB
space of memory, and if your memory usage grows over that, it
wraps silently down to zero again, and ...

A year and a half or so ago I had these problems with XL/233
machine with 128 MB of memory.

By looking at 2.1.42 sources (arch/alpha/config.in, and
arch/alpha/kernel/apecs.c, arch/alpha/kernel/bios32.c) I see a
plenty of references to CONFIG_ALPHA_XL, but none yield to
an easy modification, because the 'Alpha system type' configuration
prompt is 'one out of many' chooser...

Perhaps there should be a config prompt for -- say:

bool 'Does the system have limited PCI<->ISA DMA-address space ?' \
CONFIG_ALPHA_ISA_DMA_LIMITED

doing what the CONFIG_ALPHA_XL does with regard of DMA
address spaces, and memory access areas. (Although it is
currently very APECS specific, and PC164 uses CIA core-
logic.)

Even more than that, I would like to see autodetection of
possible PCI<->(E)ISA bridge chips, and have system to configure
itself according to those -- plus some MILO option that can
set the mode on (or off) in case the possible PCI<->ISA bridge
is not (yet) known, but is nevertheless causing problems.
