upgrading to RH 4.2 showed some problems:

trying to upgrade the XL266 from CDROM was a major pain, because
the TOSHIBA XM-5301 cdrom drive needs ~5+ seconds to "initiate"
after a SCSI bus reset. as a result it was not yet ready
when the kernel tried to mount the root fs from CD and it died.

here are the SCSI messages for normal startup:

ncr53c8xx: at PCI bus 0, device 6, function 0
ncr53c8xx: unit=0 chip=810 rev=0x1 base=0x51000000, io_port=0xb400, irq=11
ncr53c810-0: restart (scsi reset).
scsi0 : ncr53c8xx - revision 1.18d
scsi : 1 host.
Vendor: DEC Model: RZ26L (C) DEC Rev: 442D
Type: Direct-Access ANSI SCSI revision: 02
Detected scsi disk sda at scsi0, channel 0, id 0, lun 0
Vendor: SGI Model: IBM 0662 S08 Rev: 1S11
Type: Direct-Access ANSI SCSI revision: 02
Detected scsi disk sdb at scsi0, channel 0, id 2, lun 0
Vendor: TOSHIBA Model: CD-ROM XM-5301TA Rev: 0925
Type: CD-ROM ANSI SCSI revision: 02

scsi : detected 1 SCSI cdrom 2 SCSI disks total.

SCSI device sda: hdwr sector= 512 bytes. Sectors= 2050860 [1001 MB] [1.0 GB]
ncr53c810-0-<2,0>: FAST SCSI-2 10.0 MB/s (100 ns, offset 8)
SCSI device sdb: hdwr sector= 512 bytes. Sectors= 1221088 [596 MB] [0.6 GB]

my first try was first only to load and initialize MILO (using "ls sda2")
and then power off/on the 2nd harddisk (sdb: SGI/IBM) because this disk
doesn't spin up at power on but only after the SCSI bus reset, so there
is enough time for the CDROM drive to settle, too.

this idea looked ok and I managed to start the installation "2nd stage"
from CDROM, but when I selected "Upgrade" the system died a few seconds
after displaying "looking for packages to be upgraded" or similar
(two times I got a "Signal 11" and "regular shutdown", a few more times
the system just froze). I never managed to get any further.
accessing the 2nd disk from VC#2 was problem, so this doesn't seem to be
the real problem.

my final solution was to load my own 2.0.30 kernel image from harddisk
which was already using ncr53c8xx driver release 1.18d. this ncr driver
release allows to specify the time to wait after scsi bus reset
with boot command line option "ncr53c8xx=settle:10".
please put the config files, which you've used to build your release kernel images,
on the CDROM too. this way it's much easier to build a new kernel image
which is identical/compatible to your installation tools to add some minor
changes or bug fixes only.

it was nasty that the first boot attempt failed again, because the install tool
couln't find /vmlinux.gz on the CDROM (which it doesn't tell me, just talked
about boot_image environment var. or bootline option).
copying the kernel image to /kernels/xl.gz on harddisk avoided this
and I finally managed to upgrade. ufff.

upgrade process went smooth and fast, but after reboot I had to see
that tons of timestamps (creation, modification and last access) are
way in the future (June 2017) because of the ARC clock stuff :-((
fixing all those timestamps for files, links, directories, devices etc.
without clobbering the correct ones was a major pain and really could
have been avoided very easily...
