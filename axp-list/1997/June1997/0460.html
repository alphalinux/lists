Thomas Pornin wrote:

Actually, I believe using 32 bits operands is better than 64 bits ones
on an alpha when only the first 32 bits are used. You may use less
opcodes to fill a register, and the memory accesses are faster (as long
as the 32 bits words are aligned on doublewords boundaries). Assembly
instructions handle well the 32 bits case.

Not entirely true.  32-bit memory accesses take exactly the same amount
of time as 64-bit memory accesses.  The primary advantage to using 32 bits
is that you can pack twice as many of them into a cache block, thus reducing
the number of cache misses.

On the possible downside, the LDL instruction automatically sign-extends
the 32-bit operand into the 64-bit register.  This may or may not be desired
behavior depending on what you are doing (okay for signed arithmetic, not so
okay for pointers or mixed 32/64-bit unsigned calculations).  I believe, though,
that a ZAP instruction will multiple-issue along with an LDL, so if it's important
that you get unsigned behavior you can do it this way with minimal performance
penalty...
