As is probably obvious from my past posts, I am not a DEC Linux guru. I
just did some benchmarks on my system with tuning the IDE disks and
using SCSI and the results may be of interest.

The system:

Aspen Durango, 64MB, 1MB L3, 433MhZ, Quantum Fireball 6GB IDE, Buslogic
BT948, Quantum XP34550S Fast SCSI-II. I borrowed the SCSI controller and
disk for this test.

According to the motherboard specs, the IDE controller is a CMD646 PCI
chipset.

The benchmark:

The AIPS DDT is the benchmark. See http://www.cv.nrao.edu/aips for
details. Basically, it runs a broad subset of the computational tasks
from AIPS on a data set. It is floating point, integer and IO
intensive. AIPS is our main software project and this machine's purpose
in life is to run it well. The code (~100MB of source in Fortran and C)
was compiled with Digital Unix f77 and cc using optimizations and tuned
for the EV5 processor. The AIPS mark is a measure of how fast the
system can run the test, with 4000 seconds being the canonical run (I
believe on a Convex). To get the AIPS mark, take the number of seconds
between the start of the test and the finish of the last task and divide
4000 by it. A 200MhZ PPro system with SCSI disks gets a 3.3. An Alpha
Station 500/600 gets 12 (at 30k$). An SGI Origin 2000 gets 13.7/CPU.
You get the idea.

Testing:

Using the IDE disk un-tuned gets 4.2. Using hdparm to tune the disk to
"-m 16 -c 1 -u 0" gets 4.5 and the machine is unusable. Tuning it with
"-m 8 -c 1 -u 0" gets a 4.5 and the CPU actually has some idle time and
can be used although sluggish and unresponsive. Un-masking IRQs brings
the system back down to un-tuned performance but makes it more usable. I
am concerned about setting multi too high and un-masking the IRQs as that
may be asking the kernel to do too much on the disks when it can be
interrupted. The results with skew of the disk parameters seem to be
based on IO performance/system resources taken from AIPS to get that IO
performance.

The root, /usr and /var filesystems are on the IDE disk. Swap, /tmp and the
AIPS disk are moved to the SCSI drive (gear the test to effectively rely
on the SCSI disk for everything). The system benches out at 9. The
benchmark slows down to 8.6 if I am using the machine at the time (read
e-mail, run netscape, etc... while living in my NFS mounted home dir).
Top reports consistently less than 5% idle time and under 8% used by the
system with the rest going almost exclusively to AIPS tasks. The disk
needed no tuning.

Results:

Going to a decent SCSI controller and a good SCSI disk moved the machine
out of the desktop PC arena and into the mid performance workstation
arena. It was really, really stupid of me to go IDE on this box! An
extra $800 (extra $400 really if we just left off the IDE disk) worth of
parts could have given us an extra 5k$ worth of performance for our
tasks but with 2GB less storage space.
