Just to let you know I'm not disputing whether or not tar can backup over a
network. I just know that not only have I experienced incidents of not being
able to get an archive off a tape, because I wrote it with tar, and it was
corrupt 1/3rd the way in..

Most of the programs that are out there have this problem. I think dump (dd as
well?) will verify the data that was written can be recovered, and that is why
if I used amanda, I would use it with dump.

Now, don't confuse 'tar -tvf /dev/tape' with being able to recover the data
that's on the tape. '-tvf' views the listing of the contents, not the actual
data, and it doesn't verify.

I also realize that it has a --verify option, but I suspect that's a simple
check to see the file it read is the file on the disk.

BRU has a CRC check that it uses, and from their web page:

My tape drive has ECC, why is BRU 2000's CRC so important

In creating a system backup, your backup software copies your data from your
file system into a buffer in memory. From there, the data is transferred
through your system and across the system backplane to the backup device
controller. From the controller, the data is passed over a cable to the backup
device. Once in the backup device's input buffer, the data is then processed
and drive-based ECC is added. BRU 2000's CRC is added when the data is copied
from the fixed disk to the memory buffer. This way, if the buffer is corrupted
on its way to the backup device, BRU 2000 will realize this and inform you. The
drive would have just provided error correction code for the bad data.
