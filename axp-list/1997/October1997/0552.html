>
> There are
> some instances in the code where XtCallbackProc instances are used and the
> old code assumed that one of the parameters, properly a pointer, could swallow
> an 'int' and not burp. Well, its burping.

A pointer can swallow an int, although without a cast will raise
objections and deservedly though. Although a function expecting
a pointer and which got an int will cause a misalignment of your stack.
Casting the other way around may be deadly. If a pointer was really
an address and not a "generic container" then you will loose half
of it.

> I do not understand how the alpha handles explicit (or implicit) int/void*
> casts. Is it doing sign extension? Preserving the lower 32 bits?

If you have doubts then writing a three line program which prints values
in question before and after will tell you the story right away.
Also printings sizeof(a_type_in_question_here) can be informative. :-)

> What I would like to find is an appropriate incantation for casting an int
> to/from a pointer that will work on the alpha, and elsewhere as well.

In general this will never work. Sometimes you can cheat. For a
portable code overlaying types a 'union' construct was invented. I know
that there is a lot of code which does not bother because "this happens
to work here". Bummer!

> Can
> this be done in general, or must I make sure that the 'int' in question is
> typed as a 'long' on the alpha so that it happily can co-exist in the space
> of a pointer.

Pointers are unsigned quantities. You can cast int to a pointer
explicitely. Making ints into longs is, in general, a terrible idea.
Once again - watch out what you are passing to functions. Religiously
supplying prototypes helps a lot.
