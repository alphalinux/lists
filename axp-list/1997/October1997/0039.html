Hey, this is cool. Behold:
------------------------------------------------------------------------
(wasatch)home/cgrayce% gcc ruserok.c
(wasatch)home/cgrayce% a.out wasatch cgrayce
Segmentation fault
(wasatch)home/cgrayce% a.out cgrayce wasatch
ruserok=-1
(wasatch)home/cgrayce%
------------------------------------------------------------------------
"wasatch" is a PC164 running RH 4.2, kernel 2.0.30.
------------------------------------------------------------------------
(flattop)home/cgrayce% gcc ruserok.c
(flattop)home/cgrayce% a.out flattop cgrayce
ruserok=-1
(flattop)home/cgrayce% a.out cgrayce flattop
ruserok=-1
(flattop)home/cgrayce%
------------------------------------------------------------------------
"flattop" is a P5-133 using Slakware Linux, kernel 2.0.30.
------------------------------------------------------------------------
(limantour)home/cgrayce% gcc ruserok.c
(limantour)home/cgrayce% a.out limantour cgrayce
ruserok=-1
(limantour)home/cgrayce% a.out cgrayce limantour
ruserok=-1
(limantour)home/cgrayce%
------------------------------------------------------------------------
and "limantour" is a Digital AlphaStation 600, running DU 3.2C.

Now "man 3 ruserok" under Linux produces zip, but under DU -- bless
those careful Digital engineers -- we're told the prototype is:

int ruserok (char *host,int root_user,char *remote_user,char *local_user);

so from this point of view it looks like your calling sequence is
backward and when it's done right things work. Why the "wrong" calling
sequence segfaults under Linux/Alpha only I've not pursued.
