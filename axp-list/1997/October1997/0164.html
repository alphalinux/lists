I hope that I am not annoying everyone (since I asked similar
questions before), but so far the answers haven't been really
satisfying either. In particular I'd appreciate some
information from the very helpful folks at DEC; I'd at least
like to understand...

Some of you may be aware that in comp.os.linux.alpha
there is a semi-religious war going on regarding the relative merits
of SCSI vs. (E)IDE. I don't want't to start any such discussions
here, I'd just like to know / understand why a pc164 motherboard
(500 MHz CPU, 2MB L3 cache, 128 MB RAM) with a
WDC AC34000L, 3815MB w/256kB Cache, LBA, CHS=7752/16/63
hard-drive is so bloody slow with respect to disk performance and
what I can do about it (one thing I most likely will do is to
get an NCR810 controller in the near future!)

Now, based on hdparm -tT, the performance isn't all that bad if I
first set hdparm -u 0, contrary to the recommendations in the FAQ.
Yeah, of course the clock looses 10 minutes a day in this case, but I
sort of can live with that. If on the other hand you set hdparm -u 1,
the performance reported by hdparm -tT goes down dramatically.
HOWEVER, somehow I don't believe the numbers are very meaningful as
any operation involving lots of disk I/O feel sluggish compared to a
lowly P-133MHz running the same RH 4.2, but having a SCSI disk system.
I don't have hard numbers, but copying one of my trajectory files
(50MB) from one partition to the other (same physical disk) literally
freezes the machine. Response is extremely bad, trying to log into
the machine (telnet) during the copy takes several seconds to get you
a prompt (normally instanteneous), and the copying itself strikes me
as slow (as if I were ftping the file and not performing a disk to
disk copy).

Now, overall I like this machine -- it's number crunching capabilities
are great, since the June (?) axp-patches to 2.0.30 it's actually stable
(not a single hang despite lots of computations), but somehow the
situation is unsatisfactory.

To reiterate, I'd really like to understand all this: What IDE chipset
is on an pc164 motherboard? (There was a post claiming that on the
LX boards, the cmd640 is used, which doesn't strike me as the best
possible choice... I built a kernel with cmd640 support, but there
is no sign that this particular chipset is detected during booting)
hdparm refuses to let me set -d 1 (DMA support) or any of the PIO
options -- is this indeed a limitation of the chipset used or is it
something missing in the alpha port of the IDE-driver / hdparm ???
Should I try any of the pre-2.0.30 kernels with the even newer
axp-patches ???

Any tips regarding IDE appreciated -- even if it's reliable knowledge
that this particular hardware can't be supported any better (at least
then I know).

Also, I/we (our group) hopes to get some money, and a SCSI subsystem for
the Alpha is first on the list of my things to get. QUESTION: Is an
NCR 810 controller sufficient/good, or should I find the extra money
for an Adaptec 2940UW or an IntraServer card. But somehow I don't see
the advantage of wide SCSI for a system that will at max. have two
disks hanging from the controller -- I am only worried about the cabling
issues.

Thanks in advance,

Stefan
