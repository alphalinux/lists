>
> You should compile f2c from the sources at netlib.bell-labs.com (/netlib/f2c) and
> it works now fine, they have already included the patch correcting the above problem.

I work with Ming (who reported that a simple program compiled with f2c
was crashing the machine).

But in any case, if a program compiled by f2c/gcc crashes the machine,
I wouldn't call it an f2c bug. It's either a kernel bug (badly
compiled kernels included) or a hardware problem. And it'd be funny
if it's hardware because we have it on both of our alphas (both 533mhz
164LXs).

The crash looks similar to one reported recently on this list by Shane
DeRidder <shane@together.net>. The console starts spewing out lines
like:

0 0000000000000198 0000019800000110
10 0000000100000084 00000001201dc850
20 000006f800000000 ffffffffffffffff
30 ffffffffffffffff 0000015555cac5b0
40 0000000120535690 00000155557c1db8
50 0000000120534ed2 0000000000000008
...

The machine responds to pings. The keyboard isn't dead (the capslock
key toggles the keyboard capslock light), but the machine doesn't
respond to control-alt-delete, can't be telnetted into, won't switch
virtual consoles...

So, anyone want to try to track down this problem? As Ming said, we
can email you the statically linked binary, or we can upload it to
you, or we can get you all the associated C code (patches against
f2c), or ...
