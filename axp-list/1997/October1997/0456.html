> I presumably am running glibc2 - rpm reports I'm using glibc-0.961212-5
> and glibc-devel-0.961212-5 (rh 4.2). So,

No, that's glibc 1.98.

> 1. Are these compatible with an unpatched kernel 2.0.30? What
> about 2.0.31? What about the 2.1 series?
> 2. Do I need alpha-patches-2.0.30-0.2a.gz, or do I just need it for
> em86? Is it compatible with the above setup?

For glibc 1.98, you need patched kernel headers in /usr/include. The best
thing to do is to install Red Hat's kernel-includes rpm, and forever leave
/usr/src/linux alone. Build your kernels somewhere else.

The only patches that are really needed from alpha-patches are the SYSV
IPC bits, and that only if you care about programs that use SYSV SHM.
Which includes the X SHM extension, used by e.g. Quake.

All the rest is either header file changes for libc, or internal VFS
reworkings for large filesystems. Oh, and new machines.

> 3. alpha-patches-2.0.30-0.2a.gz is pretty new, so I had assumed it
> included all well known bug fixes. I take it this isn't the
> case? If not, is it going to get fixed?

Probably. Jay?

> 4. My /usr/include header files seem to include things from
> <linux/...> and <asm/...>. I thought someone mentioned that
> glibc2 no longer depends on kernel header files. What's the
> story here?

First, you've not got glibc2; glibc 1.98 was working toward *tighter*
integration. Which was a mistake, mostly. But even the latest glibc
uses kernel headers to some extent, though vastly reduced.

> 5. Should I upgrade/change my glibc? Is there an appropriate RPM
> or should I build it from scratch? Will doing so break
> anything?

It's not trivial. Wait for Red Hat 5.0.
