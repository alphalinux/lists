Dave Wreski writes:
>
> I got the impression he was looking for something reliable, and can recover
> when something goes wrong with the archive.

Ahem, gnu tar is pretty good at that. :-) Did you try? Not all tar
utilities are born the same. BTW, this is directly from 'tar --help':

Usage: tar [OPTION]... [FILE]...
...
Device selection and switching:
-f, --file=[HOSTNAME:]FILE use archive file or device FILE on HOSTNAME
--force-local archive file is local even if has a colon
--rsh-command=COMMAND use remote COMMAND instead of rsh
....

and this indeed does work. The only problem can be an expected
location of 'rmt' but this can be resolved with a help of a link
or two. Use 'ssh' for '--rsh-command' if you need a secure connection.

> You've been lucky with tar,

I doubt if luck has much to do with it. :-)

> and
> you can't have volume labels, etc, with cpio...

But you can with afio, I believe. And you can run compression
on a file-by-file basis if you are short on tape.

Whatever. I do not doubt that various backup strategies fit various
circumstances but the question was "what will work over a network".
Most anything, I guess; in the worst case in an explicit cooperation
with rsh or ssh (well, recently I have seen 'dump' which steadfastly
refused to write on a pipe :-).
