On Mon, Oct 06, 1997 at 12:56:20PM -0500, Kees Cook wrote:
>
> Once gcc makes it to "crypt.o", it causes VFS errors, and
> eventually gcc dies with "gcc: Internal compiler error: program cc1 got
> fatal signal 6":

No, signal 6 is abort() -- a plain old assertion failure aka bug in gcc.
I fixed this one a few months ago and posted the patch here. I don't
see it laying about handy or I'd send it again.

> Oct 6 11:55:27 kernel: ext2_free_blocks: nonexistent device<1>Unable to
> handle kernel paging request at virtual address 4857525042000000
> Oct 6 11:55:27 kernel: update(293): Oops 0
> Oct 6 11:55:27 kernel: pc = [<fffffc000031afa0>] ps = 0000

This is something completely unrelated. But you might look up
that pc in the System.map that goes with your kernel and see
where it is coming from. Though I'd hazzard that it is a dangling
pointer, since that address is "HWRPB\0\0\0" which is the magic
number for the hardware parameter block.
