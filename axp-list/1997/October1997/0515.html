I am not aware of any published critiques of the alpha; however, compiler
writers (of which I am one) can usually spot the problems upon a single
reading of the architecture manual. Its been about 3 to 4 years since I
last looked at it, but these are the things I remember.

1. Not byte addressable. This will make it cumbersome and inefficient
to generate code for exisiting apps (e.g. all the vax stuff)

2. missing support for byte and 16bit instructions (e.g. sign extend move)
I believe this was added in subsequent revs (upon insistance of Oracle)

3. Strict alignment requirements. Most exisiting code is not aligned on
natural boundaries (e.g. PL/I structs, Cobol records, Fortran named
common, C aliasing) resulting in many mask and shift operations. A
Barrel shifter in the path between cache and ALU could handle most
unaligned accesses with an extra clock tick.

4. No string support. Maybe not so important for C code but very useful
in PL/I and Cobol.

As a result the strength of the instruction set is somewhat reduced. I've
seen a number of postings where people are comparing the performance of
various packages on both alpha and pentium at various MHz, which I believe
reinforces the point, and probably gives an empirical way (more-or-less)
to determine the strength metric, assuming that the code generators were
written with equal care.
