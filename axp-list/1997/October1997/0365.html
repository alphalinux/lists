Not so much "Everyone else is wrong" as it is "We've got the *right* approach,
and you all don't."

There are some things about the *BSD systems that are "superior" to Linux;
they already have things like logging/journalled and semi-memory-resident file
systems. These are things that it would be *nice* for Linux to have; the
former provides a fast file system that's more robust than ext2; the latter is
very nice for temporary files.

Linux of course has had a "concentration" on having *LOTS* of variant sorts of
file systems, supporting various MS-DOS FAT variants as well as file systems
for all sorts of "obsolete" computer systems.

There are different approaches being taken; I think I prefer the "Linux
approach" particularly in the view of the "Cathedral versus the Bazaar"
approach (see Eric Raymond's essay at URL <http://www.ccil.org/~esr/writings/ca
thedral.html>).

The quick summary is that Linux has seen *rapid* development on its kernel by
enormous numbers of people due to the "open"/"Bazaar" approach taken to kernel
releases where there's a release every week, bugs fixed or not. The *BSDs
have generally taken the "Cathedral" approach, where a small core group of
developers periodically release a "beautiful" version for the rest of the
world to admire.

The "openness" of this approach has encouraged wide participation, and is a
strength of Linux. Which doesn't make the other guys awful/bad. There have
been contributions back and forth:

a) Linux networking code was originally based on that from *BSD.
b) FreeBSD in particular has borrowed device driver code from Linux, and there
have been returns to Linux.
c) Many tools and applications run quite nicely on Linux and *BSD.

There are some disagreements on development approaches; the *BSD groups eschew
the GPL, feeling that it denies them some freedoms that the "Berkeley License"
provides them. I would suggest that *that* is the primary issue separating
them.

The "BSD" argument is that the GPL is a "licensing virus" that forces software
to be "free" whether the authors want that to be the case or not. They would
like for a commercial enterprise to be able to take the code and be able to
"privatize" derivatives. RMS and the FSF considers that to be just a
*horrible* idea; "All software ought to be free."

I think that both situations can reverse themselves:

- Companies will be very happy to see "Berkeley code" as it allows them to
appropriate it for themselves. Commercial enterprises are unlikely to
substantially contribute "free Berkeley code" as they know that others can
appropriate it without them receiving any intrinsic reward.

Why would Digital contribute to a "BSD" project, when a competing company
would be free to grab that code, modify it, and make the results secret?
Digital doesn't benefit.

- Companies would be more willing to contribute to GPLed code as it means that
if other companies do further development, what goes around comes back.

If Digital contributes to GCC code optimization, and Intel takes that and does
further development, those results *must* be then available to Digital.
Everyone, notably including the original contributor, benefits.

In any case, there are some strong feelings surrounding the legalities, which
is a significant barrier to many of those that are "BSD Believers."
