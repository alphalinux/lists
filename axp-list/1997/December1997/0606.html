At 11:11 AM 12/16/97 -0800, James R. Shipman wrote:
>At 04:04 AM 12/16/97 -0500, you wrote:
>>Far and away the largest stability problem with my Linux/Alpha box is
>>errors of the following form:
>> Unable to handle kernel paging request at virtual address <foo>
>>followed by an Oops (usually from a daemon, but occassionally from a
>>user process).
>
>I see this problem too on my PC164LX, and there is NO WAY that it is a disk
>failure!

I am helping one of our Maths people who is seeing this sort of kernel oops
rather too frequently on his XL-366. A "simple" amount of debugging with
gdb lead to the conclusion that there was an error somewhere in
linux-2.0.32/fs/proc/array.c within the function get_array. This problem is
triggered by running top which examines /proc to get system information. So
this type of kernel paging fault does not necessarily mean that the disk is
bad. Of course, it could be the cause here, you get interesting effects if
you swap code or data out and something different comes back from a bad
disk....

I've got the code at home and am trying to find out which bit of c is
failing. I've got the alpha assembler code, the c code, the guide to alpha
assembly language program - all I need is the time to look at it. So far
this week our Student Enrollment system has hiccuped and the finance
database has corrupted itself - for some reason my boss doesn't see linux
kernel debugging as a high priority....
