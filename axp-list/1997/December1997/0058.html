>>> Carlos Carvalho said:
>
> In the MILO howto there's a big description of how much MILO does,
> from PAL code setting to memory management and video initialization.
> However, in the srm howto, if you choose to use the raw loader,
> nothing of this is done. So, how can the kernel work in these two
> completely different environments?

In the second case, all that stuff indeed *has* been done by, or is provided
by, the SRM console itself. Nearly the same BIOS emulator is included in SRM
console, along with PALcode which is roughly the same as the EBSDK PALcode
provided by MILO, and finally, the memory management that MILO does closely
resembles that done by SRM.

In fact, SRM was the *first* Alpha console used by LINUX (and Linus :-).
It was on a JENSEN and since no EBSDK PALcode has been (or ever will be)
available for that platform, there was no other choice but to depend on
the resources provided by the SRM console (use of NT PALcode is NOT an
option! :-).

> Also, I've just (for the first time) compiled a kernel on the alpha,
> and the SRM booting option in make xconfig was unavailable. Why?

Because both "xconfig" and "menuconfig" use different scripts from the basic
"make config", and they are not automatically updated when changes are made
to it.

> I'd rather use a raw boot process than a MILO one, because it does
> look much simpler...

Simpler perhaps, but less flexible (IMHO) because of some drawbacks; since
there's no filesystem on the floppy, when you want to change the kernel
image on there, you have to re-write the whole thing, or have multiple
floppies if you want to choose between kernels; and floppies are slow, and
rawboot isn't as "nice" to use on harddisks.

But, it's your choice...
