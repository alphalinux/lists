This was a tongue-in-cheek comment. Sources in question use difftime()
to compute "Real time". This function is documented to return time
in seconds and does not seem to return any non-zero fractional part -
even if a return type is double. To me it is far from clear from a
documentation if this is a library bug or not. The above result likely
really means that "Real time" was below 6.5 seconds.
