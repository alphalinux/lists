>>> "Jim. Shipman" said:
>
> I'm trying to boot a PC164 from a bootp server and mount an NFS root. I
> can boot from floppy with milo/ARC and NFS root works, but I just can't get
> the system to boot the kernal using bootp and SRM! I've read all the docs,
> howto's, minihowto's etc. to no avail. Does anyone know how to get this to
> work?

First off, it will be much easier to forgo MILO and just boot a kernel
directly from SRM via BOOTP. You'll need to build the kernel specifying
that SRM console will be used, during "make config".

Second, to build the BOOTP image, do "make boot" followed by "make
bootpfile", then look for the BOOTP image as arch/alpha/boot/bootpfile. For
this, you'll need to use the alpha-patches-2.0.30-0.[12].gz that are
available on gatekeeper.dec.com, applied to stock 2.0.30 sources (*not*
the RH kernel sources RPM!!!).

Third, you'll prolly need to fiddle a bit with the "ewa0*" envvars, so that
the correct protocol and such are used. At minimum, you'll need to do:

>>> set ewa0_mode <whatever-is-correct>
>>> set ewa0-protocols BOOTP

> So far I can't even get aboot to work from a floppy. It uncompresses the
> kernal and starts initializing, but it can't find any floppy controllers
> (its there), and when it gets to the SCSI interface it again looks like it
> initializes correctly, but can't find any SCSI devices and hangs looking
> (they're there too).

It's the kernel that can't find the floppy and SCSI disks, right? That's
because, if the kernel is built for running under MILO and you run it
directly from SRM (via "aboot"), the interrupts will be delivered in a way
that the kernel isn't expecting.
