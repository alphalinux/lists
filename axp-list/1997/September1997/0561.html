> Ok, that's a possibility. I have two questions:
>
> - When I declare a variable as volatile (which I do for all pointers
> to hardware), shouldn't caching be disable for this piece of code?

No. Volatile is actually a bad way to go about this; you
really need explicit memory synchronization primitives.

Unfortunately, there are no such explicit architecture
independant macros for such in the kernel currently. What
there is is hidden behind the in/out/read/write macros.

There is also an mb() floating about the alpha asm headers.

> - Does anybody there know about the Alpha specs in regards to the
> caching issue Jim is talking about ?

Please see the Alpha AXP Architecture Handbook (available as
pdf from ftp.digital.com), sections 5.2 (Physical Address space
characteristics) and 5.6 (Read/Write Ordering).

What it may boil down to is that unless you issue an explicit
mb(), your writes may be sitting in a queue waiting for either
bus time, or to be coelesed with other writes to the same line.
So the card may not have seen the write before the read, so it
may not be in the state you think it is in.

It is really hard to say without seeing the actual code and
knowing what the card will tolerate.
