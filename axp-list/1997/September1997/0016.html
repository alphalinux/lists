Ronny Ranerup(ronny@axis.com) wrote:
... stuff deleted
>I just got an AlphaPC 164LX, 533Mhz. Slick machine... Twice as fast as a
>PPRO when running gcc and three times as fast running dhrystone (yes a
silly
>benchmark, I know). 530.57 BogoMips...
>
>Anyway, we bought it to try it out as a server running our
ASIC-simulators,
>but of course we didn't get the kind of speed we wanted. In fact the first
tests
>showed it to be a tad slower than our PPROs. Sad. We knew beforehand that
all
>the byteaccesses in the simulator might mess up the Alpha and it seems we
were
>right. So we changed all bytes into ints and tried again. 20 percent
faster, although
>the code got half the size of the byteversion and even smaller than the
PPRO one.
>Cache trashing or something? I don't know quite what is happening.

The Alpha processor is a 64 bit machine. Most arithmatic and logical
operations
occur using 64 bit values in the registers, and memory writes of less than
64 bits requires a read and merge operation. Because the compilers define
the int size
to be 32 bits the intermediate results require extending and truncating, so
using
ints is simular to using bytes. If you want the best performance try using
long
ints (64 bits).

>I'm not sure but I think the CPU has instructions for byte access, but
>gcc doesn't use them yet. I've heard that gcc for Alpha makes bad code
>so I would like to see if DEC cc does a better job.

I've gotten faster code (heavy byte manipulation) using gcc, although I
haven't
used the newest DEC compiler, still at 3.0B.
