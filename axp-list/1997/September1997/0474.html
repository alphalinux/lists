I think you missed my point. The mission statement of the design team
was flawed as evidenced by the fact that later revisions have added new
features such as byte load and store, and I am happy to hear unaligned
loads and assume stores. There are really two problems, (1) there is
about a couple of hundred billion lines of code in production most of
which is unaligned and (2) most languages provide for some type of
aliasing of data types which could lead to unaligned code. Even if the
compiler did a better job of detecting aliasing it doesn't mean that you
could align things (e.g. fortran named common). IBM new this when they
designed their chip so that unaligned access cost is only one clock tick.

On the first alphas, 16bit assignment of the form c = a + b; took 19
ticks, but 32bit only 4 ticks (if I remember correctly) that in my
view is a design flaw.

So you really have to live with unaligned acces, of course you always
have the option when writing new code that you can more carefully align
things.

> what they were doing. Any modern memory is organized in words (often 4

> Sometimes, however, unaligned loads are necessary. In particular,
> the Ethernet packets, which are delivered by hardware as a binary block,
> have internal parts that aren't aligned. If one just grabs them,
> one ends up with an unaligned load. Of course it is possible to copy
> them byte by byte to an aligned buffer and access it there, but this
> requires special-case code in the driver, which wasn't done.
