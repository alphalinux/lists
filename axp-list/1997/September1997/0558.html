> My interpretation of gnu, gcc and glibc faqs is that user apps should never
> call include/linux tree, and that where a gnu/something.h exists, you should
> use that instead of something else, ie use gnu/types.h instead of sys/types.h.

Correct about not using anything from <linux/*.h>. But
<sys/types.h> is in fact the correct one to use.

We got rid of (moved) <gnu/types.h> for glibc 2.1 to clear up
this confusion. All of the private, do-not-touch, libc internal
only files now reside in <bits/*.h>, and all bear a legend and/or
#error message reminding the user of this fact.

> FWIW 2, my powerpc linux box seems to have much cleaner headers
> and I was thinking I'd try to reconstruct based on the dependancies
> there.

You're probably using glibc 2.0 or even 2.1 there, no?
