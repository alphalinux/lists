Hi, sorry about the delay in answering you; are you still in need of help?

>>> Ivan Passos said:
>
> I am porting the Cyclades-Z multiport card driver to the Linux/Alpha
> and I'm having several problems with that, due to the card mapping
> (it's a memory-mapped card ...). Also, this card is an intelligent
> card with an MIPS3000-compatible RISC processor.

By "memory-mapped", you mean the card has local memory which is seen
via PCI bus memory space? These are not memory-mapped registers, are they,
as opposed to just "memory"?

> The only thing I'm currently trying to do (in regards to transmission)
> is read the tx_get, the tx_put and the tx_bufsize from the dual port
> memory and doing some calculation (to check the available space in
> the transmission buffer). If I comment out this line, the machine does
> not halt. If I uncomment it, the machine locks completely. This is a
> pretty basic operation and I don't understand why it is locking the
> machine.

Well, it's seems that you're not accessing the local memory on that card
correctly, I'd guess. It could be either the addresses you use are not the
necessary ones for touching PCI memory space, or are not aligned correctly.

You can read the PCI config space base registers of the board, and get the
assigned memory addresses, right? Is this the base address you use in any
attempt to access that local memory?

You should be able use the kernel's readb/writeb/etc routines to access that
memory, I'd think, just by providing those routines with the PCI address
you wish to access; they'll do the magic that's necessary for accessing
that local memory.

> if there is any detail about bus arbitration in the Alpha that I don't
> know, it would explain the machine lockups).

This is probably not your problem.

An example of use of readb/writeb/etc may be found in:

drivers/scsi/ncr53c8xx.c
