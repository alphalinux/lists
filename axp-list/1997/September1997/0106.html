> Has anyone sucessfully built XEmacs 19.15 or 20,2 for Linux/Alpha?
>
> I'm attempting to build it on a fresh RH 4.2 install only to have it
> crap out with hundreds of truncation warnings and cast errors.
> Suggestions?

Well, I haven't tried to build it from scratch, but I found a
pre-built version (20.2) on redhat's site in the alphabits
directory. I untarred it from /usr/local. However, when I try to
run it, I get the following:

>>>

WARNING: couldn't find an obvious default for load-path,
doc-directory, and data-directory, and there were no defaults
specified in paths.h when XEmacs was built. Perhaps some
directories don't exist, or the XEmacs executable,
/usr/local/bin/xemacs is in a strange place?

Without both exec-directory and load-path, XEmacs will be very
broken. Consider making a symbolic link from
/home/mrl/lib/xemacs-20.2/alpha-unknown-linux to wherever the
appropriate XEmacs exec-directory directory is, and from
/home/mrl/lib/xemacs-20.2/etc to wherever the appropriate XEmacs
data-directory is, and from /home/mrl/lib/xemacs-20.2/lisp to
wherever the appropriate XEmacs lisp library is.

Without lock-directory set, file locking won't work. Consider
creating /home/mrl/lib/xemacs/lock as a directory or symbolic
link for use as the lock directory. (This directory must be
globally writable.)
Segmentation fault (core dumped)

<<<

and I get the following in /var/log/messages:

>>>
Sep 7 11:47:16 delenn kernel: xemacs: memory violation at
pc=1555690fa40 rp=155568f929c (bad address = 00000002)

Sep 7 11:47:16 delenn kernel: xemacs(897): unaligned trap at
0000000000000025: 47ff041f47ff04d7 29 27
<<<

I know that I've probably missed some small step somewhere, and
that I'll say "Doh!" when I figure it out, but if anyone out
there has gotten this to work, please let me know.

Oh, BTW, I'm running a stock RHL 4.1, all updates applied, and
kernel 2.0.27.
