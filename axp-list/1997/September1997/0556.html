> It seems that pid_t is undefined by default on linux/axp, even though
> the man pages say it should be defined in unistd.h:

The man pages are wrong; pid_t shall be defined in <sys/types.h>.

As for Linux/Intel, you are probably using libc5, which isn't
very good about following the namespace rules of the various
standards, and so you get the observed bleed-over.

Current glibc is very close to conformant with ISO C 89,
ISO 9946-1996 (POSIX.1) and XPG4 (aka Single Unix Standard).

> Finally, if I add #include <sys/types.h> to the top of the code (as
> Donald Lewine's POSIX Programmer's Guide indicates is needed), the
> code compiles without problem *without* -D_LOOSE_KERNEL_NAMES, and
> causes *many* warnings when compiled *with* -D_LOOSE_KERNEL_NAMES.

Ignore _LOOSE_KERNEL_NAMES and __KERNEL_STRICT_NAMES. There are
quite a few problems with libc headers and kernel headers in the
glibc 1.98 19961212 snapshot.

Glibc 2.0 (stable) and 2.1 (development) fix the bulk of these
problems, hopefully all of them. It is kind of a pain to upgrade,
but you may soon be able to try out a beta of Red Hat 4.8 that
will hopefully make this less traumatic.
