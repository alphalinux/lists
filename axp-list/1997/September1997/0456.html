Tom Linden wrote:

> This is not true with the Power PC, BTW The alpha design team didn't
> really know what they were doing.

I wouldn't go so far as to say *that*! The Alpha design team made a
particular set of design decisions, all of which were focused on making
the core CPU as fast as possible. Sometimes this resulted in pushing
boundary cases off onto the programmer. Other architectures decided to
make things more convenient for the programmer. What may surprise you
is that nearly *ALL* architectures exact a penalty for unaligned data
access. They don't always *tell* you about it (i.e. they silently take
extra cycles to do the unaligned access). Alpha merely makes this
explicit.

> In any event to answer your question,
> its the job of the compiler to properly align things. If you get this
> message then its really a compiler bug, i.e. the compiler that was used
> to create the object module, which when excuted causes the trap.

Unfortunately, a C compiler cannot always detect this... especially
if you're doing things like pointer arithmetic. And there's a LOT of
that in kernel code!
