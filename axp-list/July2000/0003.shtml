On 01-Jul-00 Jeff Sturm wrote:

> I'm curious... why use MILO at all now that SRM is freely available for
> the 164SX? MILO has several serious disadvantages. Also, why the need
> to upgrade MILO? Any version that recognizes your machine should work.
> It doesn't need to match your kernel version. If the PALcode had
> changed however, it could be worth upgrading, but isn't MILO still using
> the old sample PALcode from the EBSDK?

Yes. Milo still uses that PALcode (with some changes). You need to update your
Milo, if you have i.e. a harddisk controller that wasn't supported by 2.0
kernels, or if you need the initrd feature.

> I'm not trying to flame here, I'm just honestly wondering why people
> still need/want MILO.

If there's an SRM version for your platform, there's no reason to keep
AlphaBIOS and Milo, except you maybe want to boot NT ;-) And, of course,
the partition table of an existing installation would have to be changed
if you can't/don't want to reinstall the machine.
Though, there are old machines without an SRM console. These will have
to stay with the milo solution.

Regards,
