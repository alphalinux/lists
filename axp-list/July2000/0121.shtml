W Bauske wrote:

> A question. I've got a machine giving me occasional machine checks:
>
> TSUNAMI machine check: vector=0x630 pc=0x12000d5c8 code=0x100000086
> TSUNAMI machine check: vector=0x670 pc=0x120032660 code=0x100000098
>
> In those PC and code numbers, does anything tell me which processor
> is having the cache/TLB problem? It would be nice if the machine
> check code could determine what processor in an SMP setup is getting
> the errors. This of course assumes only the processor that really
> had the machine check also services the machine check. Is that a true
> condition? Then all that's needed is the machine check code would have
> to figure out the processor it's on. Any thoughts appreciated. I'd
> like a more precise message so I don't have to start swapping processors
> around to figure out which one is sick.
>

I believe Jay once mentioned (here I go again, quoting Jay...;) that
mchecks are asynchronously delivered, except for 660 & 670. This would
mean that the CPU servicing the mcheck may not be the one that caused it.
If my memory hasn't completely left me, with 660 & 670 being the exception
to this, you should be able to get the CPU number from within the mcheck
code and dump it out if it's a 660 or 670 vector. Of course, trying it
doesn't hurt, I'd imagine. ;) Worst thing should be that it randomly jumps
between processors and you haven't gotten any further.
