Hello Ricardo et al,

} I've mounted a new system with the microway screamer sx164 533Mhz
} and I'm having my file system corrupted on the SCSI disk.
} I have one adaptec 2940 and a quantum SCSI disk, and
} I've also tested with one 2940Wide and a IBM SCSI wide.
}
} The IDE disk works fine.
} On the SCSI disk, I can format a partition , copy files to it,
} and everything looks OK.
} The problem is that after the system reboot, the partition
} gets corrupted...

When running 2.0.30 (from Jay's sx164-31oct-2030+.gz kernel)
I get a good stable system on my "PC164SX/533, 2940UW, Quantum
viking UW scsi disk as /, other scsi disk, scsi cdrom". When I run
a newer kernel (self compiled), I get file system corruption when
syslogd is start (a message about exception indicates the moment).
Delete the offending line from /etc/rc.d/init.d/syslogd, and at
least you have a well booting system. Currently this is my solution.

Alternatives to the above problem are reported to be upgrading
from RedHat 4.2 to RedHat 5.0, or perhaps compiling the sources
for yourself (I didn't succeed in that).

Alternatively, there is a different problem: mine too :-(.

After running a recent kernel (e.g. 2.1.96 or 2.1.101) for some
time (days or so) I get scsi errors about timeouts and other nasty
things. One or two messages appear to be recoverable, several
message often result in a fully hung system, with corrupted but
luckily sofar still repairable file systems. I have no solution
for this problem, yet. I trying to find out whether this is kernel
related or (I hope not) hardware related.

Just trying to be informative.
