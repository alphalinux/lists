On Mon, 22 Jun 1998, Paul Tomblin wrote:

> Quoting Christopher Curtis (ccurtis@ee.fit.edu):
> > On Mon, 22 Jun 1998, Michal Jaegermann wrote:
> > > Agreed. Unfortunately X, for example, uses pointers all over the place
> > > as "universal containters" and there is no good way to change that.
> >
> > ... yet another reason to replace X with a more modern system.
>
> If there were such a thing...
>
> Note that in order to produce such a thing it would have to be:
> - network transparent
> - portable as all hell
> - upwards compatible with X servers/clients.
> - truely object oriented instead of the half-assed "objects in C" like Xt and
> Motif currently do.
> - Good support for accelerated hardware without cutting out "normal" hardware.
> I think you could achieve something like this with a new API that still spoke
> X protocol underneath, providing it wasn't any slower than raw X. Implement
> it in C++ and Java to start, then turn it over to the community to do "my
> favourite language" bindings.

It just might happen, now that XFree86 and TOG have split... now they are
"free" to continue to develop X11R6.3 into anything they want.

X certainly could use an overhaul. But starting from scratch would a
very, very bad idea. There's a lot of good code in there already.

(I have a side project to reimplement Xlib in pure java. I was suprised
at how many simple optimizations can be made... Xlib is a very ugly library.
The low-level X protocol is basically sound, it's just missing a few
essential features of modern window systems, like matrix transformations
for one...)
