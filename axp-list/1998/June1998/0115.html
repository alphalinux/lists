Greg Lindahl wrote:
>
> > Is this an acceptable solution to making programs 64 bit clean and is it
> > portable across 32 bit and 64 bit platforms?
>
> This is the easiest way to fix broken code. However, it isn't
> necessarily portable to other 64 bit machines -- I think everyone
> else (?) defines long == 32 bits and long long == 64 bits.
>
> Your best bet for a clean, easy solution is to typedef an int64 type
> and use that, with an appropriate definition for each machine. If you
> are clever you can put that into a configure script...

If you have to do it, long has been the traditionally safe way to
convert between integers and pointers. I can't find where I read that,
but the ARM said in sec. 5.4, "A pointer converted to an integer of
sufficient size (if any such exists on the implementation) and back to
the same pointer type will have its original value; ..." I haven't
heard of any platforms for which long is smaller than a pointer; long
long is not yet supported on some compilers, so I wouldn't rely at all
on its portability. So I use long, when necessary. Of course, I do it
through a typedef (iptr) that I define in a header that has other
machine-dependent typedefs. I wouldn't waste space and performance by
forcing 32-bit pointers through an int64.
