On Mon, 22 Jun 1998, Paul Tomblin wrote:

> For the purposes of discussion, lets call this new windowing system "Y"
>
> Quoting Christopher Curtis (ccurtis@ee.fit.edu):
> >
> > > - upwards compatible with X servers/clients.
> >
> > Not necessarily. It will need a X server client of it's own (that is,
> > it's own version of eXceed).
>
> I'm confused. You mean a client that runs on an X server that implements a
> "Y" server? Because you'd need some way to make sure the thousands of X

Nope - it will need a program that runs under "Y", that *is* an X server.
This would allow X clients to talk to the X server, which is itself a "Y"
client. Just like eXceed is an X server for "Y"indows.

> clients still run, and you'd also hopefully want to still run on all those X
> servers out there. Do I have to mention my NCD X-terminal here?

Your X server would be obsolesed. It would still run X programs, but not
"Y" programs. Unless, of course, someone wrote a "Y" server for X.

> > Absolutely under no circumstances whatsoever. ANSI-C *only*! This is one
> > of my biggest gripes with the Berlin people. The next is the registry.
>
> Registrys are for idiots, but I disagree entirely about OO. There are a lot
> of things that OO does really badly, but windowing systems is one of the
> things it's good for.

I do not discount the benefit of OO - as long as you do not discount the
bloat (at least of C++). I like the whole OO *idea*, but the
implementation thus far is relatively poor. That's why I don't mind so
much the idea of Java. It is a decent language.

> Xt is a half-assed attempt to do OO in C. It's not bad, but I've seen better.

Well, if it's not bad, it's still usable. The same could be said that X
is not bad, and I don't mean to throw out all of X, I'm sure a lot of code
could be reused, but a distinct portion (if not majority) is severly
lacking. This is not the same for C, though. C does OO "relatively" well
for a proceedural language, and I see no reason to throw this out. Most
*nix code is C, and there's no need to do C++-isms when you can strongly
type the structs to be so closely OO-like.

> > me to go into the details here, but the biggest failing is in colormap
>
> But I think that's an API problem, rather than a problem with the underlying
> protocol.

API? It is an intrinsic problem of colormap handling, which is so poor it
can't even be classified as half-assed.

> > handling. Run X in an 8-bit display sometime for a while. Under Solaris
> > anyway, colors disappear in time. Netscape (with the -install option)
>
> I run an 8-bit display all the time (UDB with TGA, remember?)

No. :)

> A lot of programmers aren't very good at managing public and private
> colormaps, and so they consume the whole colormap pretty fast. I have
> strategies for that (mostly avoiding running Netscape until *after* running
> anything color intensive).

But that's not really a solution. Windows runs decent in 4-bit color.
8-bit color is marvelous. Under X, 8-bit is a bare minumum and even then
you have to put up with colormap flashing. Admit it, it's horrible.

> > The next problem of X is the bandwidth. X is virtually unusable across an
>
> Well, you've already mentioned LBX. I think it's good that X is extenisible
> in order to add stuff like that, and still remain backwards compatible.

Extensibility is fundamental, but the X "extension" method itself is a bit
of a hack. Another problem is with printing which has also been addressed
in recent versions of X, but these are truly hack-ons, and not part of
what should be integral.

Does X-Free even support the printing extensions? I know they don't
support multiheadedness (yet?)...

> > defaults mechanism is also needed. There should, IMHO, be three
> > configuration files for each program - one in .../app-defaults, one in
> > .../app-defaults/fixed, and one in the user's home directory - just like
> > the setup for Pine. There should be *NO* registry.
>
> That sounds pretty much like the standard set-up to me. Just set up the
> system wide XAPPLRESDIR, and you're golden.

I should explain. :)

The three files would be:
../app-defaults/fixed - settings that can never be overriden
../app-defaults - default settings that may be overriden
~/. - user overrides

This was most benficial in Pine where the domain name can be fixed and not
changed by users under any circumstances. I'm not sure if any other
programs need it except maybe xscreen{saver,lock} but if the file doesn't
exist, then it simply falls back to compile-time options.

But it is pretty much in place. The only thing I add is the "fixed" file.

> > To be honest, there are some things for X to learn from Windoze. The
> > WinFrame stuff is pretty efficient (of course, done so at the expense of
> > configurability, but nonetheless...)
>
> The MAIN thing I hate about Windows is the lack of an equivalent to the X
> Resource Database.

Does xrdb notify the clients of the changes?

There's just a few integration issues that are missing ... XFree allows
you to change resolutions without restarting, but what about color depth?
I guess it's the color thing that's most notably problematic. Aside from
that, a more efficient protocol. I just don't think that X in its current
form would be able to handle all the changes needed to bring it up to
date. Maybe X12 is "the answer", but I would argue that just as likely a
new windowing system could be built to succeed X11.

I think, for instance, that some commonality would be benficial, like
common dialog boxes. However, I would say that these need to be put in
the window manager, but you need the back-end support from the client.

I don't know ... getting too tired to think.
