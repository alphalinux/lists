For the purposes of discussion, lets call this new windowing system "Y"

Quoting Christopher Curtis (ccurtis@ee.fit.edu):
>
> > - upwards compatible with X servers/clients.
>
> Not necessarily. It will need a X server client of it's own (that is,
> it's own version of eXceed).

I'm confused. You mean a client that runs on an X server that implements a
"Y" server? Because you'd need some way to make sure the thousands of X
clients still run, and you'd also hopefully want to still run on all those X
servers out there. Do I have to mention my NCD X-terminal here?

> > - truely object oriented instead of the half-assed "objects in C" like Xt and
> > Motif currently do.
>
> Absolutely under no circumstances whatsoever. ANSI-C *only*! This is one
> of my biggest gripes with the Berlin people. The next is the registry.

Registrys are for idiots, but I disagree entirely about OO. There are a lot
of things that OO does really badly, but windowing systems is one of the
things it's good for.

Xt is a half-assed attempt to do OO in C. It's not bad, but I've seen better.

> me to go into the details here, but the biggest failing is in colormap

But I think that's an API problem, rather than a problem with the underlying
protocol.

> handling. Run X in an 8-bit display sometime for a while. Under Solaris
> anyway, colors disappear in time. Netscape (with the -install option)

I run an 8-bit display all the time (UDB with TGA, remember?)
A lot of programmers aren't very good at managing public and private
colormaps, and so they consume the whole colormap pretty fast. I have
strategies for that (mostly avoiding running Netscape until *after* running
anything color intensive).

> The next problem of X is the bandwidth. X is virtually unusable across an

Well, you've already mentioned LBX. I think it's good that X is extenisible
in order to add stuff like that, and still remain backwards compatible.

> compression. Now, I'm no X expert so I may be way off base here, but
> another thing that would be nice is an event mask. A lot of applications

There already is one, but I think it's on the client side, not the server
side.

> defaults mechanism is also needed. There should, IMHO, be three
> configuration files for each program - one in .../app-defaults, one in
> .../app-defaults/fixed, and one in the user's home directory - just like
> the setup for Pine. There should be *NO* registry.

That sounds pretty much like the standard set-up to me. Just set up the
system wide XAPPLRESDIR, and you're golden.

> To be honest, there are some things for X to learn from Windoze. The
> WinFrame stuff is pretty efficient (of course, done so at the expense of
> configurability, but nonetheless...)

The MAIN thing I hate about Windows is the lack of an equivalent to the X
Resource Database.
