> Anyway, my problem is related to the fact that when I malloc anything more
> than 128kb the allocated memory suddenly jumps to a very high address:
>
> # testalloc 131048
> allocating 131048 bytes
> address is 0x150000250
>
> # testalloc 131049
> allocating 131049 bytes
> address is 0x15555568010

I'm just curious; why is this a problem? Surely you're not assuming
that malloc'ed memory is contiguous?! Malloc can return you a pointer
anywhere it pleases 8-)

Seriously; what you see is normal malloc behavior. The smaller malloc's
are being done as an sbrk() off the end of the program's data
segment, whereas the larger ones are being done as anonymous
mmap's(). In the case of 64-bit Alpha processes, mmap() starts
searching for free memory at 0x15555560000 (TASK_SIZE / 3, rounded
up to a page boundary).

> Running the same test program on an Alpha running OSF/1 v3.2 shows no such
> behavior. Is this related to the same bugs in the linker and/or assembler
> which cause the infamous "relocation truncated to fit" errors discussed
> back in April of 1997?

No relation whatsoever, just different mmap behavior.
