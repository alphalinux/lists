On Wed, 1 Jul 1998, Greg Lindahl wrote:

>In 2.0.33, I have a program which crashes the machine.
>
>In 2.0.34, this program dies with a floating-point exception. While
>this is far superior to killing the machine ...

My simple example was acos(-2.). Yea, I had the same experience with
2.0.32 -- after an FP exception, the machine would be unstable. 2.0.34
fixed that. The reason it kills the process now is that AXP is not IEEE.
IEEE compliance slows the computation down. You can add -mieee to your
compile switches to get this: you'll get the correct answer (NaN) and a
domain error flag set (and a
kernel: moo: arithmetic trap at 00000001200007b0: 02 0000040000000000
in /var/log/messages).

Actually I prefer the program to die. Dividing by zero (for example) is
an error I want to know about rather than having it simply keep going with
infinity or NaN.
