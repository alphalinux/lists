On Mon, 22 Jun 1998, Paul Tomblin wrote:

> Ugh. Windows is horrible. The only good thing about it is also what's wrong
> with it - that it's closely integrated with the OS so all the applications
> keep a similar look and feel. But the major inconsistencies in the way
> various dialogs work, the crappy file open dialog, the registry (ack, spit).

This is something that I think can be handled by a window manager. GTK+,
for instance, has common dialogs, and whatever the GNOME window manager is
will probably use GTK+ and thus have access to these dialogs. What would
be nice at this point is a method for a client program to request the
window manager to display the dialog. This can be implemented in X as it
is now, but every window manager will need to be rewritten to handle these
exotic new clients. Whereas it's okay for the WM to ignore Motif icons,
the program will not function without a dialog it expects to be present.
I was just thinking that since there are many things like this that
*could* be done, and which probably ought to be done, so many things that
traditional X does would break these, so there's not really any need to
preserve backward compatibility. I guess this isn't really true, though.
Maybe I'm not talking about a new environment, but simply a major overhaul
of what has been the "X windowing system". I'm sure that at the protocol
level X is probably mostly stable. However, I'm also willing to bet that
there are some simple optimizations that could be performed as well.
Could they be done while preserving compatibility? I don't know.

> From a human factors point of view, Windows 3.1 is actually a more consistent
> and better interface. There is a great web page about human factors and
> interfaces I found the other day, probably off a link from slashdot, and it
> was great. I've been doing user interfaces for 10 years, and I learnt a lot
> from those pages. If I find the URL, I'll post it.

http://www.useit.com/ maybe?

On Mon, 22 Jun 1998, Bryan W. Headley wrote:

> But what's really wrong with X? There are too many window managers and
> too many widget kits. If everyone likes MFC and a Windows 95-like window
> manager, that's what has to be done. And some pre-canned apps. NeXT and
> Be went in the right direction, but went proprietary.

Maybe yes and maybe no. Right now there's actually even an additional
layer of configuration options. Has anyone used the latest E?
(www.enlightenment.org) I haven't - I don't even know if it will compile
on the Alpha, but the big introduction there is GTK+ "themes". So even
within your single window manager, you can redefine buttons and other
various widgets. I'm assuming this only works with GTK+, but that's
essentially an abstracted (or "widgetless") widget set.

In my opinion, I think that what E is doing is a better solution than what
has been done in the past, but that comes at the expense of bloat, etc,
etc. No, overall I like the concept of X, there's just some issues
regarding the internals (mostly how optimized it is) and then resource
management.

On Mon, 22 Jun 1998, Paul Tomblin wrote:

> Well, I started with C++ back in the days when cfront took your C++ code and
> produced C with a boatload of funnily named structs with function pointers in
> them.
>
> If you're going to do OO in C, you end up with something similar - Not
> horrible, but it looks a little nicer in C++.

I'll agree; C++ has some "slickness" that is good for inexperienced
programmers not used to memory management, etc. But I don't like the idea
of simply abondoning C for something that should be as integral to unix as
windows is to DOS.

In light of this, remember that my subject line is the "New" X, not the
replacement for X. :)

> I never said it was. And I think Windows is just as bad under 8 bits as X.

How could you possibly say that?? Under windows I can open Netscape and a
graphics viewer and a photoshop on an 8 bit display with virtually no
significant loss of color-isms. Under X, open netscape, xv, and the gimp.
Can you really say these are equal? I either get warnings about unable to
allocate color cells or the program simply refuses to load or the display
flashes when I move in and out of windows. Maybe you have a better X
server than I do, but this is true of XFree on Intel, Sparc, and Alpha and
on vanilla X for SunOS/Solaris on Sparc, and OpenWindows on SunOS/Sparc.

> > Does X-Free even support the printing extensions? I know they don't
> > support multiheadedness (yet?)...
>
> It's hard to do printing in an operating system independant way.

Very true - the actual printing. But look for example at NeWSPrint.
Basically if I understand it right, what you draw on the "screen" becomes
postscript, which can then be saved/printed/whatever. In opposition to
each program having to write its own PS driver.

> As for multi-headedness, I used to have a two headed VaxStation GPX running
> two heads on X10! I'd be real suprised if X11 doesn't do it.

X11 does, yes, sorry. XFree does not. I'm assuming here that everyone is
running XFree on their Alpha/Linux boxes. I'm simply using my Solaris
examples to show that it is not a XFree-only affliction.

> Programmatically, you can force a value to not be overridable, or to set
> defaults that can be overridden by the database. If you arrange the
> XAPPLRESDIR path and permissions on the directories correctly, you can have
> all sorts of control over what's overridable and what's not. For instance,

But users can modify their XAPPLRESDIR. There's simply no absolute way.

> > Does xrdb notify the clients of the changes?
>
> No, but then again you don't have to reboot the whole damn machine every time
> you make a change to the database like you do with so many windows programs.

I agree completely with you here - but simply because one method is better
than an existing method does not mean that the method is the best. If
there is room for improvement, that change should be made. Nothing is
going to stand still when it can improve over something that's "better".
NT, for example, rarely needs to be rebooted when doing "normal" things
like changing IP netmasks, etc. Yet X doesn't have the ability to change
a default background and have the applications accept the changes, whereas
Windows does. All I'm saying is that there is room for improvement. And
I don't know that the existing X mechanisms are sufficient and/or could be
upgraded without breking previous apps.

On Mon, 22 Jun 1998 cbbrowne@hex.net wrote:

> > Your X server would be obsolesed. It would still run X programs, but not
> > "Y" programs. Unless, of course, someone wrote a "Y" server for X.
>
> This being rather unfortunate. There's still a remarkable amount of useful
> old hardware out there.
>
> If you obsolete equipment, then there will be a proportion of people
> that will reject your "cool" Y server. If it at least provides a
> "translator server" that would be a "proxy" that would allow supporting
> X11R5 servers, even with somewhat degraded performance, that would be
> better than nothing.

Absolutely true. All I meant to say that was if it wasn't really possible
to preserve the backwardness of X, that effort should not be expended or
"Y" be hobbled to compensate. Every now and again there has to be a clean
break from the past into the new. Writing a translator shouldn't be that
difficult as both systems are (would be) open source, and the X shouldn't
be any worse-off than current X, just not as efficient as the newer "Y".

> The registry notion isn't *all* that bad; the fact that Microsoft has
> shown us a ludicrously poor implementation of the idea does not
> invalidate it altogether.

Maybe you are right, but I've never met a registry I liked. Personally, I
think the "benefits" are far outweighted by the risks.

> Having a DBMS for configuration information has some merit:
> - It allows one to avoid parsing configuration files

If the parsing is done by the toolkit, it doesn't matter.

> - It may allow, with suitable indexing, O(1) access to values of interest
> rather than O(size of config file)

Even in the '386 days, this was never a major obstacle. And today, on our
zippy litle UDBs, the difference is that between waiting .3 seconds versus
.02 seconds - an over tenfold increase, but still neglible.

> - It provides a common place to stick somewhat random stuff.

Not a benefit. If there are just standards for the .config files the same
could be said of that. For instance, global fixed defaults; global
defaults; and user defaults in ~/.Xdefaults/app-name, where the
application has complete control over only that single file. The only
real problem with what exists now is that there's no way to save settings.
Again, all of this should be implemented in the toolkit. And again, not
all toolkits support this, so they will have to be rewritten ...

> - With a suitable transaction processing scheme it can provide a high
> level of robustness

And this would eat away much of the performance gained by storing it in an
indexed database.

> And of course there are demerits:
> - You need some special tools to get at the data rather than grep, vi and sed

This is very bad. It's very not-unix. In fact, it's 100% not-unix.

> - If abused, it just turns into a big bundle of meaningless crap.

There are many circumstances under which it becomes meaningless crap.
Abuse is one - disk error is another. Is it a standalone database with
its own daemon? What happens when this crashes and transactions aren't
completely fully? What if you accidentally put garbage into the database?
To me, it just seems like a lot of unneeded headaches, and I think that it
what it eventually becomes. It's much easier to deal with single files
that can be removed when an application is removed, rather than having a
database with 15 years of application data still stored after the app has
been removed. And then that data adds to the overhead of reading and
writing settings whereas the file-for-app method would never even worry
about it.

> Java has the merit that there are some big people standing behind it.
> Sun is not the important "player;" *IBM* is.

Sun is an extremely important player because they control the language.
Sun can't make Java succeed, but they can surely kill it. Recent actions
by Sun reinforce this.

> A significant problem with X as it is now is that there are altogether
> too many places where settings can come from. In effect, it's too
> flexible to be usable.

To an extent, yes. I advocate only a more precisely defined mechnism,
preferrably maintained by the toolkits and with a common client API.

> But if it's not nicely backwards compatible, you'll immediately break huge
> quantities of things that function (such as they do) now.

Sure - only having an X server that runs under this incompatible
environment should not be a big deal. eXceed does it for windows and
there's no reason that two open source projects (especially if the latter
is based heavily on the former) can't interoperate with a similar
translation layer. Should it be an integral and transparent part? I
suppose it should. Does it matter in the design? Not really.

> It's not much good to have a new system that is regrettably incompatible
> with *all* of the graphical applications that have been developed for
> Linux, thus requiring that people reimplement all their graphical apps.

But the object is to have them all reimplemented eventually.

> The thought that I keep having is that the "right" way to create a new
> system would be to step back to X11R4, whose code is available, works
> (on some platforms), contains the vast majority of the functionality
> that actually gets used in applications, and happens to be a tiny
> fraction the size of X11R6, and try coding in a new direction.

But then losing out on the XRX, LBX, and print extensions.

> This would be a similar thing to the Lesstif project, with the
> difference that it wouldn't need to be coded from scratch. Huge
> quantities of code ought to be integratable from X11R6.
>
> It probably becomes more realistic to consider something like this when
> XFree86 4.0 comes out, as it appears that it will be more modular, which
> should make it a bit easier to decompose the system for further
> maintenance.

XFree could very well be the method used to bring about this change.
Maybe GNOME/GTK+ will be the method to get my desired window manager as
comptroller changes. Or maybe it'll end up being a huge complicated mess
that even fewer people understand and will act as the impetus for change.
Right now there's a huge influx of non-unix people with 28k modems who
don't know much about X. I'd like to think I know a bit about X but will
be the first to admit I know, truly, very little. They will want change
and will drive changes, but I think it's important that these be forward
changes. I suppose there will always be people with the same objectives,
but I think that time is getting short and that a lot of these little
hacks to X are going to start to really bring it down. MS is starting to
realize that, as Sun has said, the Network is the Computer, and will be
putting a lot of effort into this arena. Can they do better than X?
Maybe. Can we do better? Probably. Who will be there first, and what
will be the ramifications?
