On Tue, 23 Jun 1998, Christopher Curtis wrote:
> I stand corrected. I never had a problem with an xterm over a dialup.
> Frame is virtually unusable.

I have found *any* Motif application unusable over a dialup. I'm not
exactly sure if this is a deficiency in Motif or Xt...

> I may be completely off, I've stated that repeatedly - but I don't see
> things being adopted wihtout many old programs being rewritten, and if
> that is the case, why not look at some other stuff that could be
> rewritten, even if it means breaking compatibility?

A lot of programs probably ought to be rewritten. That doesn't mean we
necessarily have to overhaul the X protocol. Keeping backward compatiblity
is necessary to leverage the existing base of X applications and servers...

Part of my gripe with the X Consortium (and now the TOG) is the pitiful
effort they have made to modernize the "sample" code. (They call it
"sample" code, but tell me most vendors don't ship a majority of it
unmodified.) It looks like 1980's stuff nowadays. And now we have to
*pay* for it...

Fortunately I don't think backward compatibility would be too much a burden
for a "next-generation" X window system. Many of the ideas in X11 are sound.
The sample implementation sucks though.

> That is not what I proposed. I, in general, like X very much. And I have
> thought of things to improve it. The only problem is that it seems to me
> that it would come at the cost of breaking backward compatibility
> (something unix in general does quite frequently anyway), but if it's
> going to be done, I think that the immediate future is a better time than
> any, and the Windows stranglehold gets only stronger. My feeling is just
> that backwards compatibility with existing X can make the advancement of X
> where it is needed most (the UI/Toolkit layer, imho) difficult to
> impossible, and almost certainly impossible without horrible hacks. Then,
> while designing this new system, why not take the opportunity to see if
> there are other areas in which X can be "cleaned".

You've made some good suggestions. Higher-level tookits (kinda like Qt)
will go a long way towards making X more usable.

I would start off by rewriting Xlib. There's a lot of crap in there now,
and almost all X programs rely too much on it. The good news is that we
could do a replacement for Xlib and not lose any compatibility as long as
we keep the old Xlib around and don't break any of the existing X protocol.

I can see the benefits to making the X server more toolkit-aware. Not that
I would want anything like Motif code in my X server though... One of my
ideas is to implement common toolkit operations like dialogs with some sort
of platform-neutral bytecode (OK, there's a plug for java I suppose), and
have a server extension so that my application can upload the bytecode to
the server where it can execute more efficiently. That'll work nicely
over a network connection and without any precustomization of the display
server. (This should sound real familar to any NS/OS/Rhapsody users
whose DPS servers have done that for years...)
