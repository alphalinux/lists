Well, as promised, here's my bug report.

In 2.0.33, I have a program which crashes the machine. It does
floating point computations with the default fp configuration, and the
answer has no NaNs in it. This process receives many SIGIO signals. It
crashes the machine. A similar process without any fp computations
does not crash the machine.

In 2.0.34, this program dies with a floating-point exception. While
this is far superior to killing the machine, it is still
sub-optimal. We believe this is a correct program; we have many
programs that don't use fp that pass our test suite.

I don't have a simple example to illustrate the problem,
unfortunately.

Do the 2.1.X kernels have different interrupt handlers than 2.0.34?

Any other speculations?
