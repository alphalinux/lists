The registry notion isn't *all* that bad; the fact that Microsoft has shown us
a ludicrously poor implementation of the idea does not invalidate it
altogether.

Having a DBMS for configuration information has some merit:
- It allows one to avoid parsing configuration files
- It may allow, with suitable indexing, O(1) access to values of interest
rather than O(size of config file)
- It provides a common place to stick somewhat random stuff.
- With a suitable transaction processing scheme it can provide a high level of
robustness

And of course there are demerits:
- You need some special tools to get at the data rather than grep, vi and sed
- If abused, it just turns into a big bundle of meaningless crap.

Java has the merit that there are some big people standing behind it. Sun is
not the important "player;" *IBM* is.

many places where settings can come from. In effect, it's too flexible to be
usable.

But if it's not nicely backwards compatible, you'll immediately break huge
quantities of things that function (such as they do) now.

It's not much good to have a new system that is regrettably incompatible with
*all* of the graphical applications that have been developed for Linux, thus
requiring that people reimplement all their graphical apps.

> I think, for instance, that some commonality would be benficial, like
> common dialog boxes. However, I would say that these need to be put in
> the window manager, but you need the back-end support from the client.
>
> I don't know ... getting too tired to think.

The thought that I keep having is that the "right" way to create a new system
would be to step back to X11R4, whose code is available, works (on some
platforms), contains the vast majority of the functionality that actually gets
used in applications, and happens to be a tiny fraction the size of X11R6, and
try coding in a new direction.

This would be a similar thing to the Lesstif project, with the difference that
it wouldn't need to be coded from scratch. Huge quantities of code ought to
be integratable from X11R6.

It probably becomes more realistic to consider something like this when
XFree86 4.0 comes out, as it appears that it will be more modular, which
should make it a bit easier to decompose the system for further maintenance.
