Hello all!
In case you are not familiar with the internals of the AXPpci33 or kernel
programming you can safely skip this somewhat larger posting...

OK. To the facts:
I am currently trying to write a device driver for those eight diagnostic
LEDs of the AXPpci33 (aka Noname) board. I think they could be very useful
for displaying all kinds of stuff or even interfacing to other hardware like
LCD display modules and thelike. So I wanted to do it.
Referring to the axppci33_ref_memory.txt file the LEDs are connected in some
strange manner to the memory bus of the CPU and are located at address
0 3880.0000 which is far away from any real physical memory. I also learned
that the lca architecture uses memory organised in four banks; the LEDs are
therefor located in the fourth bank.

My first guess was to write a driver which just writes to this address
translated using phys_to_virt. But nothing happens! No kernel messages, no
reaction of the LEDs.

Second thought was that the strange latch circuit which is used to connect
the LEDs might need some rising and falling edges to detect the new data so
I first wrote a 0 then the desired value and a 0 again. Nothing. Vice versa
the same process (value - 0 - value) also nothing.

Then I tried to read the address range of the flash memory which is in the
same bank in order to find out if I am in the correct address space. And I
really got some data which quite looked like some piece of MILO ;)

Then I thought the firmware uses the LEDs to display it's status during
power-on. After that the LEDs keep the same pattern throughout any other
operation, i.e. starting MILO and Linux. So I thought that the firmware
might mask this memory area out using the Bank Address Mask Registers, i.e.
BMR3 for the fourth bank at address 1 2000.0038. Since this address is write
only I tried writing 0 to it to enable all adresses (0=evaluate address
1=don't evaluate). After writing 0 to it and writing to the LED's address
caused some kernel messages!
It said that an access to non-exsitent memory 0x38800000 occured which is
the LED address. And strangely I got two messages: One about an illegal read
and second about an illegal write, but I don't read the LED address!

OK I thought. Maybe the ref-text is false and the register has to be set to
1 in order to enable the decoding so I wrote 0xffffffffffffffffff (the other
bits are supposed to be ignored!). Now the machine froze completely without
any message ;)

And now here I am and am completely clueless.
Are those damn LEDs controllable after MILO and Linux have booted?
Is the BMR the cause I cannot write there?
Do I need to enable some hidden feature to gain access to the LEDs?

I already asked Jay Estabrook about that problem and he also had no idea why
this does not work but maybe some other Alpha hacker has.
If yes and even if it is just a "small" idea, please tell me! I soldered
this thing together and want to have it working now ;) And besides this I
think it could really be usable.

OK. Thank you for your attention!
Hopefully read you soon ;)
