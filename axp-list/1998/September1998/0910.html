On Mon, 28 Sep 1998, Robb Romans wrote:

: Just as a data point, 2.1.122 works fine here on my XL300. Built-in NCR
: SCSI, Quantum Fireball 1.2GB, Quantum Empire 1.0GB, both SCSI-2.
:
: CONFIG_SCSI_NCR53C7xx=y
: CONFIG_SCSI_NCR53C7xx_sync=y
: CONFIG_SCSI_NCR53C7xx_FAST=y
: CONFIG_SCSI_NCR53C7xx_DISCONNECT=y

That's the config I've got.

Version 2.5f (from the 2.1.107 kernel) worked fine for me. Switching
to 108 (which has version 3.0e) didn't work. I tried a few later
kernels, up to about 110 or so, all of which gave me problems.

On Mon, 28 Sep 1998, Michal Jaegermann wrote:

: What problems? Are you aware of any particular problems about which
: you can claim, with a high probability, that they are NCR driver fault?
: We (i.e. Hard Data) use NCR based cards quite extensively and I hope
: that I would know by now about show-stoppers.

What would happen is that the hard drive would stop spinning with a
clunk, the screen would fill up with scsi error messages, then the
system would freeze. It only happened when I switched to a kernel
higher than 2.1.107, which had major (as described in one of the
readmes) changes in the NCR scsi code. I didn't make any other
changes, and I never had problems with 107, so I assumed it was the
driver.

The machine is a bog standard XL300 with no modifications.

: In any case I was using for some time 3.0g version of this driver
: on various Alphas and it works. ftp.tux.org/pub/roudier.
: This drivers works both with 2.0 and 2.1 series of kernels.

3.0g comes with kernel 2.1.122, so I'll give that a try.

thanks for the responses,
-Wes.
