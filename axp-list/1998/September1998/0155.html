Mark Dohojda wrote:
>
> Hello everyone
>
> I have an Impassable situation here (or so I believe). I have a
> customer here who just installed Linux on alpha LX600mhz, 512mb of RAM.
> He run C program in NT and in Linux (on the same box), and NT proved to
> be 30% faster. Now I don't know much about the code, however I know
> that he used the same sources on both of the operating systems. Also
> the source (to the best of my knowledge) was in no way optimized for NT.
>
> My question is, how can NT beat Linux????? Is that possible? And what to
> do, to make sure that Linux is not going to look bad.

The simple answer is that he is probably using a better compiler on the NT
box. Since he is using RH5.1 first make sure that he is using the 'egcs'
compiler and not the gcc compiler to compile his program (just use the
command egcs instead of gcc). Secondly he should be looking at using
the flags to tell EGCS that he has the 21164A processor (its something
like -mcpu=21164a - someone correct me?).

> He has RedHat 5.1 no special editions or patches other then the man
> pages patch.

Does his program make heavy use of square roots? If so there was an
issue mentioned recently on the list about the square root implementation
in the library.


> Also one other question. Part of the problem might be his swap
> partitions. He has total of 8 swap partitions 128mb each, how can I
> make sure that Linux is using all of them? or does RedHat installation
> automatically makes sure that all the swap partitions are working
> correctly?

Its probably not a good idea to have 8 swap partitions on a single disc; that
would be slow; if they are on separate discs then RedHat should use them all
properly. (You can set priorities to indicate if some discs are preferable to
others).

But the other question is why? Linux/Alpha is quite capable of having one
big nice 1GB swap partition (I think) - its a limitation on architectures
with smaller page sizes which causes the 128MB limit.
