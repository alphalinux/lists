Roman Martonak wrote:
>
> Does anybody know how to fix the strange behaviour of command line input
> (echo lagging behind the input by few keystrokes) of rasmol which was not
> present with RHL 4.2 but appeared with 5.0 ?

Some other things are afflicted in the same way. This was posted
on this list a few months ago:

Date: Tue, 11 Aug 1998 20:45:40 GMT
From: raphaelfinkel@my-dejanews.com
Subject: Linux/AXP 2.0.30 bug: TCSETAW
Organization: Deja News - The Leader in Internet Discussion

I have had to modify ispell and metamail for Linux/AXP 2.0.30 because when
they wait for a single character to be typed, they are not awakened until 4
characters have been typed. The problem does not occur on Linux/i386. My
fixes have been in this form:

original:
static struct termio sbuf;
sbuf.c_lflag &= ~(ECHO | ECHOK | ECHONL | ICANON);
(void) ioctl (0, TCSETAW, (char *) &sbuf);

modified:
static struct termios sbuf;
(void) tcsetattr(0, TCSADRAIN, &sbuf);

both have lines like this:
sbuf.c_cc[VMIN] = 1;
sbuf.c_cc[VTIME] = 0;
sbuf.c_lflag &= ~(ECHO | ECHOK | ECHONL | ICANON);

The fact that the fix is necessary implies that the kernel is not handling
TCSETAW correctly.

I do not think that putting a blame on kernel above is correct. This
is, most likely, a library issue and likely simply means that an
application was compiled with an incorrect terminal model (termio
instead of termios). I do not investigate if an original RasMol includes
relevant configuration/compilation options but, IIRC somebody even
mentioned or posted corresponding RasMol fixes (they do not seem to
be anywhere around here but likely they are simple).

> I noticed that it disappears when
> rasmol is stopped by Ctrl-Z and restarted by fg, but then the backspace on the
> command line and also the history do not work.

They do, they do, but they have the same "delay" you noticed.

BTW - ispell called from other program works just fine as there is no
interactive terminal input involved. Similarly piping commands to RasMol
from stdout of anything (could be another shell running 'while read line ;
do echo $line ; done' :-) will make this delay to disappear.
