My take on this is: "it comes with the territory"...

Think about what "ps" (and "top" and "vmstat", among others) might be
attempting to do - access kernel data structures via /dev/kmem. On the
Alpha, we can't allow a user process to "cli()", ie disable all interrupts,
as is done on the INTEL, IIRC. This makes it possible for things in those
data structures to change unbeknownst (:-) to "ps", leaving it with stale
addresses which could certainly cause kernel Oopses if/when they are used.

I believe the same type of Oops happens frequently with "top" on Alpha SMP
systems under heavy load. It's all a bit frustrating, because "top" and
"vmstat" are such useful tools to see what's going on in one's system... :-\

Anyone have any ideas about how to work around this?

> Oct 12 22:20:08 sigi kernel: Unable to handle kernel paging request at
> virtual address 30a044e001189008
> Oct 12 22:20:09 sigi kernel: ps(32047): Oops 0
> Oct 12 22:20:12 sigi kernel: pc = [<fffffc00003a5b58>] ps = 0000
> Oct 12 22:20:12 sigi kernel: rp = [<fffffc00003a6008>] sp =
> fffffc0000655d28

The numbers here are part of the register dump; you can use GDB on a kernel
that you build to figure out where things went awry.
