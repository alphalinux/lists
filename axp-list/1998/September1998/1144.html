On Fri, 2 Oct 1998, Nikita Schmidt wrote:

> On Thursday, 1 Oct, B. James Phillippe wrote:
> >
> > I'm not sure I understand exactly what you're trying to do. You must use
> > aboot if you wish to boot with the SRM console.
>
> `Can' is the correct word. You can still use MILO if you wish.

Based on some of the things I've seen people mention lately, I've been
wondering if this was possible. But nowhere in any of the documentation
that I've seen is it mentioned that it's possible. There must be some
reason for this? Is it only as of late that it's possible, or is there
some disadvantage to using MILO with SRM? When I installed my Alpha, I was
literally informed that aboot was my only option for SRM. The SRM-HOWTO
still says this. What's the catch?

> > You must use MILO if you
> > wish to boot with the ARC console. AFAIK, MILO requires a DOS FAT
> > filesystem to exist somewhere, either on floppy or on the fixed disk.
>
> No, it's an ARC/AlphaBIOS requirement. MILO understands ext2, FAT and
> iso9660 equally well.

Yes, I should have explained that better. I meant that "MILO needed to be
installed on a DOS FAT in order for ARC to load it". The ARC console can
only load the stage-two loader from a DOS FAT. This is the result of
Micro$oft being involved in the development (to whatever extent) of ARC.

> > SRM
> > does not require any DOS filesystem, but does require the use of BSD
> > disklabels instead of PC-style partition tables, and some reserved space on
> > the beginning of the target drive.
>
> In fact, SRM itself doesn't care about partitions; I believe it doesn't
> understand any partitioning schemes. A small partition at the very
> beginning of the disk is a common place to put the bootloader to, although
> with the aid of programs like e2writeboot one can easily keep the
> bootloader on any ext2 partition.

Yes, it does not understand partitions or filesystems. It reads certain
bytes of data containing the offset and size of the stage-two loader. It
expects to find this information within the first 512 bytes of the device
(like 16 bytes starting at 480, and a flag word and checksum), which
conflicts with the existance of a PC partition table. I was under the
impression that e2writeboot was only advised for floppies because of the
fragmentation requirements (the second stage loader must occupy contiguous
blocks), and should not be used for filesystems on fixed disks?

Thanks for the clarifications, btw.
