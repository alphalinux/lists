Quoting Jeff Sturm (jsturm@gatecom.com):
> What can be done to get at the core of this syslog problem? It never
> happens to some users, apparently at least not to the developers who are
> the best bet to solve it. Anybody have any useful statistics, like what
> machines/kernels/libcs/dists/whatever causes the runaway syslog?

Dist: Redhat 5.0, all updates from the ftp site dutifully applied.
Kernel: 2.0.30, 34, and 35.
Machine: UDB/Multia, 96Mb RAM, 80Mb swap, 2Gb SCSI disk, 4Gb SCSI disk, 1x
SCSI CD-ROM, SCSI Zip Drive.
Running: mail server (sendmail 8.8.x, majordomo, procmail, newsx), news
server (innd 1.7), mail clients (mutt and elm), news client (trn), ftp server
(wu-ftpd), httpd (apache), X windows.
Networking: ppp connection to ISP, ethernet connection to local ethernet.
On: 24 hours a day, 7 days a week.
Longest Uptime: 39 days

Situation: Running syslog locally will run-away in anywhere from 6 hours to 3
days. Running syslog with logging remote to another Intel Linux box works
fine (that's how I set the personal uptime record). Doesn't seem to matter
what file-max and inode-max are set to (although the first time I set them up
higher it seemed to take longer before it ran-away). Doesn't seem to matter
if I'm running with klogd -s or not. I've never checked *during* a run-away,
but checking at random times shows that file-nr is between 400 and 445 90% of
the time.
