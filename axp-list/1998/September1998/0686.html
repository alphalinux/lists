I have just overcome exactly the same problem: it isn't related to the video
card as I had this behaviour with the adapter in the 64-bit slot. I am not
completely sure, but I think these problems arise if you don't use dd or
rawrite to make the floppies. The second time I used rawrite
(ftp://www.deskstation.com/linux/install/rawrite.exe) on Alpha/NT and milo
loaded within a number of seconds.

Here's what I found to work for the 3 floppies:
ftp://gatekeeper.dec.com/pub/DEC/Linux-Alpha/Kernels/Test-Milos-2.0.30+/ruff
ian.img

ftp://www.deskstation.com/linux/install/ruffboot.img

ftp://www.deskstation.com/linux/install/ramdisk.img

Further Problems:

I found trying to install from the ATAPI CD-ROM that came with my machine
caused reset, with 'data type 9 not recognised' as the last gasp. This may
be specific to my setup, but if you have this problem you can still install
over the net as described at http://www.deskstation.com/linux/install.html -
although I suggest a fresh supply of coffee & doughnuts if you're
contemplating this...

That page also has some useful info concerning the partitioning process.

I had a nasty problem arise at the end of installation, when the kernel
image floppy is requested: the floppy wouldn't mount, with the error 'mount
failed - invalid argument'. There is no way forwards or backwards out of
the Red Hat installer if you get this error. I found that if you use
arcinst.exe on the NT CD-ROM to make the primary FAT system partition
(instead of the RH partitioning tool), the mounting problem miraculously
disappeared. You might be able to get the same result by setting the boot
partition toggle in the Red Hat partitioning program (or somesuch), but
that's pure speculation. It doesn't make much sense, but that's what worked
for me!

Thanks go to Furio's message on this subject matter yesterday - very much
appreciated. And to whoever set out that page at DeskStation. Between
these two sources I managed to work through those problems above.
