Greg Lindahl wrote:
>
> It's fine to fix the kernel, which will paper over the other bugs, but
> it would be good to fix the other bugs too. We know that syslogd also
> has a bug because it is reading a bogus fd.

No, actually we don't (unless this is a lack of a guard against
a buffer overflow somewhere where there is no way to control input
data). If you will trash a stack than any outcome is possible
including, obviously, killing a process as reported in another posting.
Actually, one would think, that the later is more likely result.

I did not analyze the bug in depth; indeed, I was not even struck by an
instance of it. Luck of a draw, I guess, or I do not run anything that
would cause an avalanche of invocations of unimplemented system calls
(if this is indeed a reason). But it **might** be that adding this
missing terminator is a proper fix.

> As for what buffer size to insert the \n at, how about the length of
> the buffer you're reading into minus one?

What about it? If some reasonable constraints are violated then
problems may show up. This is an abstract discussion. At this stage
we need to see some code, I guess.
