How long are these "hangs"? I get about a 100ms or so delay, which is pretty
trivial (and also, the same sort of delay has always been present for audio
on ANY platform that I've ever worked with, so I think that's just load time
for the audio).

> All the sound I've been playing is through mpg123 though, which *DOESN'T*
> used mmap(). It's supposed to, but fails somehow, and I haven't figured out
> why yet.

Hmm.... my problem exists if I'm using mpg123, play [from the OSS package],
mikmod, gmod, xpe, whatever. If it uses /dev/dsp or /dev/sequencer, it croaks.

The oddest part is, it's not the app itself that falls over. Granted, it
usually does, but yesterday, I wrote a quick little shell script (at my
prompt) to play a sequence of WAV files... after the Eterm which had the
script running, a few other Eterms, my mixer, and my X server all fell over,
the audio was *still playing*, and the script didn't end until it finished
playing all of the audio. The 'play' process was running in unpriveleged user
space (yeah, I know... /dev/dsp is world writeable. I was testing. =P), so
there really shouldn't have been any way for it to cause OTHER programs to
segfault unless it's mucking with kernel memory. That's what's pointing me at
the OSS driver.

> Is the AWE64 Gold an ISA or PCI Card? I suspect some problems with the

It's ISA... the PCI/ISA initialization stuff sounds interesting... but I don't
think it's related to my problem... though it might be...

> I also get low (400k/s max) rates with my PCI 3Com ethernet card, while
> nearby Intel boxes can get 1M/s on the same network. Possibly this is
> related?

I haven't had any network problems at all... I get consistently higher transfer
rates than all of my Intel bound friends on my network segment. I'm only
running a 3c900 Boomerang in one of my 64 bit PCI slots, and it works fine,
though I haven't tried it in a 32 bit slot yet. Have you tried moving it
around? I don't think it should make a difference, but it might.
