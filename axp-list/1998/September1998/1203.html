there seems to be 3 camps for analysizing this problem

1) upgrade to 5.1, and for some ( by virtue of the silence ) the problem
has gone away ( maybe - as the deveilish peggy bundy would say ).

2) it is a syslogd-only problem, with the most recent theory that the # of
sys chans are exhausted

3) klogd is annoying the hell outta syslogd ( which seems to be prevalent
to 5.0 folks )

In ur klogd/syslogd pair, it is easier to dump the contents of the kmsg
buffer ( about 4k ) and the line buffer ( about 1k ) when the problem
occures - utherwise u have to sit there and wait for a break on the
klogd/bad channel condition.
if u wish to wait, ( I wouldn't ) then just attach klogd, put a
breakpoint on the place that gave the bad chan message, and observe what
the kernel gave back in the kmsg buffer, or see whats in the line message
buffer ( sorry about being vague about the variable names ) It then should
be evident as to who is behaving badly. with 13 systems, thats a lotta
watching.

I always like to set a trap for random problems, where u dump key pieces
of info only when the trap is set off. u dont always get this right, as
the problem can be something quite different ( ie expecting a squirrel to
show, but instead u find a gopher with the similar feeding habbits )

so what would u like to do
gat

BTW can u also rebuild the kernel? ifs its a kernel/alpha prob then it has
to be tested there.
