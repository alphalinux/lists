At the time I released those patches, I was working on interrupt-related
issues on some platforms/controllers/bridges/whatever, and needed the extra
info in case something went wrong.

> This practically
> prevents using such kernels on assorted boards. Not necessarily
> 'interrupt 0'. For example I would be flooded with such messages
> from SCSI controller on EB64+. Once commented out a kernel
> recompiled things behave fine once again.

Interrupt 0 is not good; normally, the system takes a few unexpected
interrupts as a normal course in determining exactly *which* interrupts are
hooked up to (for example) IDE and floppy controllers, and serial
ports. That's why most folk have been seein these messages for IRQs 3, 4, 6,
14, and 15, as those are the IRQs normally assigned to those devices.

However, there really is something wrong if you get a *lot* of these during
normal operation. Commenting out the printout does *NOT* get rid of the
interrupt, it just stops telling you about it.

Now, back to the original problem (on an AS200):

If you are booting from SRM console directly, you should have used the
"avanti-s.img" file for creating the kernel floppy.

If you are booting from ARC console into MILO and then the kernel, you
should have used the "avanti.img" file for creating the kernel floppy.

If you were booting a kernel built to run under the *other* console, you
might end up with these messages, as interrupt delivery is expected/handled
differently by the two kernels.
