On Sun, 17 May 1998, Uncle George wrote:

> strace may be usefull, as I tried "attaching" on an alternate console with
> gdb . the mozilla pgm is simply in its "idle thread" loop. Unlike Java where
> I can get a thread dump, I dont know how to do it in netscape.

Did you say that you could or could not log in from another machine? If
'yes' (actually, it shouldn't matter), log into X, then log in from
another machine. From the test machine, do an "xhost +localhost", and
from the remote machine, do a "export DISPLAY=localhost:0". Still on the
remote machine, "gdb netscape" and "run". Use Netscape as normal from the
test machine. When it hangs, go back to the remote machine and interrupt
it (^C a couple times). If you end up an in idle thread "info threads"
will get you a little further, but specifically where I don't know.

debugging symbols are essential, methinks.

> One other thing, the strace does not hang - so go figure ;-(

Odd. This might help - gdb doesn't (I don't think) intercept every
message. However, if strace prevents the hang, it would seem like some
sort of timing issue (there is a severe performace hit for strace'd
binaries) - something I wouldn't want to get near to save my life. ;-)

I had a lot of similar problems on an NT system because I can't get it to
lock a mutex when it's supposed to, and it often blocks on mutexes that
aren't locked (and no, it's not that I have the logic or a parameter
revesed -- trust me, I looked -- it's not consistent behavior).

Scary thing now, it works with debugging symbols, but not without.
I *hate* NT.
