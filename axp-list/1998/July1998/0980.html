> It means that some inodes which were deleted between the start of the
> backup and the end of it may not be consistent on tape: not too big a
> deal. Any data which was stable during the backup WILL be backed up,
> and any data which was created after the start of the backup will make
> the next incremental dump.

This is the conventional wisdom, but it isn't true. In my testing of
backups, stable files have sometimes been the ones lost. In addition,
the block numbers in an inode should always be in range, because
inodes are all written when the filesystem is constructed and are
never overwritten with garbage, but dump is claiming that inodes
contain block numbers which are not in range.
