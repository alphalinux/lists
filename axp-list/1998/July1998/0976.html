Hi,

On Wed, 29 Jul 1998 12:12:31 -0400 (EWT), Randy Carpenter
<rcarpen@celt.network1.net> said:

>> If this is a backup of an active live filesystem, then these kind of
>> errors are expected. dump(8) reads the low-level filesystem directly
>> from the block device, and if that is changing under your feet, then
>> it can't get a consistent image of the disk. All versions of dump do
>> this; it's not just a Linux thing.

> This is what we were thinking... we run MRTG which has alot of file
> updates... WHen I get this error, does it mean that there were files
> that were *not* backed up? The dump finishes ok, other than these errors.
> Should the backup be ok, or is it destroyed as a result of these errors?

It means that some inodes which were deleted between the start of the
backup and the end of it may not be consistent on tape: not too big a
deal. Any data which was stable during the backup WILL be backed up,
and any data which was created after the start of the backup will make
the next incremental dump.
