>>> Matti Aarnio said:
>
> I am not sure whose Alpha-SMP patches got into 2.1.92

I'm the guilty one... ;-}

> There have been two patches on linux-kernel -list, and those seemed to
> be mandatory for compilation.

Were these Alpha-related, or generic?

> However third change would be to turn
> off BWIO compiling even though I do have BWIO capable gas.
> The nasty surprise seems to be that according to /proc/cpuinfo this is
> EV5 machine, not EV56 -- and frankly I don't remember when the BWIO
> appeared into the processors.

Yes, I've heard of that; are you using the latest MILO for MIATA from
gatekeeper.dec.com:/pub/DEC/Linux-Alpha/Kernels/milo-miata-980324?
If it still happens with this, let me know, but it must be what the
hardware is (erroneously) reporting, MIATAs were never built with EV5,
only with EV56, and they all had the Byte/Word Extensions, aka BWX.
The BWIO stuff is something that PYXIS supports; use of BWX insns to
directly reference bus I/O and memory space bytes and shorts without
the need for SPARSE addresses.

> All in all, turning SMP off, and disabling BWIO instruction usage
> gives me a kernel that boos as far as to report "Starting kswapd ...",
> and right after giving me oodles of lines of:
> device_interrupt: unexpected interrupt 0

Have you installed the latest AlphaBIOS for MIATA? That *should* enable
BWIO; if it doesn't, PLEASE let me know!

But it sounds like something else besides bad BWIO, as by that point its
done a *lot* of BWIO references. If you had BWX disabled, you'd get illegal
instruction faults, and if BWIO was disabled, you'd get a bus hang, most
likely, *real* early on...

> I think letting SMP be there did do exactly the same, but compiling
> with BWIO allowed only outputs from 386 bios emulation to reach the
> serial console, and then it was time to reboot again.

Yes, please turn off SMP if you've only one CPU; it should run OK, but you
won't like the scheduling behavior, IMHO... :-\
