> >>> "Thomas M. Langley" said:

> > 'pre-2.0.31' from the Digital ftp site, installed it and the clock set this

>
> I think "culprit" is a rather harsh word... ;-}
>
> The XL-233/266 MILO available as part of RH 5.0 identifies itself
> (in /proc/cpuinfo) as exactly what it is: a machine in the XL "family".
>
> Prior MILOs for XL reported the machine as an AVANTI, which is not totally
> accurate, but was used to "bundle" the two together for correct handling by
> LIBC. When I/O space accesses (like inb/outb) are done by user apps ("clock"
> and "X" are the most frequent), they go through LIBC, which must *accurately*
> know which "family" the current machine is in for correct accesses to take
> place (I/O space accesses differ significantly between "families").
>
> Unfortunately, GLIBC-2.0.5c which ships with RH 5.0, did *not* have XL in its
> internal table of recognized Alpha "families".
>
> I've put on gatekeeper.dec.com the following files:
>
> /pub/DEC/Linux-Alpha/Kernels/glibc-205+.22jan.README
> /pub/DEC/Linux-Alpha/Kernels/glibc-205+.22jan.gz
>
> that solve the above problem, not only for XL, but SABLE and NORITAKE as
> well. Please read the README before attempting to install it.
>
> For those whose machines are still not recognized by GLIBC, continue the use
> of /etc/alpha_systype as before.
>
> BTW, the reason behind the increased accuracy of the newer MILOs is that
> we are contemplating a "generic" kernel, which would need precise information
> about the platform it was running on, in order to choose the correct hardware
> accessing techniques. BUt don't hold your breath, as multiple PCI busses and
> SMP are currently higher priority than generic kernel... ;-}
