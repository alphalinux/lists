The EB164 family has the following members:

member CPU core logic formal name
---------------------------------------
EB164 EV5 CIA
PC164 EV56 CIA-2 AlphaPC164
LX164 EV56 PYXIS AlphaPC164-LX
SX164 PCA56 PYXIS AlphaPC164-SX

Now, the PC164 *does* have an EV56, but by default (SRM and ARC consoles)
the BWX (Byte-Word Extension) is *not* enabled, I believe. So when you
attempt to run code that includes BWX instructions, you *will* get an
illegal instruction fault. Control of this option is in one of those CPU
registers that can only be accessed via PALcode, so I guess one could
modify the MILO PALcode init to enable this, if one wanted.

Reason for the default being disabled? Perhaps the DEC compilers and/or OSes
may not have been up to the task at the time of hardware release, but I'm
not sure. And even though CIA-2 purportedly allowed byte-word access to I/O
space (ie in addition to the normal SPARSE and analogous to DENSE), it never
worked. PYXIS was the first core logic to implement this and have it work.

Be aware that PYXIS-based platforms *do* have BWX enabled by default (at
least in the latest console revisions, I believe), which includes LX164,
SX164, RUFFIAN, and MIATA.
