Ok, I'll be more clear this time:
I'm using the SRM console.

i) I set up the ew0_... variables to fetch the vmlinux.bootp file created
by "make netbootp" which supposedly contains aboot and my kernel. I setup
the bootp and tftp servers as was explained in the SRM HOWTO (and that
works quite well, since MILO boots that way).
ii) I do "boot ewa0", so I have a blue screen, a little dots telling me
how far along the boot process it is then
iii)some text, last lines are
Linux alpha SRM loader 0.5
jumping to OSF/1 PAL code v1.46
iv) the screen turns brownish and the ensuing text is on a black
background. It displays all the usual boot stuff:TCP/IP, tty, memory, etc
etc
v)it detects the SCSI hard drive as usual then:

Partition check sda1 sda2 <sda5 sda6 sda7 sda8>
VFS: Mounted root (ext2 filesystem) read only

Then it crashes: it displays lots of nonsense such as
+1d0 6fb1002 66310000
+1e0 6631000 47ff0000
(...)
then the backgrounds switches back to blue, the way it was before
the kernel started outputing information.

If I ask it to "boot ewa0" again, it dumps registers and reboots the
alpha.

Some background on my config:
----------------------------
alphastation200(4)/100 with kernel 2.0.33 (and 2.0.33 patches)
all updates done. Kernel compiled with SRM-loadable option, COM1 dumping,
my SCSI driver. The kernel I use with MILO is exactly the same except for
the SRM option.
My partitionning:
/dev/sda7 62187 28308 30668 48% /
/dev/sda1 1686362 472631 1126580 30% /usr
/dev/sda6 124407 8871 109112 8% /var
yeah I know it was stupid to use a logical partition but I didn't check what
DiskDruid was doing when I was installing

I'm stumped ! Why does it crash everytime?
