And for the one person who asked me about MySQL: It *just* finished going
though configure, I can't guarantee past that, but here's what has to
happen:

First: Ignore what the MySQL documentation says. It lies. However, it
is close. DO get the package it recommends. DO NOT try to get it
directly. It has moved to: ftp://www.tcx.se/pub/mysql/Downloads/Linux.
DO NOT try to get it from the US mirror ftp.mysql.net - there are no files
there whatsoever. DO NOT download glibc - this isn't even needed. DO NOT
download linuxthreads - they won't work without glibc, and you DID NOT
get that. Untar the package you downloaded. DO NOT believe the readme.
There is libpthread.a, and that is good. There is libpthread.so, and that
is BAD. It is an Intel binary. DO this:

# cd /lib
# mv libpthread-0.5.so libpthread.-0.5.so.orig
# rm -f libpthread.so.0
# cd /usr/lib
# mv libpthread.a libpthread.a.orig
# cp <untardir>/libpthread.a .

Then run configure. This is as far as I've gotten.

Also, I have compiled and installed an EGCS snapshot:

$ which gcc
/usr/local/bin/gcc
$ gcc --version
egcs-2.91.13 980308 (gcc-2.8.0 release)

It COULD NOT compile glibc 2.06, it would always seg fault in the sunrpc
directory (actually, it was the linker that was segfaulting - I read that
I could use a different linker, I may have needed a newer binutils, but
that's nothing I wanted to go through right now). However, it seems to be
compiling everything else alright, so far. Of course, I don't use C++ so
I can't testify to that, nor have I done any performance tests, but it
seems to make larger binary files. What that tells me I have no idea, but
then again, I don't even know if that is true. Oh - there was a problem
with it - it had an undefined reference to "HZ". I #defined HZ to be 1024
(as per the standard header files) and it finished alright (once I got
past the RLIMIT_CPU... stuff).
