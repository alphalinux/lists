The numbers posted previously do appear to
be good within the significance imposed by
the roundoff problem. The slowdown I refer
to above appears to have been caused
by some unpredictable paging caused by a newer
(beta) version of the assembler dgemm.

The Fortran "time()" call is inadequate for
these tests. Matrix size must be large to
make a significantly long test, but paging
becomes a problem. I would like to continue
to use the Fortran benchmark code if possible,
since the context that most of us may be using
a fast dgemm in will probably (?) be Fortran.
Does anyone have a good idea about how
to time the dgemm call?

Here's what the original looks like:

S0 = DBLE ( TIME ( ) )
S1 = DBLE ( TIME ( ) )
CALL DGEMM( TRANSA, TRANSB, M, N, K, ALPHA, A, LDA, B, LDB,
$ BETA, C, LDC )
S2 = DBLE ( TIME ( ) )
TIMES= ( S2 - S1 ) - ( S1 - S0 )
RNOP = DBLE(2*M)*DBLE(N)*DBLE(K)
IF( TIMES.LE.0.0D0 )THEN
RESLTS( IMVAL, INVAL ) = 0.0D0
ELSE
RESLTS( IMVAL, INVAL ) = RNOP/( 1.0D6*TIMES)
etc...
