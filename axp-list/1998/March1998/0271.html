Quick comparison of three different dgemm
(double precision matrix-matrix multiply) packages
on Linux and Dec Unix. Matrices were 1012x1012.
The 625 MHz alpha is a Dec 8400 (21164A) running Dec Unix.
The 433 MHz machine is a PC164 (21164A) running Linux.

______________________________________________________________________________
MFlops -- CPU ---- OS -------- Compiler ----- BLAS

691. -- 625 MHz -- Dec Unix -- Dec ---------- Dec DXML dgemm

415. -- 625 MHz -- Dec Unix -- Dec ---------- Kazushige Goto's assembler
dgemm

415. -- 433 MHz -- Linux ----- egcs-980221 -- Kazushige Goto's assembler
dgemm

296. -- 433 MHz -- Linux ----- Dec ---------- Kazushige Goto's assembler
dgemm

27. -- 625 MHz -- Dec Unix -- Dec ---------- Public Domain dgemm
(netlib)
23. -- 433 MHz -- Dec Unix -- Dec ---------- Public Domain dgemm
(netlib)
24. -- 433 MHz -- Linux ----- egcs-980221 -- Public Domain dgemm
(netlib)
______________________________________________________________________________

I used the level 3 blas timing routine:
PROGRAM DB3TIM
* -- Written on 1-July-1988.
* Jeremy Du Croz and Mick Pont, NAG Central Office.
with N=M=K=1012. Only transa = transb = 'T' are show.
This program has some roundoff problems.
It was written in the days when a large matrix multiply
would take more than a few seconds, so timings are
in seconds. I haven't had a chance to fix that yet.
So, for example, there are two benchmarks at 415 MFlops,
due to this roundoff problem.

The assembler dgemm, written specifically for the
21164A, requires that matrix size
must be aligned (a multiple of four). It drops back
to a somewhat tuned c language routine running at
about half the speed of the assembler when the size
is not aligned. The assembler routine is specific
to the transa = transb = 'N' case. Cases requiring
a transpose use the tuned c routines.
The speed in Mflops varied significantly with the
size of the problem.

Generally, the assembler dgemm that was compiled with egcs
and run on the Linux machine appears to perform
significantly better than it does compiled
with Dec compilers and run under Linux
(compare 415 with 296). So it appears that
entirely native Linux/GNU code may be faster than
DEC's DXML, for which a direct comparison could
not be made here.
