>
> Since getting them I have so far solved some instabilities by making my
> own kernel

Can you be more precise? I compiled a number of different kernels
on a similar machine and using few different compilers ("distribution"
gcc, egcs-1.0.1, various egcs "snapshots") without any problems.
It is true that I did not try to use sources as supplied by Red Hat.

> 1) X instability. The machines have Matrox cards (spit),

I did not try Matrox with 5.0 but I worked with it on 4.2. I think
that this is an excellent graphics card.

There is a known bug in an original 5.0 release (fixed in updates)
in a definition of FD_ISSET macro (missing cast). It may hang X
(and emacs and few other things). You may need to recompile your
libraries.

> 2) g++ instability. This one is _nasty_. Could someone please get an alpha
> Linux RH5.0 machine and verify if this crashes or not:

g++ as delivered on RH5.0 is simply broken.

> compile with "g++ -o test test.cc"
>
> ==== cut, begin test.cc ==
>
> #include <stdio.h>
> int main(void) {
> FILE* pFile;
> pFile = fopen("a file", "w");
> if (pFile != NULL) {
> fclose(pFile);
> }
> }

Very good! You pinpointed a bug which costed me quite a few hours
of a debugging time as it was hidden in bowels of X. If you would
pay a bit of attention to this list you would notice that some
ten days ago, or so, I posted a warning describing the bug, symptoms
and a workaround.

A workaround is to link with gcc and not with g++ - if only possible.
You will get different, and correct, fopen and fclose.

The real solution is to get g++ from egcs (or maybe 2.8.x) and replace
g++ libraries with the current stuff.

All your problems were already discussed in the past on this list.

Michal
