>I should have once again an access to Alpha with 5.0 in a day or two.
>I am willing to bet that kermit will compile with possibly a small number

I think that new new libc has introduced a number of new dark corners.
For example compiling ckermit's ckutio.c gives about 80+ lines of errors.
One bunch come from the socket stuff - i.e.
/usr/include/linux/socket.h has
#define SOCK_STREAM 1
but /usr/include/socketbits.h has SOCK_STREAM as part of an enum
enum __socket_type
{
SOCK_STREAM = 1,
...
}

These are functionally the same but different as far as gcc is concerned.
Likewise the Address families like AF_UNIX in these files are defined just
differently enough to cause gcc to complain.

The other difference that is 'really neat' is that in
/usr/include/asm/termbits.h CBAUDEX is defined as
#define CBAUDEX 0000020
But in /usr/include/termbits.h we have
#define CBAUDEX 0000000

Kermit (at least 2 or 3 modules of it) wants to know too much about linux
internals so it ends up including some of the /usr/include/linux headers and
the mixing creates a real mess.
