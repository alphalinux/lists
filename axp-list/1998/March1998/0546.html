> The latest posting from Loic correctly idnetifies the "time drift" cause
> on a number of boxes, MIATA and SX being just two of them.
>
> It is indeed caused by the timer interrupt handler being entered two times
> for a single interrupt; seems that PYXIS propagation delays allow the
> interrupt status to not be clear by the time the interrupt handler exits.
>
> However and unfortunately, this problem was fixed for the MIATA with the
> MILOs of RH 5.0 (which introduced an unrelated bad bug), but was perhaps not
> fixed for the SX in those same MILOs (can't remember).
>
> I've put two new MILOs out on gatekeeper.dec.com:/pub/DEC/Linux-Alpha/Kernels/:
>
> milo-sx164-980324
> milo-miata-980324
>
> These should fix the clock drift problem (its a fix in the PALcode).
>
> Also included is code back-ported from 2.1.88 to calculate a (hopefully)
> reasonable value for hwrpb->cycle_freq. This should help the stock kernel
> code which attempts to detect missed clock interrupts.
>
> Whether these together will solve the 2*clock problem seen on the SX is
> anyone's guess; it's certainly worth a try. I've been running two "custom"
> kernels with the older SX MILO (from RH 5.0):
>
> 1. pristine 2.0.32 plus alpha-patches-2.0.32-0.1 built under RH 5.0
> 2. pristine 2.0.33 plus alpha-patches-2.0.32-0.1 built under RH 5.0
>
> Neither exhibit *any* problems for me on my SX; it's got only IDE HD and CD,
> Millennium, and DE500, 533MHz 21164PC rev 1, 128Mb and 1Mb cache.
>
> Note that I also tried activating the special RTC code in 2.0.33 in another
> kernel, and also encountered no problems.
>
> Let me know if these help or not; if not, a /proc/pci and the .config for
> the failing kernel may be helpful.
>
> Crossing my fingers... ;-}
