On Fri, Dec 04, 1998 at 06:29:24PM -0800, Tom Linden wrote:
> This is generally the right approach, but having done this many times
> there is a bit more to do. Once you have breakpointed, you need to
> determine which variable is offending, and this generally means examing
> the assembler statements with gdb and relating it to the source. Once
> identified you need to find the declaration of the variable whether
> static or automatic, and by looking at the declaration trying to figure
> out how it got allocated on the inappropriate boundary. If it is this
> simple then padding by dummy allocations of the appropriate size will fix
> it. If however, this is the result of aliasing, this may require more
> difficult source code changes. use of 'nm' helps a lot. good luck.

AFAIK, the primary source of these aligned accesses are pointer
tricks. Writing structures and using malloc should not cause
unaligned acceses. Instead, these tend occur when authors cast
pointers of one type to another. For example, allocating untyped
memory and coercing the pointer into pointer pointer.

  void* pv;
  pv = malloc (CB_DATA);
  char** ppch = (char**)((char*) pv + 1);
  *ppch = "Hello"; Whoops!

Please correct me if I have gone astray. I don't believe it is
possible to generate unaligned accesses without performing pointer
arithmetic with casted pointers.

Why is this interesting? It can you help find the source of problems
when you know where the culprit hides.
