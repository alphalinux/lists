Greg Lindahl wrote:
>
> > > I have the impression that might be the function routines that are not
> > > written to NOT return denormals, for example exp of a large negative
> > > number could return a denormal. This should be changed.
> >
> > Possibly.
> >
> > OTOH, I would think only if we can determine that the caller
> > is not prepared for such a value. I do not know how best to
> > do this though.
>
> If the compiler would make a symbol available based on the -mieee flag,
> i.e. #define __mieee 0 or 1, then the include files for the math
> library could pick routines which do or don't generate denorms.
>
> #if defined(__mieee)
> #define sqrt sqrt
> #else
> #define sqrt sqrt_no_denorms
> #fi

YEP, sorry I did not follow this for some time. This is about what I was
finally thinking, too. We would at least need 2 different libms. The
current lib(d)m has even more problems on it's own. Instead of returning
+/-Inf as a (hex) value, for example, it's written that way that
+/-1./0. is executed, relying on the trap handling to clean it up,
clearly not good for non -mieee programs. Hopefully later versions of
libffm can replace that.

BTW, do you know how "mother" DEC handles this ?? Are **there**
different libm's used for different compiler switches (-mieee and -fptm
n/u) ?? Or how do they resolve that ??
