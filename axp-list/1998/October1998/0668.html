furio ercolessi <furio@spin.it> writes:

> >
> > > Tonight I had a horrible machine crash during a local disk
> > > backup using rsync.
> >
> > The fact that the system crashed while rsync was running (& presumably
> > writing substantial system changes) would account for the massive
> > filesystem corruption.
> >
> > The fact that the system crashed at all is probably a kernel bug or a
> > hardware problem. There are things which can crash the system (such
> > as using up all of memory) which maybe should be considered a lack of
> > robustness and not a kernel bug. But in your case, if rsync had a bug
> > & ran amok, sucking up all of memory, then this probably would have
> > pushed the disk cache onto the disk, thus minimizing file system
> > corruption. On the other hand, if rsync was running as root (which I
> > assume you were probably doing), it could have started writing all
> > over the disk, causing the crash.
> >
> > So, I'd give you a definite maybe on the rsync bug... :).
>
> The answer from the rsync author is that it is probably a kernel
> bug. I am not sure I will pursue this, as it happens on a
> production machine and our support starts getting calls
> a few milliseconds after any stop.
> Now I will probably go back to good old rdist for a while
> and this should rule out h/w problems (which I do not think
> to have).
>
> http://samba.anu.edu.au/cgi-bin/rsync/incoming?id=1388;user=guest
> contains the whole exchange with the rsync people.
>
> Yes, rsync was running as root, updating 62000+ files in 1.6GB.
> System is LX533, 256MB RAM, 512MB swap, lightly loaded.

You might want to ask them for load estimates (memory + cpu) expected
for such usage. I think rsync can be a memory hog in certain
situations (such as syncing a large number of files), which could
cause alpha problems. I've also seen alpha problems when memory usage
is heavy - programs running regularly that suck up most of memory, or
programs that write tons & tons really fast, causing the write cache
to fill up. My guess is that this causes some sort of memory
fragmentation from which the system sometimes doesn't recover.
