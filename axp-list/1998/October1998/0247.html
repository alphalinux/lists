Greg Lindahl wrote:
>
> > That statement in itself
> > indicates there must be FP calculations in the library,
> > which in turn means -mieee is not "free".
>
> Sorry, I thought you were reading the thread. The original bug report
> included a gdb session which makes it clear that the library was doing
> an fp operation (d < 0.0). I then speculated on the relative number of fp
> operations in the library, and predicted that no one would lose if we
> made -mieee the default for the library.
>

If you have time, try determine roughly how much
FP work is done in the libraries. Based on that
estimate, it may make sense to make -mieee a default
but since it's not that way now, it always helps
to quantify as much as possible, what the change would
do, particularly if it can potentially impact performance.
g77 gets lots of heat due to performance issues so it
seems prudent to not make changes that make performance
worse without a strong argument. The question is
"how much worse?".

> > Be that as it may, the solution for DgncppB
>
> I was posting about a different issue: all codes that use -mieee need
> a library which can print NaN's. If you want to talk about debugging
> that one code, that's great -- what we need for that is an exact
> prescription of how to get precise exceptions and core dumps in the
> face of NaN and Infinity generation. But be aware that there is more
> than one issue being talked about.
>

Well, I was basing the discussion on the subject line.
NaN handling is a reasonable goal. It just goes
against high performance on Alphas, and, as such, should
not be in the main path of the code unless one explicitly
selects that option, for a particular application.
