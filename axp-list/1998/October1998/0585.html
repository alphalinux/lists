Let's see.. I've got three alphas (one multia) netbooting here.. What
I did was set the SRM variables to appropriate values... (using 'set
variable value'). For me that means:

bootdef_dev ewa0
boot_osflags root=/dev/sda4
ewa0_protocols BOOTP

Modify the boot_osflags to contain the appropriate device for your
machine (in Linux device nameing semantics). Once those are set, type
'init' to reinitialize the console. When it comes back.. re-check the
variables ('show boot*'), you should see that boot_dev has been
automagically set. At this point you should be able to just type
'boot' and have the machine spring to life. Once you have it reliably
booting in this manner, you can even set auto_action to boot and have
it auto-boot when powered on.

Another caveat which I'm not sure you've compensated for is that
although the SRM console allows you to use a serial terminal to
operate it, the Linux kernel won't without special configuration. The
2.0.* series didn't include options for this AFAIK. There's rumored to
be a patch hanging around that will allow you to do it, though. The
2.1.* kernels do have support for a serial console, but it needs to be
configured. On 2.1.129 the options you're looking for are:

Virtual terminal (CONFIG_VT) [N/y/?]
Standard/generic (dumb) serial support (CONFIG_SERIAL) [Y/n/?]
   Support for console on serial port (CONFIG_SERIAL_CONSOLE) [Y/n/?]

And don't forget to not enable the TGA stuff later on.

Hope this helps! If you have any other questions.. let me know.
