Bill Broadley wrote:
>
> ----- Forwarded message from Kevin.Brown@asu.edu -----
>
> >PIIs have 512K L2 cache and 32K L1 cache so they will perform even better than
> >an overclocked celeron.
>
> Ummm depends on what your doing.
>
> For jobs that mostly access L1 (32k) they will be identical (per Mhz).
> For bigger jobs that mostly fit in 128k the celeron will be faster (per Mhz).
> For bigger jobs that mostly fit in 512k the PII will be faster (per Mhz).
> For bigger mostly memory bound jobs the celeron will be faster (since cache
> misses are faster, and latency to main memory is lower).
>
> It's very hard to know where a particular application will fall without
> actually benchmarking it. Benchmarking YOUR application is always highly
> apprehended.

The above analysis is ok for a uni-processor system but
if you are concidering a multi-CPU system it should be because
you have many processes that you want to run all at the same
time. Having even 16 CPUs will not make any one process go
faster. It only allows you to run more processes without taking
a big preformence hit. So, to modify the above statement you need
to "benchmark YOUR application mix". Cache use is more complex
when we are talking about running 4 copies of gcc at the same time
as a DBMS and web server all while supporting a few telnet sessions.

It is clear the cache will get thrashed but it gets worse: You
need the keep the caches in sync. This is a non-triveal problem
for designers of multi-CPU systems and a big system bottleneck once
you get more than two CPUs. The problem is a write to memory by
one CPU may invalidate the content of the cache of every other CPU.
DMA controllers can do writes too.
