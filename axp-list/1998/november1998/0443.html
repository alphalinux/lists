> Disaster. I just started a sendmail 8.9.1 in a production
> environment, and several connections initiated from another
> server on the same LAN remain in a state where they remain there
> forever eating CPU time.
>
> This seems to happen after the mail has been delivered to the
> local user, and netstat shows a CLOSE_WAIT state for these connections.
> On the other server that initiated the connection, the connection
> remains in FIN_WAIT_2.

Found a workaround. This is a bug which has been observed also
on other platforms (Sun ,HP), involving sendmail 8.9.1 when used
together with Berkeley db. Discussed yesterday in
        http://x4.dejanews.com/getdoc.xp?AN=413789609
        http://x4.dejanews.com/getdoc.xp?AN=413875184
I can now attest that this problem also occurs in AlphaLinux.

The workaround consists of configuring db with --enable-diagnostic
turned on. Enabling diagnostics suppresses the problem (!)
Apparently sendmail 8.9.2 will fix the bug.
