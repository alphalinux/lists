On Thu, Nov 19, 1998 at 01:20:48PM -0700, Maurice Hilarius wrote:
> Yes, but partitions dedicated for swap usually run faster. With an extended
> partition and many logicals you pay no real penalty for this, other than a
> rather large and confusing fdisk table..

I figure that paging is darn slow and even _if_ paging to partitions is a
little faster than to swap files, it's still just the difference between
really really slow and really really really slow. :-)

Anyway, how would one go about measuring the relative performance of swap
files and partitions? In the absence of hard data, perhaps we might be
able to start a war instead? The partitionists versus the swapfileans? ;-)

> Besides just how much swap can you really use?? If you are using a MB of
> swap, something is "fishy"..

We run some fortran 77 codes (with static memory allocation). They tend to
(pre)allocate lots (TM) of memory and while they may never use it all, Linux
won't let them run unless it has at least as much virtual memory as the
process might ever need.

If I watch some of these programs with `top', they might never be seen
to use more than 25% of the memory they asked for at any one time. For
them to take full advantage of (say) a 0.5GB machine they will need
there to be 2GB of swap, at least!

The joy of fortran 77, eh?
