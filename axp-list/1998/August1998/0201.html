Hi Christopher !

> > which card, 0,2,0. Since you can query the bus to
> > find out what each device is, I see no problem with
> > the way cdrecord under Linux numbers things.
>
> Neither do I. Having significant Solaris experience, that it how I would
> expect it. The problem is that *Linux* does idiot SCSi remapping. For
> example, right now I have three SCSI devices. A hard drive, tape backup,
> and CD burner. The hard drive came with SCSI ID 6. Linux calls this
> /dev/sda. When I put the CD-ROM (ID 4) into the chain, the hard drive now
> becomes /dev/sdb. Duh? Solaris does it right. Linux does not.

Adding a CD-ROM should not affect disk ordering at all !
I hope that you meant /dev/sga, /sgb.
There is no right or wrong. Linux supports only 16 SCSI-HD's because
we have only 256/256 major minor numbers. This is really bad (and stupid),
but not wrong.
cdrecord uses /dev/sg + an ioctl to get the hda:id:lun. So
scanning /dev/sg* can be used to map relative (/dev/sg*) to
absolute (hda:id:lun). So the minor-# may change but there
is no problem to find absolute devices there at all.

I've ported cdrecord to DU (CAM) last March and there is only a
single device-node (/dev/cam) for all SCSI devices. This requires to build
pseudo fd's for cdrecord (shifting bus/tgt/lun) and demultiplexing
it in scsi_send.
ANSI-CAM would be a nice interface to have in Linux not only because
it has only a single entry. But it's impossible to implement it in a nice
way.

I know that Joerg Schilling likes Solaris very much. But Solaris
SCSI-layer is also proprietary. The only standard out there
is ANSI-CAM (as supported (but a bit broken) under DU).
