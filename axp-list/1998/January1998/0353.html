> dg@cogency.co.uk (David Gilbert) wrote:
> > So what happens without -mieee? He is seeing traps without -mieee.
> If you do not compile with -mieee, application will receive a SIGFPE
> exception, and crash if the signal is not handled, in the case of NaNs,
> Infinities, and denormals.

oups, hmm, ok, you agree with me that the following is heavy outdated ?

--------------
This is Edition 0.06, last updated 23 December 1994, of `The GNU C
Library Reference Manual', for Version 1.09 Beta of the GNU C Library.
--------------

--------------
None of the mathematical functions ever generates signals as a
result of domain or range errors. In particular, this means that you
won't see `SIGFPE' signals generated within these functions. (*Note
Signal Handling::, for more information about signals.)
--------------

What means this good old text: None of the ANS-C fp-funcs (in libc) use
if(NAN != NAN) raise(SIGFPE); internally,
but the kernel may raise real fpu-exception by sending SIGFPE ??

I interpret this as: If you use glibc ANS-C fp-funcs only (not asm-inline),
installing an SIGFPE handler is a paranoid action.

I think this info-file is only outdated and obsolete.
I don't have an AXP nor I wrote fp-code over the last three years...

gcc handles NAN well under i387+ because FPU-exceptions on an intel
can be masked. I don't know it, but your posting makes me think
that masking is not possible on AXP, so this info-file is simply
outdated because it depends on FPUs where exceptions may be masked.
Is this assumption correct ?
If it's correct than "-mieee" means libm installs a SIGFPE handler itself and
fakes errno, but without "-mieee", SIGFPE is passed to the app ?
I'm in panic for now. If SIGFPE cannot be avoided at all
(ok, it can, SIG_IGN, -mieee), fp-pthreads programming must be near to hell.)
At moment, I know nothing but I'm heavy confused. :-(
In my "dream word" SIGFPE doesn't exists.
Give me some light, please.
