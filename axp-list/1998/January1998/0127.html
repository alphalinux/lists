Mike Learn wrote:
> Has anyone experienced this problem? Could it be about the
> monitor/keyboard missing? (I could steal both from one of my other
> machines in order to just get it loaded, if required.)

Yes, that's exactly your problem. The reason for the inconsistency is
that
each element behaves slightly differently in the absence of a
keyboard/monitor:

Console firmware (both ARC and SRM) detect the lack
of keyboard and switch to using the serial port.

MILO is dumb; if it's configured with serial echo enabled
then it uses both the serial port *and* kbd/monitor at the
same time (i.e. you can type at both the keyboard *and* the
serial line, and everything is echoed to both video and serial)

The Linux kernel, by default, is even dumber; it uses ONLY
the keyboard/monitor as the console device. There's a
patch set for 2.0 floating around somewhere that allows the
use of serial console, and this functionality is built in to
2.1.

Unless you cobble up a kernel with serial console support, you will not
be able to do your Red Hat install from a serial line. Even then, I
don't
believe there's any way to tell the Red Hat install procedure what
*kind*
of terminal to talk to... I'm pretty sure that "Linux console" is
effectively
hard-coded. Which means that unless your terminal emulator can do a
faithful
emulation of the Linux console, you may end up seeing only garbage.

Your best bet is probably to snarf a keyboard and monitor to do the
install...

> A couple other minor questions..
> Is a multia always equal to a "UDB"?

Essentially. To be more precise: the hardware is exactly the same. The
name "Multia" refers to the bundle that includes the hardware plus
Windows NT.
That, in fact, is why we had to come up with the name "UDB". DEC had an
agreement with Microsoft that every "Multia" system would be shipped
with
Windows NT installed. If we wanted to ship such a system "bare", we had
to
give it a different name.

> I would welcome any personal experience regarding the performance of
> the 266mhz UDB versus some of the other platforms running Linux.

To be honest, the UDB (and the NoName) are the slowest Alpha boxes there
are. They are based on the 21066 (LCA) chip, which is a 21064 (EV4)
with
a PCI bus interface wrapped around it. The PCI interface really bogs
down
the chip performance. A 166-MHz UDB performs about like a 90 or 100MHz
Pentium. The 233-MHz UDB performs like a 120 or 133MHz Pentium. I'm
not
aware of DEC making any 266-MHz LCA's or UDB's... either you've got your
numbers wrong or you've got an overclocked system. This will probably
work,
but DEC doesn't support it.
