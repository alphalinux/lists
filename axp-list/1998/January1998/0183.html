Oachkatzlschwoaf,

> ioctl(4, CDROMPLAYTRKIND, 0x1201033cc) = -1 EIO (Input/output error)

> My audio system seems to be working well, as I can 'cat foo.au >
> /dev/audio' with no problems.

That's for sure not the problem.
If and only if it's not a Kernel bug (AXP related) there are IMHO only some
possibilities:

1) There's no audio track on your CD-ROM media. :-)
(Try another one)
2) This CDROM can't be read by your drive.
3) Your CDROM doesn't support audio at all. (SCSI-standard: Audio is
optional, and if audio is supported, _ALL_ audio related commands must
be supported. (grabbing is not part of the standard, but the IOCTL
CDROMPLAYTRKIND doesn't grab)
It performs a "PLAY AUDIO TRACK INDEX" (opcode 0x48) command.
I really don't think so. I know only one NEC drive without
SCSI audio commands. It's an old single-speed diskman clone with an
optional early SCSI-II draft compatible interface.
4) Your CDROM has a broken audio support. Try a player with
generic SCSI-support (/dev/sg?) supporting "PLAY AUDIO MSF"
for playing.
5) The program has a bug and requests a TRACK/INDEX combination
not in range.
6) The CDROM media has a bad TOC. (see 5, but it's a hardware bug :-)
7) It's however luck if CD playing even works with CDROM-ioctls.
Playing over track boundaries depends on SOTC-bit (audio mode page)

> Before I start diving in to this ioctl stuff trying to figure out
> what's up, I thought that I might ask here first: Does anyone know of
> program that works?

It doesn't look like a program problem, more like a CDROM-{media,drive}
problem.

> NEC 8x SCSI CDROM (not sure of model #)

What's about "cat /proc/scsi/scsi" ?

If you really want to know what's happen, recompile your kernel
with "Verbose SCSI error reporting" enabled and check syslog after playing
audio. (Mail me sense key/code infos if you have problems interpreting it)

I highly recommend using Michael Brecker's "Don't try this at home" album
for testing :-))

Good luck.
