Well, the saga continues.

It seems to me that PYXIS driver is indeed unrealiable
to do PCI configuration space register lookups.

After failed boots with 2.1.79 yesterday afternoon, I began
again today morning and by disabling SCSI device BIOS ROM
disable routine from arch/alpha/kernel/bios32.c, I perhaps
increased system propability to boot with sensible PCI probes
in place.

Question is, why these two devices very often yield invalid
base memory addresses ? Manifesting as if they want more
than 9 * 16MB memory for local access -- and because they
are referred to via sparse memory routines on DEC PWS433a,
the system gets machine checks, and its reboot time...

Bus 1, device 10, function 0:
Ethernet controller: DEC DC21140 (rev 34).
Medium devsel. Fast back-to-back capable. IRQ 44. Master Capable. Latency=32. Min Gnt=20.Max Lat=40.
I/O at 0x9000 [0x9001].
Non-prefetchable 32 bit memory at 0xffffff80 [0xb100000].
Bus 1, device 9, function 0:
SCSI bus controller: Adaptec AIC-7881U (rev 0).
Medium devsel. Fast back-to-back capable. IRQ 40. Master Capable. Latency=32. Min Gnt=8.Max Lat=8.
I/O at 0x9800 [0x9801].
Non-prefetchable 32 bit memory at 0xfffff000 [0xb101000].

These cards are: SMC made 10/100 bps enet, and Adaptec AHA 2940UW

To review the PCI programming details et.al. of pyxis.c file,
I tried to figure out which chipset the machine has and if I can
find documents for it, but I could not figure out which Digital
Semiconductor's chip id refers to PYXIS.
Opening the box does not seem to help either, the only chip I can't
figure out is weird IBM OMPAC box with lots and lots of connectors,
but it doesn't have DEC 21XXXX series ids at all..

There are troubles with code-name usage... Sigh. No easy mapping
from code-names to chip designators :-(
