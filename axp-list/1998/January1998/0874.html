Jay.Estabrook@digital.com writes:
>
> >>> Johan Almqvist said:
> >
> > Applying the patch can be somewhat tricky, though: I need to have dirs
> > (symlinks) called /usr/src/new/ and /usr/src/old/ and /usr/src/linux/
> > all pointing at /usr/src/linux-2.0.33/ for it to apply correctly. This
> > seems weird, but the kernel's a rock. Even my tulip worked with the stock
> > driver.
>
> As has been mentioned before on axp-list (do I hear an echo? :-), it is NOT
> recommended that building of a custom kernel based on alpha-patches take
> place in /usr/src/linux, as the patched headers will replace the "tuned" Red
> Hat headers for application builds, and may cause headaches and shortness of
> breath... ;-}

So, if when compiling the kernel we get the -D__KERNEL__ flag why can't
the headers be set up to have one set of values for kernel compiles
and another set for regular compiles like so:

example------------------------------------------

/* blah.h
* blah blah, blah blah blah blah
* blah blort blat blah
*/

#ifdef __KERNEL__
/* place kernel modified header stuff here */
#else
/* place stuff for normal compiles here */
#endif

end-----------------------------------------------

or will this break. I am guessing (and most likley making a fool of
myself) that this will work and if it does it would mean that you
could build the kernel and other apps with the same include files and
not have to worry if one set or the other is out of date or not.
