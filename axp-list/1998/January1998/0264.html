> Egcsg77 looks to me very strict:
>I had already compiled these routines with gnu g77 and
>a fortran77 on a solaris, and did not get any compilation error.
> Now one of these routines uses a work space
>that is an array of float and pass it to another routines that needs
>a work space of integers.
> Egcs detects that the two types are differents and gives error.
>Of course that may save one from bugs now and then , but in this
>practical case that`s just some memory space, so the type
>does not matter.
> May be I am missing some feature, I don`t know
>if typecasting exists in fortran, or may be I could
>change the behaviour with some switch.

The reason egcs appears stricter to you is *probably* because
it is based on g77 0.5.21, and you're not used to using that
particular version of g77. (Most of the global, or filewide,
analysis facility was added in 0.5.21 -- including the kind
of bad-code detection you're talking about.)

Could you report what terms you looked for in the index of the
g77 documentation, so we can add more entries to help people
find out about options like -fno-globals?

Also, could you show the diagnostic(s) you got, so we can
tell which ones need pointers to the relevant portions of the
document added? (E.g. newer g77 diagnostics contain text
such as "[info -f g77 M GLOBALS]", which shows the exact command
to type to find more info on the diagnostic.)
