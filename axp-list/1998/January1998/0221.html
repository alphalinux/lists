> tom@kednos.com said:
> > A cross-compiler is just a program that reads an input file and
> > produces an output file. The fact that the output file is object
> > code and maybe for a 64bit architecture is irrelevant.
>
> Yes, one would think that. In practice, though, there are a lot of cases
> where expressions need to be evaluated from the source being compiled,

Hey think !
His definition is far from reality. :-)

I read it four times and now I finally accept that "cp" is a
cross-compiler under Tom's definition. (only when used with two args.)

His definition depends on exact one input file.
It's impossible to cross-compile C-code that uses "#include"-statements.
(but this is a special case, Tom talks about abstract cross-compilers)

His definition may be valid for some "filter" programs, but
not for something known as cross-compilers.

In fact, it's no problem to cross-compile from 16-Bit with 128-Bit
target (maybe the cross-compiler should use a mp-library) but
gcc is something special. It's not a simple stupid
abstract cross-compiler it's a real-world solution.

For a definition write to Richard Stallman <rms@gnu.org> with
body "I'm using Linux. What is a cross-compiler ?"

I bet (2 US$) that you get two replies.

One contains "That is unfair. Your are using GNU-Linux. That is a modified
variant of the GNU operating system...."

... and the second includes a useful exact definition.

It's really important that you are careful when you write and speak.
[And it's even harder if English is a foreign language for you.]
