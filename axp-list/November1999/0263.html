On Fri, 5 Nov 1999, Jeff Sturm wrote:

> > Possibly, but I still find it unlikely that the Athlon will be faster than
> > a '164 clock for clock, let alone a '264 based on my experience. I simply
> > don't believe benchmarks, the best benchmark is your own code.
>
> Very true. The scary part of benchmarks like SPECint95 is that CPU
> manufacturers may target benchmark performance specifically to get an edge
> over the competition, which does not necessarily benefit real-world code
> to the same degree.

Yup. As I've said, I have yet to see the x86 processors perform anything
like as well as Alpha for my work regardless of what optimisations I might
put into the compiler. But, I use very little floating point.

> Anyway, doesn't the Alpha have real advantages over x86 (besides clock
> speed, in which they are now neck-to-neck)? My asm is a little rusty, but
> I believe the Alpha has far more GP registers, and can use registers for

Sure does, if you really want some fun you can get into the asm (it's so
nice on an Alpha, its like an high level language) and tune the living
daylights out of the program core.

> parameter passing... does that mean it relies less on slow main memory or
> cache? Also, the Alpha traps unaligned memory accesses, and the x86
> ignores them by default... that means they are far more likely to be fixed
> on Alpha than on x86, even though they still carry a penalty. I'm sure
> there are other differences that I can't think of right now.

Indeed. Other points to remember when dealing with any processor is
maximizing your cache usage, try and keep your memory accesses nice and
linear so that the cache won't be being filled with lots of data you
won't be using (O'Reilly has a nice book on high performance computing
which goes into this in detail). x86 has such a feeble amount of cache
that for anything that is seriously compute intensive and that uses the
cache well on an Alpha you are really going to see some rewards.

> I write business software. I don't need fp performance. Typical business
> software really isn't heavy integer processing, as some others have
> claimed. Rather it is very heavy on data processing, i.e. memory and
> disk-intensive. Large amounts of RAM are common to exploit disk caching
> to the fullest. Large processor cache helps too. Any tricks to avoid
> memory fetches and processor stalls are beneficial. The importantance
> of clock speed diminishes as much processor time is spent waiting for
> memory or bus.

Yep, actually sounds pretty similar to our code although we actually use
very little physical RAM (1MB) and the algorithm is slow enough that even
an IDE disc can keep up. More than anything else I think this is why our
code behaves so well on the Alpha. Plus we use every register we can get
our hands on.

> The Alpha seems to cope well. Most of our software I've tested on Alpha
> performs as well as or better than the x86 version (both compiled with
> gcc). I realize I don't have the latest and fastest PIII, but then I
> don't have a 264 either... I guess I'm working with the previous
> generation of both architectures.

Ditto here although we are negotiating the price on a UP2000 with single
processor at the moment to add to our cluster of '164s.

Basically though, if your code is plain 32 bit and compact so it uses the
cache well the '164 Alpha is able to outperform clock for clock its x86
rivals. Also, if you can get the dual instruction pipeline running
nicely on the Alpha (add a few no-ops here and there to stop it stalling)
you can bump the speed up again. Anything you need to do in 64 bits is
going to be much faster and if you can do SIMD in the 64 bit word as we do
then you can slaughter x86. This completely ignores FP as I don't use it
by and large.

Want a comparison? If I run our Smith/Waterman algorithm on an Intel PII
400Mhz box against a database it will take around 15 mins to complete a
database search. The same algorithm (not the same program since it won't
run in less than 64 bits) will complete the task on a 533 '164 in less
than 48 seconds. Hopefully if we get a UP2000 box we can do it even
quicker.
