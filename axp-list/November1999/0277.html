To everyone who asked, I have a PC164 running 6.0 with kernel 2.2.9, which
seems the most stable out of all the 2.2 and 2.3 versions I've built.
I've tried gcc 2.91, 2.95.1 and even 2.95.2. I settled on gcc-2.7.2.3 and
kernel 2.2.9 after lots of freezes and grief. This combo gives me the most
uptime, but I still have freezes. I used to have freezes that were network
related, but after juggling different ethernet cards and different slots,
I settled on a NetGear with the Tulip driver as a module. No more network
freezes.
My sound drivers are built as modules. For variety, I tried building the
"OSS modules" option statically so I could use the "Persistent DMA buffers"
option, but the actual sound drivers are modules. I don't have a desktop
Intel PC, so I can't set the sound card's i/o, irq, and dma to fixed values.
I'm stuck using the PnP and sound drivers as modules.
My sound card is an original SB16 ISA PnP card. I've built my kernels as
generic and also as PC164 specific, both with and without leaving the SRM
PCI setup intact. No difference. I still get freezes most notably when playing
mp3's (within seconds) and when playing xmame (at random).
A friend of mine with a PC164 and soundblaster has the same problem I do, and
I know other people on this list have voiced their similar problems.
Would it make that much of a difference if I was able to statically link in
the sound drivers to the kernel? Again, I'd have to get to a PC in order to
run Creative's config program. My conf.modules file is as follows:

alias parport_lowlevel parport_pc
alias eth0 tulip
alias sound sb
pre-install sound insmod sound
alias midi opl3
options opl3 io=0x388
options sb io=0x220 irq=5 dma=1 mpu_io=0x330

The "pre-install sound" line used to have "dmabuf=1" at the end, but I tried
removing it when I tried out the "Presistent DMA buffers" kernel option.
No difference. Is there some peculiar problem with the Saturn I/O PCI-ISA
bridge driver on the PC164? Could it be an IRQ problem? I noticed someone
else is using their soundcard on IRQ 10 instead of 5.
All suggestions welcome. I hope I can get RedHat's attention, as I've filed
a bug report with them.

-Mark
