Huh? So SPECcpu2000 is going to fix this SPECcpu95 'problem' and provide a
single number which is a good relative measure of computer system
performance which is relevant for all applications? I would have thought
that was only possible if all applications made the same demands on a
computer system.

As unpalatable is it is to everyone, there's never going to be a single
number which provides an accurate measure of computer system performance for
all applications.

As Greg mentioned, SPECint and SPECfp (both 95 and 2000 ) are made up of
real world application codes. If the committee chose your application for
inclusion, you can just look at that component score to see how a computer
system will perform
for you. If not, the best benchmark is still to run your own application.

Here's a real world example of the range of performance possible. SPECcpu
scores are calculated as a ratio of the tested machines performance to the
performance of some lowly SPARC system. A 667 MHz UP2000 ranges from 19 to
134 times faster than this reference system. That's more than a 7 times
difference in performance. Even comparing the 2nd highest to the 2nd lowest
score gives a differential of more than 4x. That's still a pretty big
difference when Intel and AMD are beating each other silly over a few
percentage points difference in performance.

The bottom line is that SPECcpu numbers are merely meant to be general
indicators of performance. As benchmarks go, SPECcpu95 is OK, although a
little out of date. SPECcpu2000 will be a little bit better and a little
bit more current, but it's not going to be any magic bullet that provides
the one true number to measure systems by.
