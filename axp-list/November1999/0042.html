> Well, 2.3.23 pre patch 1 is running on my system without any problems
> nor any warnings, using bttv, and frame buffer...
While this is true, it should not be used to extrapolate a general trend
for all 2.3.x kernels. They are, as was mentioned by someone else,
experimental kernels. A few months ago I coded an axp asm implementation
of mmap to fix a problem in a dev kernel. Something like that
is "new" and has the potential to break things - other kernel coders
take the same attitude, and you end up with a lot of volatile code
in the dev kernels. In other words, expect things to break between
minor revisions - it is expected.
 
> It seems that some intel modification impacting general code were made
> without reporting the change on the alpha.
I don't know how others feel about this, but that turns my knobs
the wrong way. This is a recurring problem which I've brought up in
the past, without changes. Portions of code in mm/, include/linux/
and kernel/ have needed "sanitization" to clean them of x86 opcodes
in recent releases.

What drives people to slap [insert favorite arch] opcodes into
architecture-neutral areas of the kernel is beyond me. It's not
just poor coding practice, it's very rude to someone using any of
the other supported architectures.
