On Fri, 5 Nov 1999, Shane Sturrock wrote:
> > 3 address, and 3 FP. The Specint95 indicates they did something
> > right.
>
> Possibly, but I still find it unlikely that the Athlon will be faster than
> a '164 clock for clock, let alone a '264 based on my experience. I simply
> don't believe benchmarks, the best benchmark is your own code.

Very true. The scary part of benchmarks like SPECint95 is that CPU
manufacturers may target benchmark performance specifically to get an edge
over the competition, which does not necessarily benefit real-world code
to the same degree.

Anyway, doesn't the Alpha have real advantages over x86 (besides clock
speed, in which they are now neck-to-neck)? My asm is a little rusty, but
I believe the Alpha has far more GP registers, and can use registers for
parameter passing... does that mean it relies less on slow main memory or
cache? Also, the Alpha traps unaligned memory accesses, and the x86
ignores them by default... that means they are far more likely to be fixed
on Alpha than on x86, even though they still carry a penalty. I'm sure
there are other differences that I can't think of right now.

I write business software. I don't need fp performance. Typical business
software really isn't heavy integer processing, as some others have
claimed. Rather it is very heavy on data processing, i.e. memory and
disk-intensive. Large amounts of RAM are common to exploit disk caching
to the fullest. Large processor cache helps too. Any tricks to avoid
memory fetches and processor stalls are beneficial. The importantance
of clock speed diminishes as much processor time is spent waiting for
memory or bus.

The Alpha seems to cope well. Most of our software I've tested on Alpha
performs as well as or better than the x86 version (both compiled with
gcc). I realize I don't have the latest and fastest PIII, but then I
don't have a 264 either... I guess I'm working with the previous
generation of both architectures.
