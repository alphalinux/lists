Stig Telfer wrote:
> I was writing a memory fill routine. When implemented in the style
> you have below the WH64 didn't have any effect. Then I changed the
> loops so that there was a WH64 loop over a consecutive 32K region of
> memory then a fill loop over the same region. This got me 77% of
> peak memory bandwidth sustained over hundreds of megabytes.

Thanks. That's just what I was looking for. It seems to me that a 16K region
aligned on a page is the largest guaranteed to be resident at once in 2-way
associative L1 cache, assuming a random virtual-to-physical mapping. Is there
any good reason you chose 32K?

> Also it
> was using kernel-mode (kseg) addresses which bypass page faults. So
> you might not be quite so fortunate.

Good point. The type of problem I am considering is e.g. a copying GC. I tend
to favor non-moving collectors over copying/compacting due to better cache
interaction... however special cache instructions might improve the situation,
this is what I am attempting to research.

After the first collection or so, few page faults should occur in normal use.
TLB misses are another matter. I looks like the 21264 is capable of clustering
pages in the TLB... don't know if Linux supports it, but that's a whole 'nother
topic.
