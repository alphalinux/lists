On Wed, 18 Apr 2001, Uncle George wrote:
> It is never perfectly OK to produce bad code. AND i dont care what
> standard one uses to justify this 'OK'ness.

So what do you want to hear? That the behavior of your code is defined
according to the standard? I don't have the relevant text handy but I'm
pretty sure it isn't. If you want to know how to make your code work
as before, Brad already said -fno-strict-aliasing gives you the old
behavior (along with lesser optimization). FWIW I didn't realize
-fstrict-aliasing was the default on Red Hat's compiler. With 2.95 you
still had to specify -fstrict-alising if you wanted it. (Anybody
try Uncle George's example on ccc, just for kicks?)

Anyhow... all standards and correctness issues aside, one problem with
that sort of pointer manipulation is that the compiler cannot generate
optimal code (i.e. eliminate evil loads/stores). Your double argument
starts in a register, then must be written to a temporary location from
which the integers are read. That's a RAW conflict, and likely causes a
replay trap.

I don't know if EV6 has a better way, but you aren't going to see one as
long as you are confusing the compiler by casting pointers. A union is
the correct way to handle it. In principle, the compiler can optimize
accesses to a union type by generating register->register ops when
appropriate. (I proved this with an experiment on x86, since my alpha is
1000's of miles away and crippled by a power storm... <sigh>.)
