On Mon, Apr 30, 2001 at 11:23:50PM -0400, T. Daniel Crawford wrote:

> The problem I'm running into is when denorms are generated within optimized
> libraries like CXML (specifically the BLAS routines).

One problem with the compaq libraries is that they all apparently
generate denorms, which is a pain when you call exp() and want zero
and get back a denorm.

I was under the impression that Goto's routines should return zero
instead of denorms. If not, you can ask him about it.

> I've tried to
> compile Kazushige Goto's BLAS codes with -fpe3 to try to eliminate these
> FPE problems, but to no avail.

Because they're assembler, they won't respect that flag. At best
you'll get undefined behavior.

> In addition, what I still don't understand is the difference in FPE handling
> between Tru64 and Linux. My denorm problems are not occuring under Tru64,

If you can generate an example, and prove that the library is the
guilty party, Compaq would probably see it as a bug. However such bugs
have sometimes turned out to be kernel bugs in the past, so both
avenues have to be explored...
