> > May be, but on my DECpc 21064 based system it is really does. it is

This appears to be some other program than previously referenced.
Sounds like it isn't functional code though which makes it's not
very interesting. It is preferable to use real codes. Also it
is not the same Alpha as the previous post, 21164sx I think.

> As it is well known for a very long time your "simple" program has not
> very much to do with a compiler but with libraries. If you want to
> compare a code generated by compilers then use the same libraries,
> especially for elementary functions, linked with your binary or you are
> talking apples and oranges.
>
> Even if you would like to refrain from using libcpml for some reasons
> then alternative, much faster, implementations of elementary functions
> for Linux/Alpha are available for many years.
>
> >From a small fragment somebody quoted the code under discussion has
> nothing do to with things like sin() and cos().
>

The code fragment looked reasonable. I looked at the package and it is
almost, if not all, integer based with much casting between 8/16/32 bit
ints.

There are many types of code where x86 can beat Alphas. For all but
a few codes, modern x86 have no problems outperforming a 21064/21164,
and even a 21264 in many instances.

My current tests with P4 indicate it is equivalent to at least a
600Mhz 21264 for my codes, with some SSE/SSE2 in use for critical
loops.
