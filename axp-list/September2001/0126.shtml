On Tue, Oct 16, 2001 at 01:33:24PM +0400, Ivan Kokshaysky wrote:
>
> Isn't there some environment variable (boot_reset = ON ?) that causes
> SRM to re-initialize the buses on halt/reboot?

I think it only does it when you start to boot, rather than when
you re-enter SRM from the Linux kernel. Unfortunately, the problems
show up upon the re-entry, rather than at the boot... :-\

> Resurrecting 2.2 code wouldn't be too hard, and I thought about that
> a while back, but it looked to me just as a waste of kernel memory then...

Yes, it shouldn't be too hard to port forward. I believe the reason it
wasn't done as part of the 2.4 development was that a *lot* of changes
were taking place *generically* in that area (PCI bus probing), and
rather than confuse the issue on Alpha, the save-restore code was left
out.

> Another possible reason of those machine checks: we change PCI windows
> mapping, and save/restore it for tsunami machines, but not for older
> systems. This might confuse SRM as well, isn't it?

Yes, the window settings could certainly be a factor, if not the main
problem...

BTW, we (COMPAQ) have done some testing on the latest boxes (ES45 and
beyond) with NOT changing the SRM settings. It seems to work quite
well, though the reasons may be more related to the fact that these
later boxes have more "standards-conformant" hardware/console than
the earlier ones, making it less difficult to translate the settings
into something that Alpha Linux can use. The issue here is mainly in
the IRQ settings and interrupt delivery rather than in DMA windows or
resource addresses.
