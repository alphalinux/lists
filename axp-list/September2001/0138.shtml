On Mon, Oct 15, 2001 at 02:05:23PM -0400, Jay Estabrook wrote:
> Well, SRM doesn't save it's settings, or get them from NVRAM; ie it's
> not like ISA/EISA configuration info. It rebuilds PCI settings every
> time it powers on. So "pulling the plug" won't confuse it.
>
> However, once it *is* running, it doesn't expect the settings to
> change out from underneath it (T64 and VMS don't do it :-). Linux
> does, and when exiting from kernel to SRM during shutdown, then it
> really *does* get confused. I believe the mchecks or PCI errors and
> such are caused when SRM tries to use either its own registers or its
> data from memory that now no longer accurately reflect the PCI
> configuration.

Isn't there some environment variable (boot_reset = ON ?) that causes
SRM to re-initialize the buses on halt/reboot?
Resurrecting 2.2 code wouldn't be too hard, and I thought about that
a while back, but it looked to me just as a waste of kernel memory then...

Another possible reason of those machine checks: we change PCI windows
mapping, and save/restore it for tsunami machines, but not for older
systems. This might confuse SRM as well, isn't it?

Ivan.
