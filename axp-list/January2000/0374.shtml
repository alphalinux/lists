"Jakma, Paul" wrote:
>
> > that seems to spend a lot of time context
> > switching between
> > the processors (judging from xosview).
>
> or maybe a case of the observed system being influenced by the measuring
> tool?
>
> ie xosview needs cpu time aswell. (and quite a lot of it).
>
> consider the unobserved system, where your app and the X server are the main
> loads. The linux scheduler will correctly try to keep X on one cpu, and your
> app on the other.
>
> Now introduce xosview. You have 3 processes fighting for cpu_time, and only
> 2 cpu's. The linux scheduler will still try maintain cpu affinity, but if a
> cpu becomes free, there's no point in wasting that slot.
>
> what you are seeing is correct scheduler behaviour.
>

Using affinity though, if the scheduler weights things reasonably,
would put the cpu constrained task on one cpu and leave it there
while putting things like X/xosview/your favorite window manager/the
clock/xterms/etc. on the other. If you want to look at a minimally
cluttered environment, run things from a text screen, not X. If your
program needs X, maybe you should separate the compute part from
the display part. (Works for me... YMMV)
