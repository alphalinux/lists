Christoph Dworzak wrote:
>
> Now the problem was, that MILO and Linux don't
> agree on the scan direction, MILO puts the 3 HD-Chain
> first and the 4 HDs behind it. Linux does it the
> other way round (where Linux=Redhat6.1 with
> Kernel 2.2.13).

You are actually seeing a difference between PCI scan order
in 2.0 and 2.2 Linux kernels. Most milo executables around
are based on the former. This scan order was changed to be
the same as the one used by BIOS; at least in most cases.

> So is there an option or so to reverse on of the
> two?

Surprisingly enough it will be 'pci=reverse'; or find milo
recompiled with 2.2 code.

> I tried two Adaptec 2940AU first and those
> were detected in the same order by milo and
> the kernel, but they wouldn't work reliably.

If that is the case, and this was the same milo/kernel pair,
then something had to be quite strange with a PCI scan in that
situation.

>
> Another problem were the new ext2-options. After
> scratching my head and searching the mailinglist
> I finally found that creating the root-fs with
> "mke2fs -b 1024 -O none /dev/sda1" makes it
> readable by milo.

'sparse_superblock' can be modified with a help of tune2fs and fsck;
block size is fixed in a time of a creation of a file system. Once
again - you are seeing a difference between 2.0 and 2.2 kernels
as new file system attributes are not supported by older kernels.
Booting from a small partition, which will be created with a block
size 1024 by default in any case, is a good idea regardless. Milo
will get an indigestion if your kernel to load happens to be
positioned further than 2 GB offset from a partition start.
