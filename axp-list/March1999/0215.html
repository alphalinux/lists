On Don, 04 Mär 1999 Helge Kreutzmann wrote:

>On Thu, 4 Mar 1999, Jochen Kuepper wrote:

>> But in the end, I would say, speed mostly depends on good algorithms and
>> efficient implemetations, not on the programming language.
>
>There is an interesting comparison done by the University of Cambridge
>regarding the possibilities to optimes fortran vs. C code. You can read it
>under
>
>http://www.hpcf.cam.ac.uk/C_rant.html
>
>It suggests that FORTRAN code can be optimized more easily thus (at least
>in theory) providing faster code.

Yeah, short but nice page. I would still kep my claim, though, based on my
(special ?) personal experiences.
Looking at the programs around me, there usually really is not much of a
difference between programs that are plain FORTRAN or completly C, but the
amount of thought/work people put into their code counts much more.

Btw: My main programming project is a mixed FORTRAN / C++ project, and thats
the way I would suggest for anybody seriously beeing concerned about speed.
Then there remains the problem of which parts to write using this or that
language. You also can use all these nice FORTRAN libs from within C++ classes,
so you can manipulate your class easily from C++, but it does FORTTRAN in
reality :-)
Here this seems to save development efforts by writing ugly FORTRAN and saves
runtime by using nice programs.
But then an good assembler BLAS is even better :-)
