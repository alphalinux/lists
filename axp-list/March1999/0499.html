"Dr . David Alan Gilbert" <dg@px.uk.com> replied to me:
> > These issues have been mentioned before (like up to 2 years ago),
> > but as far as I understand, nobody has made necessary tweaks to
> > PCI bus device drivers to do sort of 'bounce buffers' scheme we
> > already use for the ISA-bus devices. (GFP_DMA, and the new
> > GFP_DMA_PCI32 ?)
>
> What happens on 64 bit PCI slots? Would it be possible to build a system
> with a LOT of RAM which only used 64 bit PCI devices (I think
> you can get SCSI, I think network as well ???)

   The Alpha systems I have seen do have 32-bit address busses in
their IO busses, however they often have mapping window machinery
to be able to map a few (4, 16 or perhaps even 32) address windows
to the global main memory bus, and the device drivers would need
to update those mapping windows every so often... In low-endish
systems the CPU/MEMORY/PCI bridge chipset (the 21*7* chips) does
DMA window mappings in between the PCI bus and the main memory,
and due to the way how those chipsets work, we can go only up to
1 or 2 GB (depending the chipset) of memory which is directly
reachable from the PCI bus.

So far I have not seen reports that PCI64 devices would use 64-bit
ADDRESSes. Apparently it is possible, but we don't have any drivers
using them yet.

On the other hand, we already have drivers which operate DMA in
ISA-bus constrained 16 MB space, modifying PCI drivers to cope
with 1-2 GB DMA address space limit in similar manner isn't
impossible, and in fact the ">4G" RAM folks for intel platforms
will need them too.

In your 2.2.* source tree, execute following to see which size is
supported (per default) by which core-logic:
   grep _DMA_WIN_SIZE_DEFAULT include/asm-*/core*.h

Right now I don't have time to tinker with this issue, but I can
hopefully review the work somebody else does, and direct people
to feasible directions in case they err to suboptimal path.

Any volunteers ?
