>>> Greg Lindahl said:
>
> OK, so that needs to get fixed on a driver-by-driver basis.
> Fortunately that's pretty easy although tedious: look for kmalloc
> calls.

That'll catch the obvious ones, anyway... ;-}

And once you *do* find one, what'll you do with it? Simply add GFP_DMA to
the flags? At the moment, that'd only work for XL and RUFFIAN, as all the
other Alphas don't limit ISA DMA addresses to anything less than 32 bits,
and that's what that flag is used for (the bounce buffer technique uses
an explicitly-tested-for ISA_DMA_THRESHHOLD value, to activate it).

It'd be much better, as Matti suggested, to introduce a new GFP_DMA32 flag
for this purpose. It would be set for all pages that could be accessed via
the direct-mapped PCI DMA window(s), on the Alphas.

Unfortunately, it still doesn't handle the case of a request for DMA to a
user buffer, when that buffer is not in the reachable range, as Steve has
pointed out. Perhaps the bounce buffer technique in the SCSI subsystem could
be modified to catch this, but then, that'd only handle the SCSI case...
