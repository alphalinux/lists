On Sat, Mar 13, 1999 at 03:57:48PM -0800, Stephen Williams wrote:
> > With 32-bit PCI devices, it is difficult to get past 2GB RAM. More
> > than that requires either bounce buffers or that PCI windows be
> > adjusted at runtime to cope with the various outstanding DMA requests.
> >
>
> The problem is buffer allocation.

Read "bounce buffers".

We do already prefer `dma-able' pages for dma tasks. There is a kmalloc
flag just for that. No matter what you do though, at some point it will
so happen that the data is required at some high physical address.

There are two ways to get the data there -- either through an intermediate
buffer in a page frame the device can actually see, as you note NT does.

Alternately, we can adjust the DMA window so that physical addresses
0x100000000 through 0x13FFFFFFF (4GB - 5GB) are visible at PCI addresses
0xC0000000 through 0xFFFFFFFF (3GB - 4GB). At which point we do not have
to copy data around in kernel space, which is a real win.

The trick of course being that there are only N dma windows, and they
must service all outstanding dma requests simultaneously, and otherwise
the request must queue or fall back to a bounce buffer.

> The interface is really simple and should be able to handle the problem
> quite easily.

But is not necessarily the best way.
