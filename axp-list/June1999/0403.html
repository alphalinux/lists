And few days ago Richard Henderson posted an explanation on this list.
Is anybody reading that? There are still unclear problems with vfork
(and apparently its cooperation with address translation tables on a
chip). For a time beeing, until somebody will solve that riddle, turn
vfork off.

Here is the patch as posted by Richard:

 Date: Wed, 16 Jun 1999 23:00:39 -0700
 From: Richard Henderson <rth@twiddle.net>
 To: "Daniel J. Frasnelli" <dfrasnel@doxx.net>
 Cc: axp-list@redhat.com
 Subject: [hack] rh6 instability

 On Wed, Jun 16, 1999 at 01:13:12PM -0400, Daniel J. Frasnelli wrote:
> By the way, several members of the Alpha Linux Org. including
> myself have noted that there seem to be major stability problems/strange
> errors under RH 6.0 with tasks that previously worked fine.

 I know. I've seen it myself. DaveM gave me the vital clue
 this evening over sushi -- vfork. The following hack disables
 vfork while I figure out what the real problem is.

 r~

--- 2.2.10-dist/arch/alpha/kernel/process.c Wed Jun 16 22:44:07 1999
+++ 2.2.10/arch/alpha/kernel/process.c Wed Jun 16 22:26:09 1999
@@ -282,8 +285,12 @@ int alpha_clone(unsigned long clone_flag
 
 int alpha_vfork(struct switch_stack * swstack)
 {
+#if 0
         return do_fork(CLONE_VFORK | CLONE_VM | SIGCHLD, rdusp(),
                         (struct pt_regs *) (swstack+1));
+#else
+ return do_fork(SIGCHLD, rdusp(), (struct pt_regs *) (swstack+1));
+#endif
 }
