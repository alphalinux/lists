Uncle George wrote:
>
> The ratio of overhead ( crc, leadin/training bytes, and sector/track numbering
> ) / block would be different. Less I/O initiations ( although now if u want to
> change a single byte u's have to read in all 1k ).
>

The newer drives use something called "ID-less sectors" so I
don't think the overhead per block is as high now. I did look
at the unformatted capacity of the Seagate 50GB drive. It's
around 60GB. That would be for control overhead and bad block
relocation I suppose.

> so u think that whatever really happens inside a disk drive now-a-days does
> not need to match what a person thinks should be happening inside. ?
> gat
>

Yes. Whole track I/O has been around in mainframes for at
least 10 years so it's just trickling down from the big
iron. As long as it creates the same end result, the
application shouldn't see any difference. It allows the
drive to combine writes (assuming you turn on write buffering)
and pre-read data so in general it helps performance.
