> >> Another problem is reading core dumps generated with 2.2.x kernels.
> >
> > I do not recall reasonable cores ever (beyond an information what
> > generated it). For this a workaround is to run a program from gdb
> > and provoke a crash. gdb will catch exception and one can usually
> > get more information than from a static core. Quite often it is
> > possible to attach gdb to a running process somewhat closer to a point
> > in which is crashing so an overhead is smaller. This is particularly
> > true with programs with GUI or some other user interaction.
>
> IMHO analysing core dumps is on of the favour "features" of unixes.
> e.g. when a customer of us has a crash he just sends the core dump. This
> can give you an huge amount of information what leads to the crash.
>

I agree, core's are useful. In fact, a while back someone
posted a fix to allow reading core files on Alphas to this
list. If I remember correctly, there is a problem in the
definitions of registers or some such that causes the core
to be unreadable. If the archives go back far enough, you
should be able to find it. Also, I think it was related to
having to be able to sort out the difference between two
formats and gdb just didn't know how to do it. Anyone ever
try the patch? I get along doing as Michal suggests.
