On Tue, Aug 24, 1999 at 10:25:39PM +0400, Ivan Kokshaysky wrote:
> Interesting. Recent 2.3.x kernels somehow break settings of the Cypress chip,
> so IDE addresses are bogus (something like 0xfff0). As a temporary workaround
> try this hack (tested with 2.3.15pre2), it set IDE i/o back to 0x1f0/0x170.

That's odd. The code that I changed was intended to leave i/o resources
0 and 1 alone (the 0x1f0/170 bits), but to change the memory region. This
because SRM was putting the memory region out at 0x10100 on my ds20, and
the recent resource rewrite uses a 64k i/o space.

I thought about just widening the i/o space, since technically Alpha can
speak to a 32-bit i/o space. But it seems quite likely that there's lots
of broken hardware out there that only wires up the low 16 bits, and so
as a rule I don't want to be allocating i/o regions outside 64k.

The correct place to poke things is in arch/alpha/kernel/bios32.c,
in function layout_device.
