Max Weninger wrote:
>
> The problems are still there. With a stock gdb 4.17 and egcs you will
> get "strange" results using "next" and "step" commands. Strange means
> that e.g. next runs till the end.
>
> The problem is that egcs generates incorrect information about function
> prologues. This problem has been dicussed on the gdb mailing list.

Somehow this does not mesh up with my experience; i.e. prologues info
is undoubtely incorrect. Still my "private" workaround is after
breaking in a function to step through one or two executable statements.
Unless I compiled with -fomit-frame-pointer gdb as a rule is "catching
up" and a backtrace is looking sane. Is that different for others?

> Another problem is reading core dumps generated with 2.2.x kernels.

I do not recall reasonable cores ever (beyond an information what
generated it). For this a workaround is to run a program from gdb
and provoke a crash. gdb will catch exception and one can usually
get more information than from a static core. Quite often it is
possible to attach gdb to a running process somewhat closer to a point
in which is crashing so an overhead is smaller. This is particularly
true with programs with GUI or some other user interaction.
