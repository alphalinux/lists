Uncle George wrote:
>
> I suspect that the dynamics are all the same cyl/track/sector. It just get
> complicated to have relocated (bad) sectors, and keeping up with 'track format/layout'

I'd think there are available spare blocks on every track
so if one did full track read/write, it's just a different
position in the disk's cache for that track which would be
minimal overhead.

> presuming that it gets published. It wouldn't surprize me that some disks do
> compression to get extra speed/storage.
> gat
>

That may work for general business data but the stuff I
do doesn't compress. Learned that long ago on tape drives
that have compression. In fact, trying to compress my type
of data usually takes about 5-10% MORE space than if you don't.
>From the compression algorithm's view, the data looks random.

Now, there are lossy, custom, compression algorithms (wavelet)
that do work but it's lossy so no one would use it without
indicating it was used or risk being sued for damaging someone's
data. That's one of the things on my list of to-do's to get
working eventually.
