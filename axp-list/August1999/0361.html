Wes Bauske wrote:
>
> Multiply ms after it's calculated by 2. For some
> reason, the newer ncr53c8xx.c is computing the
> wrong MHZ for my scsi cards.

I have seen something, so far only on dp264, which **may** be related to
that. On various occasions with **the same** version of a driver, on
the same machine and without touching anything in a controller itself I
see reports "NCR clock is xKHz" (this is from ncr_getclock() in
sym53c8xx.c) where x is either 21626, or 40037 or 80074. The third case
is followed by "PCI clock seems too high" and no disks. It does not
appear to matter which of the other two values was chosen - things
are ok in both cases.

> This has happened on
> 875/875J/895 so it isn't isolated to a single type
> of card.

What I see seems to be related how a kernel was built (configuration and
maybe a compiler itself) but I still cannot point to a culprit with any
conviction. I start to suspect some timing issues in reading an
information from a hardware/firmware. Maybe some delays are not not
long enough for ev6? Messing with a disk controller without knowing
what I am really doing does not strike me as a very exciting option;
especially when I cannot predict which clock value will show up.

> /* wsb */
> ms=ms*2;

You know that this is safe or it just happens to work?
