The kernel doesn't read the correct cycle frequency upon startup and
instead trys to "estimate" it, where it does a bad job. It calculates
arround 100MHz instead of the correct 300MHz.

As a result, the clock and any related timing processes run about 3
times to fast. Hard coding the estimated cycle frequenzy to
300,000,000 fixes this for me.

I first experienced this with a 2.1.131 kernel (which happend to be
the latest available version to me when I tried video4linux), but
since the pre-2.2.0 kernels are out, I decided to test 2.2.0pre6 and
the but is still present. 2.1.110 worked fine, so somewhere inbetween
this got messed up.

Sorry, I don't have the time to look further into this myself right
now, I hope somebody recalls what has changed and might cause this
trouble.
