Munagala V. S. Ramanath wrote:
>
> > This is somewhat what I do though in my own code. I implement
> > a sqrt() as a table lookup plus twiddle so I don't have to
> > rely on anyone's sqrt() function call. What I do is not good
> > to the ULP (unit of least precision, ie., last bit),
> > but good enough for my algorithm. It is also faster than any
> > silicon implementation I know of. Doing that sort of thing
> > requires a good understanding of how your algorithm works.
> > Anyway, my code is about twice as fast on a 533LX vs a PII400.
>
> That's great and exactly the kind of experience I was hoping for.
> How much faster on the 533LX is your code compared to the native
> silicon sqrt() on the PII400 ?
>

Never ran just that code.

> Could you post the code or is it proprietary ?
>
> (Knuth also has a table lookup based implementation for a variety of
> the transcendental functions in Metafont and may be a useful
> starting point for people trying this approach.)
>

It's embbeded in the application so it would be
difficult to just show that. The basic idea is
that if you know the range of sqrt() input values
your code will generate, build a table of values
sampled in that space. Then all you need is some
sort of twiddle. Those depend on how good you need
the answer to be. One simple form is an interpolation
between values. You could look at the math libs to
see how they hone in on a value also. If Knuth has
it in Metafont, it should be reasonable. He probably
did it for the same reason I did.

Be careful with recursive/iterative codes because they
can be sensitive to errors.
