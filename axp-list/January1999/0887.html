Pretty close, but... ;-}

1. I'm fairly certain that ldmilo was not written by Deskstation until much
   after the board was designed (when having a LINUX loader was deemed to be
   "advantageous" :-).

2. the reason that the linload sources were not available at the time that
   Deskstation went to write a LINUX loader, was that the sources used by DEC
   were encumbered; they utilized parts of the API of the NT SDK, which were
   controlled by MS, and *not* freely available/distributable.

But the primary reason that (the original) linload won't work on a UX is
staring you in the face; does ARCSBIOS really resemble AlphaBIOS so much
that you assume that the underlying support for "console applications" would
be identical? Deskstation rolled their own, including BIOS emulation code.
Good for them (they control it all), but bad for compatibility.

The latest ldmilo was built (by DEC) with sources/development kit made
available by Deskstation (thanks!); it has close to the same functionality
as linload, with, AFAIK, the only constraint being that the MILO file must
always be named "milo". Have you tried loading the Debug Monitor using this
latest ldmilo found on gatekeeper.dec.com:/pub/DEC/Linux-Alpha/Kernels/?

In addition, rumors have surfaced recently of a port of linload to ARCSBIOS;
perhaps when this becomes available it will prove to be able to run the
Debug Monitor.
