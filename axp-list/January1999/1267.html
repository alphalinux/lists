Munagala V. S. Ramanath wrote:
>
> > Michal Jaegermann wrote:
> > >
> > > Munagala V. S. Ramanath wrote:
> > > >
> > > > There was one guy who posted here a few weeks ago about running
> > > > a loop which simply computed sqrt(). The 450MHz Pentium II
> > > > outperformed the 533MHz Alpha.
> > >
> > > Big deal! Pentium has sqrt in silicon while Alpha, as RISC, in an
> > > external library.
> >
> > Multiple RISC implementations have sqrt() in silicon.
> > It can improve performance significantly just by not
> > having the function call overhead.
>
> I'm just curious about areas where the PII outperforms the higher
> clocked Alphas and what the reasons might be. Is it possible that
> a tightly hand-coded inlined sqrt() on the Alpha could beat the
> PII ? After all, the higher clock speed should mean something ...
>

Not many people have that little piece of code you describe.
This is somewhat what I do though in my own code. I implement
a sqrt() as a table lookup plus twiddle so I don't have to
rely on anyone's sqrt() function call. What I do is not good
to the ULP (unit of least precision, ie., last bit),
but good enough for my algorithm. It is also faster than any
silicon implementation I know of. Doing that sort of thing
requires a good understanding of how your algorithm works.
Anyway, my code is about twice as fast on a 533LX vs a PII400.

I think PII can outperform Alpha on transcendentals and divide.
Watch how you use those and you'll have fast code in general,
not just Alphas. Note I use mainly FP calculations, very little
is integer in the kernels I worry about performance on.
