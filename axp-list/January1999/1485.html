We have experienced the following vexing problem and wonder if anyone else
has seen something like it:

We are running a few large Fortran codes which are compiled on a DEC Alpha
workstation running Digital UNIX V4.0B with Digital Fortran for Digital
UNIX Alpha Systems V4.1. The executables are copied to several Linux
systems, each DEC Alphas running Red Hat Linux 4.1, 4.2 or 5.1. When the
programs are executed, we sporadically get an unexpected termination with
the following type of message:

forrtl: severe (29): file not found, unit 8, file getwd: can't lstat
.././LOOKOBS

where LOOKOBS is the name of a file which the Fortran program is trying to
open. The open statement which preceded the error is something like this:

      OPEN (LUNLOOK,FILE=OUTPATH(1:IOUT)//'LOOKOBS',STATUS='UNKNOWN')

where OUTPATH(1:IOUT)='./'

The error happens with different files but always while opening an output
file. The same programs happily open and write to these same files
successfully MANY times on previous program runs before generating this
error. The relative filename given in these error messages (.././LOOKOBS)
actually appears to be wrong, one directory above where the Fortran code
says it should be. But the program routinely opens the file in the right
place, so we think the relative pathname is being mangled by the system
somehow.

When the error occurs, it can usually be replicated a few times, but
generally the error will go away if any files are removed or altered in the
directory where the program is running, even if the removed or altered
files are unrelated to the one which was not successfully opened.

It doesn't really seem to us like a Fortran or data problem, but rather
just flaky behavior somehow related to the fact that we compile with
Digital Fortran on a DEC Alpha, then run the executables on a DEC Alpha
running Linux. When we run the same programs on Alphas running Digital
Unix, the problem never occurs. Does anyone have any insight into what the
"getwd: can't lstat" message might be indicating?
