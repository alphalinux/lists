On Wed, Jan 13, 1999 at 11:58:56PM +0100, Joachim Wesner wrote:
> On Wed, 13 Jan 1999, Ross A. Knepper wrote:
>
> >however, the processor still has to ensure that the execution behaves
> >exactly as if the instructions were executed in their specified order.
> >imagine trying to keep track of many in-flight instructions and once and
> >complete them all in-order by hand ... it is not a simple task! (i've
> >done it) so, while the 21264 is much more flexible in execution than the
> >21164, it still has very little knowledge to work with. by contrast, the
> >compiler knows much more about what instructions are interchangeable, etc.
> >the compiler also doesn't have buffer limitations, so it can look at much
> >larger sections of code in order to re-order instructions. that is why
> >(particularly on the alpha), it will always help to have a better
> >compiler.

That's not allways true, since at runtime the processor might know
what a conditional branch will do, while at compile time you might
have no idea.
Also the time that is needed for some instructions is not known at compile
time. So at run time the processor could execute instruction while waiting
for a load or fpdiv. (Given of course there are no dependencys and there
are execution units left.)

> For example, does anybody know how gcc for Ultrasparc (latest version, can't
> tell the exact code in the moment) compared to the original manufacturers
> compiler rates ?? This is interesting, because only recently the 2nd real
> 64-bit Linux was starting to materialize, Ultralinux 64 for Ultrasparc. From
> what I read, the Ultrasparc CPU, highest version, is a heavily out-of-order and
> register-renaming and ... CPU. Anybody got some details on benchmarks for
> both compilers ??
>
I have made some test with egcs and ultrasparc-II and alpha EV56 cpus.
I think that they have nearly the same problems. I tested two very little
pices of code and while on both cpus the compiler edmitted the same
instructions, the scheduling was different given a 100 % time
difference.

On this test egcs was giving the fastest code, followed by the Sun Pro C
compiler and gcc was slowest.
