David Monro wrote:
>
> get too much contention on the bus or something). I've no idea what you would
> do with that rate of data; even an Alpha is going to have a hard time doing
> anything sensible with it!)

Have you ever run a database server? We were I/O bound even with a fiber
channel external RAID box. The box had six independent SCSI controllers,
it's own CPU and a few megs cache onboard. It connected back to the computer
with two fiber channels at 100MB/sec each. Still, we waited on the disks. We
could manage only about 1000 transactions per minute. Story #2: On a Sun Ultra
SPARC with two CPUs, 512MB RAM and UWSCSI disks. I had to rebuild the index
files on a 10,000,000 row database. It took about nine hours of disk thrashing.
Same job on a Pentium with IDE drive took over two days.

So to answer the question. Anyone running a database server can use all the
IO bandwidth they can get, and then some.

You may notice that some of the bigger servers now have multiple PCI busses.
The idea is you put only one UWSCSI controller on each PCI buss so as not
to overload it. So there IS a demand for high IO bandwidth but I don't see
the need for it on most desktop workstations.
