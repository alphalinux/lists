Hello Linuxers,

I've seen a weird problem with Redhat 5.2 (Apollo):

My Hardware:
Miata (Digital Personal WorkStation 433au)

1. Boot Digital Unix 4.0c
2. The system date (software and hardware clock) will be set by ntpdate
3. Boot Redhat Linux 5.2 (with original Kernel or 2.2.0-final)
4. Have a look to the system date: Same month, day but for 1999 the year is set
to 1910 (software-clock, displayed with date) or 2047 (hardware-clock,
displayed with /sbin/clock).

After some minutes the system will become difficulties in using rpc-services,
ypcat gets timeouts and mounts don't work either. If I set the software-clock
to the correct year the system will recover and act as wanted.
The output from the two clocks were:

[[root@dahlie dat]# date
Tue Jan 26 13:07:48 MET 1999
[[root@dahlie dat]# /sbin/clock
Sat Jan 26 12:07:06 2047

(The hardware clock should run with GMT). The Difference between 1910 and 2047
might be a result of the limited bit size of the internal Unix-time. On our
other Linux-Alpha (a Sable) I haven't seen this problem. The greatest annoyance
are the resulting weird network problems, not the wrong year itself.

Any clues or workarounds?

Andreas
