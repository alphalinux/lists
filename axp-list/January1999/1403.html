Dr. Andreas S. Tomiak wrote:
>
> 1. Boot Digital Unix 4.0c
.....
>
> (The hardware clock should run with GMT). The Difference between 1910 and 2047
> might be a result of the limited bit size of the internal Unix-time.

This is hardly likely. On Alpha time_t is a 64-bit type. Watch this
(on Linux-Alpha, of course):

$ perl -e 'print (scalar gmtime 2**55, "\n")'
Sun Jun 13 06:26:08 1141709097
$ perl -e 'print (scalar gmtime -2**55, "\n")'
Sun Jul 20 17:33:52 -1141705158

The range is indeed limited but I would worry about few other things
first before bumping into limits of that sort. :-) This range is
actually wider but you run into Perl argument/types handling if you
would like print bigger values in such simple manner as above.

Have you, by any chance, have set your clock type to ARC in
/etc/sysconfig/clock? If you are double-booting with DU then you
have SRM console, right? 'clock' program is supposed to be "smart" on
5.2 but maybe you have old binaries, or a new binary does not work
like it is supposed to on your hardware?

> The greatest annoyance
> are the resulting weird network problems, not the wrong year itself.

Lot of things will go haywire with a weird clock when time stamps
are not monotonically increasing.
