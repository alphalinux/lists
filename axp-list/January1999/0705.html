> This is STILL pretty puzzling. Weren't we told that EV6 will be/is so much
> more "brains" that is nearly automatically optimizes instruction sequence(s)
> and register usage (e.g. register renaming etc.) that optimizing code
> already by the compiler is much less critical than on ev5(6) ??

the 21264 is the first alpha which supports out-of-order execution. what
this means, roughly, is that within a group of instructions fetched
from memory, if the first one is still waiting on a dependancy (e.g. a
load instruction -- slow!), it can go ahead and take the next 4
instructions in line, and come back to the first one once the dependancy
is satisfied. the 21164 and previous would simply wait until the first
instruction is ready to execute, doing nothing in the meantime.

however, the processor still has to ensure that the execution behaves
exactly as if the instructions were executed in their specified order.
imagine trying to keep track of many in-flight instructions and once and
complete them all in-order by hand ... it is not a simple task! (i've
done it) so, while the 21264 is much more flexible in execution than the
21164, it still has very little knowledge to work with. by contrast, the
compiler knows much more about what instructions are interchangeable, etc.
the compiler also doesn't have buffer limitations, so it can look at much
larger sections of code in order to re-order instructions. that is why
(particularly on the alpha), it will always help to have a better
compiler.
