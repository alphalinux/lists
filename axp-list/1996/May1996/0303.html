>
> 1. unless Harald used the same "magic" as Erik Troan of RedHat, those
> servers will *ONLY* run on the EV4-based machines, much as my original
> set based on 3.1.1 and 3.1.2 releases.
>
> 2. the "magic" Erik used is available via the RedHat 3.0.3 LIBC; I am working
> the appropriate patches for 3.1.2E to make the *next* set of beta servers
> able to run on EV4 *and* EV5 machines, same binary executable. The same

I used the libcaxp-0.43-1.axp.rpm from RedHat 3.0.3 so it's includes all
the "magic" needed (being always curious I've looked into the sources first;
mainly to know if all out() routines really use mb()...)

but of course I couldn't test it on anything but my machine (but that's what
a beta test binary is good for ;-)
so please let me know if this binary works for you...
btw, what about Jensen ?

and are there other architectures out there aside from EV4/EV5/Jenson which
do need any other special support ? it would be nice to be able to have
real support for all platforms before the next XFree86 version is finalized...

> Now, on a somewhat different topic, is anyone:
>
> 1. interested in patches for the XFree86 Mach32 server to run on Alpha.
>
> 2. working on/interested in having, the SVGA server running on the Alpha.

YES! even for testing purposes I'd like to have an VGA16/SVGA server for
S3 cards etc. that's a nice method to figure out if some problems depend
on accelereated code or in cfb/mi routines.
when I was looking for the font bug I tried to build an SVGA server for
a short while but there were more and more x86 assembler routines so
I gave up and built Xvfb (isn't that interactive but at least for that purpose
it was helpful)...

also this would give at least some basic support for lots of cards out there.
support for as much cards as possible (also Mach64, P9000 and...) is important
for people how think about replacing their intel-based main board by
a cheap alpha board and want to re-use as much hardware as possible for the
first time to keep financial damage as low as possible.
