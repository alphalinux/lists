Hey Alpha group,

The statement was made a few days ago by no less than Linus that continued
binary compatibility of Linux with DEC OSF images is questionable.

I think losing binary compatibility with OSF images may be a mistake for
Linux/Alpha, especially before the compilers mature and the application base
expands. There are several reasons why binary compatibility may be a real
blessing (provided the effort of maintaining that compatibility is not
excessive)...

First, the current compilers available for Linux do not take advantage of the
speed that can be had from an Alpha processor. With the gcc compilers a
300Mhz 21164 processor runs at about 2.5X-3.5X the speed of a Pentium 90Mhz
machine. Not bad, but this is really wasting a lot of processor speed. I have
been running some benchmarks and code examples that show, with Fortran numeric
codes particularly, the 21164 Alpha sustains speeds of 10X-15X that of a 90Mhz
Pentium (LAPACK matrix operations with this machine are in the 100-200 MFLOPS
range). This is a 3 to 5 fold increase in speed and makes it possible to use
the Alpha for some really extreme computing tasks. This is largely due to the
DEC Fortran compiler and its high degree of optimization for this processor.
Although DEC does not appear to have any plans for selling their compilers for
the Linux market, if we are binary compatible we could compile a code under OSF
as a non-shared image and run it under Linux and get all this performance. Do
we really want to throw that away?

Second, there is a lot of software out there on the net that has been ported to
Alpha/OSF. It is not a rare occurence to see an application available for the
Alpha under OSF, while they are just not there yet for Linux/Alpha. Some
examples are the xforms library, the geomview geometry engine, the Sun binary
emulator and much much more - many of which do not have sources available. Do
we really wish to shut the door on this large list of potential applications?

How do the rest of the axp'ers feel?

Hal Y.
