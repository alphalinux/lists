>>>>> On Thu, 30 May 1996 10:56:13 -0400, "Michael K. Johnson" <johnsonm@redhat.com> said:

Michael> With libc 0.43, I've found these two lines in
Michael> /usr/include/sys/utsname.h:

Michael> #ifndef __linux__ #include <linux/utsname.h>

Michael> That looks like a typo to me, since there is no domainname
Michael> element except in the new_utsname in <linux/utsname.h>, and
Michael> especially since if __linux__ isn't defined, it doesn't
Michael> make sense for <linux/*> to exist.

Well, it's broken all right, but it's not the "#ifndef" that's wrong.
I think it should read "#ifndef __alpha__". The definition in
<linux/utsname.h> is _wrong_ for the old libc since it uses the OSF/1
utsname syscall (which returns fields that are 32 bytes long).

glibc has this fixed. It uses the Linux uname syscall, so it can use
the declaration from <linux/utsname.h>.

Michael> If it's not a typo, there should probably be a comment
Michael> there explaining why not.

Changing it to "#ifndef __alpha__" would be the right thing to do.
I'm not sure it's worth bothering, though. glibc is making good
progress and it is what will be used with ELF. Even if ELF is not
ready by the time the next libc release happens, I'd like to see
people start using glibc (but the way things are moving, I struggle
just to keep up with the progress that Eric and Richard are making
with ELF... :-). The only reason I'm not just uploading a
pre-compiled glibc is that some of the external file formats changed
(e.g., timezone and the utmp file). Also, all other libraries have to
be recompiled (X libs, tcl/tk, etc.). I have done this on my system,
but to make this generally available it would be necessary to put
together an entire distribution based on glibc. If somebody wanted to
start working on this, I'd be very happy to lend as much support as I
can.

--david
