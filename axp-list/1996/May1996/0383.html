> I have two versions of libdpml.a, an old one from OSF-1.3 and a
> quite recent one. Linux's ld has no problem linking with the old one but
> bombs out on the new one and returns error code 6. It appears the problem
> might be due to a change in the relocation entry format between the old
> and new versions as objdump works fine except for relocation information
> (-r) for which it aborts. the relocation area of the respective object files
> also looks different to the naked eye.

I just checked the header files on an OSF/1 3.0 machine here and found
that somewhere along the line, they added 4 new relocation types. The
current binutils (obviously) do not know how to deal with them and
will barf -- signal 6 is the result of an abort.

If you can find the object files in the archive that, when dumped with
"odump -r", has a relocation with type > 16, and can send me the output
of "odump -rt" and "dis", I'll see about adding those new types to the
next binutils.
