> > one argument against this scheduler (at least in it's current form)
> > is it's speed (cpu time taken): I run
> >
> > ev4sched -r XF86_S3
>
> Hmmm... so treating all of the X server as one optimization block is a good
> test of an optimizer? Try testing it on 10 instructions and multiplying that
> by 10,000 and I bet you'll get a far better number.

of course not. some lines down in my mail you can find timing data for
different block sizes which shows a minimum for ~64 instruction blocks
(changing the initialisation of the schedular might give even better results
for smaller blocks ?!)

> ev4sched is not meant to run on programs, just basic blocks.

I was just curious what happens when a "basic block" would be one subroutine
(or a large code sequence due to inlines) which can be (and is the the Xserver)
quite large.

but maybe it's always sufficient to use code between to jump/branch statements
as basic block because of the alignment of branch targets ? just guessing...
but I'd really like to see how much real improvement such a better code
scheduling could give for the current gcc (and how much CPU time it'd cost ;-)
