> It would, however, be an interesting project to replace the model
> currently in gcc with the one used in this scheduler.

one argument against this scheduler (at least in it's current form)
is it's speed (cpu time taken): I run

ev4sched -r XF86_S3

which is 2.8MB code size. this took exactly 1 hour CPU time on a 21064/266.
code rearangements for a ~550k code size numerical (fp, "shelx93") program t
ook 25min! but this can be improved by using smaller blocks:

block_size cpu time for ev4sched -r shelx93
[words]
16 182.98user 1.00system 3:03.95 elapsed 100%CPU
32 97.03user 0.57system 1:37.60 elapsed 100%CPU
64 73.35user 0.48system 1:13.84 elapsed 99%CPU
128 110.75user 0.69system 1:51.43 elapsed 100%CPU
256 336.05user 1.58system 5:37.57 elapsed 100%CPU
496 1426.05user 4.77system 23:50.57 elapsed 100%CPU
512 1487.34user 5.04system 24:52.00 elapsed 100%CPU

that's quite a lot just to get maybe 10-20% more performace (which is the
average ratio between cycle time with and w/o code rearangement). but on the
other hand for a final production version...

the following example shows that even these 20% are by far not enough:
here are cpu times for this fortran program (~20k lines) using f2c to
convert it to C (~40k lines) and then using either gcc-2.7.1 or the
DEC cc from DEC UNIX 3.2 or similar (I have to check this again).
both (static) binaries are then run on my XL266 on Linux.

gcc -O3 1556.99 secs
cc -O3 824.56 secs

this almost a factor of TWO:-(

I even tried "gcc -O3 -funroll-loops-funroll -all-loops-fstrength-reduce"
and similar with no significant difference (other than the compile time).
