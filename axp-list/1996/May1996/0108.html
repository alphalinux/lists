It really is time somebody write that paper entitled "Why Low-End
Multiprocessing is a Dumb Idea (unless your a nuclear physicist or
Intel)".

Yes, multiprocessing has it's place. But the problem with low-end
multiprocessing is that people seem to believe that if they stick N
CPUs in a given box, then everything will go faster by a factor of N.
There are several reasons why that isn't always true, some obvious,
some more subtle:

(a) Any given application will run at the speed of exactly one CPU unless
that application is parallelized. That is, you'll wait for gcc, make,
and povray just as long as you did before.

(b) Parallelizing an application will fix some of the problems with (a),
but the cost (and complexity) of doing so is not trivial. Furthermore,
many applications just don't parallelize all that well. Even in a good
case, it is often considered a great achievement if your application
speeds up by a factor of 3 when throwing 4 CPUs at it. Sometimes an
application doesn't speed up at all because the synchronization overheads
are so large.

(c) Job level paralellism is the cheapest (and IMHO the most practical)
way of exploiting multiple CPUs: you just tell make, for example, to get
N compiles going simultaneously. The problem here is that if you have
N apps going simultaneously you better make sure you have (i) enough
memory and (ii) enough I/O bandwidth. Considering that most low-end
PCs have super-fast CPUs on top of a crummy memory system with 16MB
of RAM and an IDE drive, how likely is it that a system with N CPUs
will have an adequate memory/I/O system? I claim that chance is virtually
nil. Why? Memory and I/O bandwidth, unlike CPUs, cost money. It's
as simple as that.

I'm not saying low-end multiprocessing never makes sense. There are
applications that are so compute bound that little else matters (e.g.,
computing Mandelbrot sets comes to mind). But most application I can
think of that currently are "too slow" either have substantial
workingsets (==lots of memory) or need lots of I/O or memory
bandwidth.

IMHO, here is where I think it is worthwhile spending money on:

- get as much memory as you can afford---the best CPU won't help
if you spend all your time waiting for the disk

- if you have a choice between a system with N CPUs or a system
with one CPU that's N times faster, you always want the latter
(all else being equal, of course)

In case it isn't clear: this has nothing to do with Alpha vs x86.
It's strictly single-CPU vs multi-CPU.

--david
