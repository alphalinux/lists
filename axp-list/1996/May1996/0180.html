Hello,

We've just got our new UDB (aka Multia) system in. It came with Linux
preinstalled, but we've been trying to set up MILO etc. to our liking.

After some difficulties, we have Linux 1.3.100 booting from MILO 1.3.95
(built with Linux 1.3.100). As per various recommendations we're using
the ARC firmware installed on our system.

However, MILO's environment variables appear to be corrupted by the ARC
firmware's settings and vice versa. In particular, when we run the MILO
"resetenv" command, the variables are cleared but the next time ARC is
booted it complains about corrupted NVRAM and all the settings must be
reset. After doing so, MILO once again complains about uninitialized
environment variables and (distressingly) a "printenv" in MILO spews a
bunch of corrupted text that resembles the ARC console's settings. Adding
more environment variables in MILO once more confuses ARC...

It seems to us like MILO and ARC are both trying to use the same area of
NVRAM for settings, but incompatibly. Yet I seem to remember that when we
first got the system it booted without any problems, so I suspect we're
doing something wrong. What could it be? Are there bugs in the latest
versions of MILO (or even the Linux kernel)?

We can boot the kernel successfully -- it's just the environment that's
messed up -- so we're doing _something_ right.

A similar problem seems to happen with the SRM console, though the SRM is
scary and difficult to comprehend.

Any ideas?

Dan
