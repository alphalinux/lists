>>>>> On Wed, 8 May 96 23:47 EDT, eric@aib.com (Eric Youngdale) said:

Eric> A few further thoughts. I am not trying to put you
Eric> off of the idea of optimizing in such a way that we don't
Eric> leave nops behind, but I just want to impress upon you how
Eric> difficult such a thing really is.

Eric> Normally the assembler will resolve some things such
Eric> as local branches without the need for emitting a relocation.
Eric> The idea is that the branch is pc-relative, so there is no
Eric> point in wasting time and space by telling the linker about
Eric> something which it doesn't need to handle anyways.

You know, the reason the OSF/1 optimizer flag is called -om is that it
uses the Object Management (?) technology developed by DEC (this is
the same OM as in AT*OM*, which allows you to do purify and pixie-like
kind of stuff under OSF/1---just more flexibly).

In other words: it's pretty heavy duty stuff.

The other tradeoff is execution time: what you can do at assembly
time, you have to do only once. This is particularly important for
(static) libraries. Worse: the size of object files is an important
consideration as well. We do not want to make object files any bigger
than the current ECOFF files are already. In fact, I'd like to see
them *much* smaller (particularly the debugging info is huge).

IMHO, binary translation would be the way to go. It allows you to
achieve the benefits of a smart linking without having to modify the
linker, the assembler, and the object file format all at once. It's
not for free, obviously, but I like the idea of having an optimizer
that's as independent of other tools as possible.

DEC is huge in this area, but I think it also means that it is about
as proprietary as it gets.

--david
