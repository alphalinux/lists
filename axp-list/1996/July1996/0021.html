> I just look into io.h.

> Address ffffbc0..... seems weird (to an ignorant as I am about Alpha).
> Are you sure that __reab() is expecting addresses that have been remapped
> by vremap() ?

no, these strange addresses are almost correct; this is the PAGE_OFFSET
(asm-alpha/page.h) or IDENT_ADDR (asm-alpha/io.h) and part of the address
for PCI/ISA bus mapping. from the Xserver sources I only know that the
ISA/PCI bus is mapped to BUS_BASE / APECS_DENSE_MEM = 0xfffffc0300000000UL,
so this really should be 0xfffffc0300000000 + 0x44000000 = 0xfffffc0344000000
AFAIK.

or for kernel symbols:
gdb /linux/vmlinux
(gdb) x/i init
0xfffffc0000310f98 <init>: 27bb0016 ldah gp, 1441792(t12)

> Perhaps we should use directly the bus base address (44000000).
> It is just an idea. I do not reply to the list because I am very probably
> wrong.

I've tried this too (one never knows;) but it really didn't work

> Have you some pointer to documentations about Alpha architecture?
> At the moment I am quite ignorant of that. I have to learn something
> about Alpha machines. I am very interested in such hardware.

I'd guess this should be all mentioned in

Alpha AXP Architecture Reference Manual
Second Edition
Digital Press/Butterworth-Heinemann
DEC order #: EY-T132E-DP
ISBN 1-55558-145-5
Library of Congress: QA76.9.A73A46

but I don't have it (yet?). so far I'm trying to avoid such problems
or find some AXP guru (they are all on this list ;-) who knows/tells me.
for the Xserver so far I only needed the bus base address, everything
else is in user space and seems to work nicely...

you can find some information in linux/include/asm-alpha/apecs.h (for my XL)
and linux/include/asm-alpha/lca.h (for UDB/Multia) which define

/linux/include/asm-alpha/apecs.h:#define APECS_DENSE_MEM (IDENT_ADDR + 0x300000000UL)
/linux/include/asm-alpha/lca.h:#define LCA_DENSE_MEM (IDENT_ADDR + 0x300000000UL)
