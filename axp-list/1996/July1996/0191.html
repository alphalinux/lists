On Jul 20, 4:33pm, Harald Koenig wrote:
> Subject: Re: Couple of newbie quesitons (sorry :)
> > The two reasons I asked were:
> >
> > 1> commercial releases. Linux is moving more and more mainstream, so
> > there are going to be a lot more "Unix newbies" picking up Slackware or
> > something similar, because $20 for aCD of Linux is a helluva better deal
> > than $100 for win 95 :)
>
> having to sets of installation binaries (and maybe one special "fat" binary
> to bootstrap the first Linux kernel to start the installation) should do
> this job. I can't see the point that I have to pay 100s of megabytes
> of disk space for unused machine code just to get installation
> a bit easier (if at all).
>
> > 2> For people who need both (ie nfs /usr/local/bin) you could save on
> > exe size if only machine specific code was duplicated, and not the data.
>
> automount, amd or similar tools are one possible method to handle this
> in a nice way; the easy solution is to mount server:/serv/bin_axp on your
> AXP boxes and server:/serv/bin_x86 on intel...
>
>
> I think the most important point here is Linux "marketing".
> it really might be important (at least for some types of users) to have
> *one* CDROM which they can use every machine in their universe
> and we should think about this a bit now that ports for other platforms
> are really usable (hi Red Hat, where is your x86/axp/sparc distribution CD
;-)
>
> but I can't find a reason why a sticker "now with FAT binaries" is the
> only (or best) way to achieve this...
>

Especially when you have glint/rpm which stores platform information, and could
be easily enhanced to install only the "right" binaries.

Or we all go p-Code, and starting distributing java-hosted versions of
commercial software. Which is where Microsoft began (e.g., MS-Word and it's
P-Code runtime environment which allowed easy porting to UNIX, DOS, Windows,
Mac, and strange hybrids like the Tandy 600).
