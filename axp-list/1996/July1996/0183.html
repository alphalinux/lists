>>>>> In article <199607160747.HAA19416@moondance.np.ac.sg>, Robert Allen Barker <rab@np.ac.sg> writes:
>> But why does it take such a long time (about 35 minutes) compiling a
>> kernel on my noname box (21066/233, 256KB L2-cache)? (it's equipped
>> with 64Megs RAM and runs the actual RedHat distribution.)

> For two reasons. 1. Compiling the kernel must be one of the least CPU
> intensive tasks around because of the large number of files, temporary files
> and passes that are required. Disk and memory performance are probably being
> compared much more than the CPUs.

Seems true. On an 150MHz(!) Jensen with 512k L2-cache and an 128 bit
wide main memory path, it just takes around 24 Minutes.

> 2. All the kernel source files, many of
> the temporary files, and much of the code generation is byte oriented and
> the current alpha hardware only supports 32 and 64 bit data types. The alpha
> must use extra instructions to extract, shift, and inset bytes. So in one
> sense byte oriented software is highly optimized for intel.

Even worse, the gcc-2.7.1 package commes with breakes on, as anything
else that has been linked with libc-0.43-1. The reason is that it
doesn't use the alpha specific memcpy routine, because of faulty
Makefiles. It uses an older one, which uses byte accesses (!) for any
size to copy. To make that clear: it uses about 8 times
2*ldq_u/extbl/extbh/ to load a quadword, and
2*ldq_u/insbl/insbh/2*stq_u to write it. This is a total of 48 memory
accesses instead of 2.

Using the optimized routine just with gcc (not as, not ld) gains 1.5
minutes. I expect, that other string library routines
could need some work, to improve overall performance.

>> The native 2.0 kernel on a 488/66 takes approximately the same time,
>> most pentium boxes do this job between 10 minutes and 25 minutes, PPro
>> machines below 6 minutes. If I compare the calculation power e.g. with
>> an (simple) dhrystone-test the axpPCI system beats the intel boxes as
>> well as an SGI with R4600/100. My impression of the performance working
>> with the noname machine is that the old 488/66 is much slower.

Also note,comparing native Linux kernel compilation on two
different architechures is not a benchmark: You actually use two
different compilers (an intel and an alpha code generator) to compile
two different programs (an intel and an alpha kernel).

If you'd like to get a meaningfull result for gcc, you have to use a
crosscompiler. But even this is not fair, as one of them will have
additional code to handle the 32/64 bit difference of the host/target
machine.
