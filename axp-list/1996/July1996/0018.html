All,
without going into too much detail (I cannot), there difference
between a Digital Unix/OpenVms and Windows NT CPU is that the SRM console
which is required to boot them will _not_ run on the WNT CPU. If the
loader cannot run, then neither can the OS. Note that the SRM console
does more than just load the OS, it also offers callback and console
services. None of which Linux wants or uses. One of the reasons that
I wrote Milo (the Linux Alpha Miniloader) was to free Linux from the SRM
and any future (now present) restrictions. So, so long as you use a
Windows NT board (sic) and Milo, then you are safe. My only concern is
that Linux sales may show up as Windows NT sales which skews the figures.
Note that Digital Semiconductor boards are sold by Digital, resold by
resellers, cloned and improved upon. Aspen is one company that takes our
designs and respins them.

Any and all future Digital Semiconductor boards (including the upcoming
PC164) will have Milo and Linux ported to them. Indeed, that has already
been done for the PC164. For various reasons, some systems from Digital
ship two variants, one for Windows NT and one for SRM. Where this is the case
then you've paid the CPU levy already and running Linux from the SRM is fine
(unless you prefer not to waste 3-4Mbytes of memory - to save memory, if there
is a Milo for that system; boot Milo from the SRM console). Porting Milo to
an SRM system that already runs Linux is an effort and sometimes this is not
always done. Particularly if no one lends me a system of the right flavour.
Porting Milo to non-Digital systems (eg Aspen, Platform2000 etc) has been done
by those companies, often based on an existing Milo implementation.
