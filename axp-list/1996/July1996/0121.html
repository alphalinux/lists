Hi all,

I've put a new dump/restore srpm and rpms up for ftp on

linux.dcs.ed.ac.uk:/pub/linux/utils/

-rw-r--r-- 1 linux linux 301319 Jul 16 14:56 dump-0.3-4.axp.rpm
-rw-r--r-- 1 linux linux 64853 Jul 16 17:20 dump-0.3-4.i386.rpm
-rw-r--r-- 1 linux linux 100415 Jul 16 14:56 dump-0.3-4.src.rpm

I'll upload to redhat's incoming shortly.

There are basically just two changes. First, there is a *very*
significant off-by-one bug in the documentation. The previous man
page claims:

<Dump levels> A level number above 0, incremental backup,
tells dump to copy all files new or modified since the last
dump of the same or lower level.

In fact, dump will copy all files new or modified since the last dump
of *strictly* lower level.

Secondly, the new dump is 64-bit clean. All of the normal 32-bit data
structures are declared with int, instead of long as previous; and the
quad_t datatypes are declared with Linux's __s64 and __u64 types.

There is one minor problem, however: the existing dump format reserves
only four bytes for a kernel time, but the kernel defines time_t as a
long. A 32-bit time will overrun some time next century (2038 or
something), but the Alpha's 64-bit time doesn't have that problem.
Since the dump format forces us to use 32-bits for time, I'm afraid
that the new dump won't work after the year 2038 on either Intel or
AXP.

However, the advantage of keeping the same format is that you can read
dumps written by dump-axp on an Intel machine and vice-versa.
