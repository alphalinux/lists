Here is a comparison of matrix floating
point calculations performed using Gaussian 94
(a matrix intense quantum chemistry system) on
1) 433 MHz PC164 running Linux, with G94 compiled on DEC UNIX Alpha 3000
with the public domain (PD) BLAS,
2) 200 MHz Pentium Pro (PP) running Linux, G94 compiled on PP with PD
BLAS,
3) 133 MHz Alpha 3000 (21064) running DEC Unix (DUnix) with
Optimized BLAS (DXML) libraries.
The same swap and memory resources were set on each machine.

The now "obsolete" 133 MHz
Alpha competes very well with the much faster
hardware, outperforming the 200 MHz Pentium Pro
in two out of three tests,
principly because of the optimized
general matrix multiply routine, DGEMM, provided
in DEC's DXML. The last two table entrys show
timings on the DEC Unix machine with and without
the optimized DGEMM. I'm hoping that this might
help generate a little more excitement about
getting some optimized matrix routines made available
for Linux. Just the one routine, DGEMM, would be
helpful.

Table. Execution times (seconds) for G94 test jobs:
_______________________________________________________________________

Machine/software/Op-Sys test050 test060 test000
_______________________________________________________________________

axp-433/PDK-DGEMM/Linux 48 1376 5
686-200/G94-DGEMM/Linux 216 3150 7

axp-133/OPT-DGEMM/DUnix 356 2200 5
axp-133/PDK-DGEMM/DUnix 343 !? 3478 28
_______________________________________________________________________

Test000: WATER STO-3G//STD WITH ARCHIVING,

Test050: MeOH freq//6-31g*,

Test060: Ethyl radical QCISD/6-31g* 5d test,

G94-DGEMM: Public Domain double precision general matrix multiply
routine supplied by Gaussian.

OPT-DGEMM: DEC DXML optimized basic linear algebra subroutine,

PDK-DGEMM: Double precision general matrix multiply routine
from the Public Domain BLAS libraries, optimized
using the KAP Fortran optimizer. The KAP optimizer
produces very different results, depending on
how the optimization parameters are choosen
and the size of the matrices used at runtime.
Generally, a DGEMM optimized by KAP is not
as fast as the DGEMM supplied in DEC's DXML
and the results for test050 above (!?) are
a surprizing anomally.

Note that it is illegal to perform these
calculations on the AXP/Linux system
using G94, or any other executable code,
compiled on the DEC Unix system
because of licensing protection applied
to the statically linked DEC libraries.
The DXML is further protected by calls to
the DEC license database that prevent
execution on machines other than the one
holding the license, so those calculations
could not be performed on the Linux system
(perhaps an NFS mount of the license database...)
Therefore I do not plan to use this particular
system in production runs, nor do I wish to
encourage anyone else to try this.
