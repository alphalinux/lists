I am going now through various errata and updates to Red Hat 4.0 Alpha
distribution. Apart from minor wailing from rpm from time to time
I run into the following:

ncurses-devel /usr/include/curses.h conflicts with file
from glibc-devel-0.960906-1
error: ncurses-devel-1.9.9e-3.axp.rpm cannot be installed

Yes, indeed, it is a real conflict and files are different. As that
one from glibc-devel seems to be a header for non-existent on a system
'libcurses' (we have only 'libncurses') then applying --force seems
to be a recommended course of action; with this exception that in
the future updating glibc-devel will clober our current
/usr/include/curses.h header (unless ncursed-devel will be not
reinstalled). Sigh!

Another gotcha. Reinstalled ghostscipt claims search path like this:

.
/usr/share/ghostscript/3.33:/usr/share/ghostscript/fonts

Still 'chess.ps' example from /usr/share/ghostscript/3.33/examples
does not search for 'cheq.ps' in any other place then the current
directory. I am pretty sure that the previous version did better
searching (and a problem with wrong font location was solvable
by one symbolic link). Putting links from 'cheq.ps' to either
of two directories listed above does not work and strace shows
that they are NOT tried.

Not of the special consequence, but... Checking what is going on
with curses.h I found that a directory /usr/lib/perl5/alpha-linux/5.003/
is chock-full of assorted .ph files. They were, quite obviously,
produced by an automatic run of h2p. There are some problems with
that:
- first of all these files are really superflous with perl 5
- second, Perl docs tell very clearly that h2p tries hard but
it is far from perfect and code produced does not even has
to be correct; it should be hand-edited after a run of h2p.
- third, these files are full of of stuff like
require 'features.ph'; and require 'stubs.ph'; and
code like that:
if (defined &_XOPEN_SOURCE_EXTENDED) {
}
if (defined &_XOPEN_SOURCE_EXTENDED) {
}
if (defined &_XOPEN_SOURCE_EXTENDED) {
}
if (defined &_XOPEN_SOURCE_EXTENDED) {
}
very exciting and possibly plain wrong.
The only saving grace is that these files really are only eating
up a disk space and that is it.
