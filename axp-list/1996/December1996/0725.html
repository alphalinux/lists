> Has anyone been able to get talk to work on their UDB running redhat 4.0?
> Whenever I try talk at console, it just flashes the talk screen and
> dies...So I tried ytalk in X, and it tells me that there's no talk
> server...what am I doing wrong?

quick checklist
---------------
1. check your ACL's
2. check your DNS configuration

explanation
-----------
1. make sure that you can allow connection to in.talkd &(/or?) in.ntalkd
from /etc/hosts.allow and /etc/hosts.deny. ie: if you have an
"ALL: ALL" line in hosts.deny (and if you have ANY conscieous and
you're on the Internet, you SHOULD), make sure to have at least
"in.talkd, in.ntalkd: ALL" or you might want to replace "ALL" with
"LOCAL, .insert.your.domain.here", depending on how anal retentive
you wish to be.

2. I had this problem with talk, ytalk, in.talkd and in.ntalkd under
Slackware on ix86, so this is just 'general advice': have the machine's
domain name in both the search line AND the domain line in your
/etc/resolv.conf. Like for a machine in the 'groundzero' domain:

domain groundzero.nyc.ny.us
search groundzero.nyc.ny.us dorsai.org medinfo.org
nameserver 206.127.32.33
nameserver 206.127.32.34

ytalk seems to REQUIRE the "search" line in order to operate properly
(ie: find the A records for the machine it's running on). I'm not an
expert in BIND, but I don't think it should be necessary if you have
your domain set (search I _think_ is just to make it easy to abbreviate
other machine names). Then again, RedHat doesn't enter a domain line
(at least not the 4.0 ix86 version I have) but makes all entries in the
search line (bug in rh4.0?).

I fumbled for a few hours the other day, and this seemed to work "ok". I
still couldn't get my SunOS 4.1.4 box to use "talk" to connect to my
Linux box (albiet, Slackware), but "ytalk" configured for non-X was just
peachy on both ends... FYI. =)
