VFS is Virtual File System and it is, essentially a common interface to all of the physical file systems.
All of the real file systems register themselves with this layer of software and, whenever a filesystem is
mounted, VFS uses callbacks to first mount it and then to navigate the filesystem, finding files and so on.
All Linux filesystems are represented in VFS by a superblock (the basic description of the filesystem) with
each file (or directory) represented by inodes (an ancient Unix concept). Whenever real filesystem stuff
happens, the code maps its data structures into VFS superblocks and VFS inodes via callbacks. This is
slightly confusing with the EXT2 filesystem as it does actually use Superblocks and inodes. The EXT2 Inodes
look a lot like the VFS ones and the mapping is easy and efficient. Buried in here is also access by the
filesystems to the buffer cache as well as VFS's use of the directory and inode caches. All of which make the
filesystems very efficient. Ted T'so (MIT?) was looking at the EXT2 filesystem extensions.

To see the code look in /fs and in /fs/ext2. I have seen some papers on this, probably at the LDP site, written
by Remy Card (inventor of EXT2) and Stephen Tweedie (expert and implementor of several go faster pieces). There
are a couple of books on the Linux kernel out, I cannot remember their name as I'm avoiding buying them; I
am in the middle of writing a book on the linux kernel myself and I don't want to be 'tainted'. Guess which
chapter I'm half way through writing?
