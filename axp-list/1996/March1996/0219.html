On Sat, 23 Mar 1996, Richard Henderson wrote:
> Well, my spiffy new computer finally arrived and I am very pleased.
> I do have a comment and some questions that have popped up
> during my installation. For reference, the computer is an eb164,
> 64M ram, with the standard ncr53c810, #9 s3-964P, and dc21140 components.

Very nice setup.

> 1) The 1.3.57 kernel that was distributed with RedHat 2.1 was almost
> unusable. I kept getting various kernel panics about every 5 minutes,
> more or less randomly. I managed to get up and running by doing a
> minimum install, then repeatedly adding packages, syncing and panicing.
> Once up, I could upgrade to 1.3.76 which has been stable thus far.

Indeed. The 21164 was not officially supported until something like the
1.3.67 kernel, and there were some mm instability problems with the
initial eb164 patches. With the current kernels it should be perfectly ok
(I have not been able to trigger any 21164-specific problems any more)

I hope the next RedHat alpha CD will have a newer kernel (and especially
newer MILO/LINLOAD stuff), so that the eb164 will be not only "supported"
but also easy to install ;-)

> 2) While I was able to get a working XF86Config on a friend's i386
> with more or less the same video setup, I've been unable to find the
> right magic for my alpha. The symptom is that the screen goes blank
> and the computer, as far as I can tell, locks up. Is there a known
> gotcha here? Does someone have a working XF86Config file for the
> same video card they can send me?

Did you get the eb164-specific X server?

The X server maps the IO stuff in user space and accesses the video
memory and IO ports directly, and the PCI bus is slightly different on
the eb164 vs the "normal" 21064 PCI setup. So the normal X server seems
to mess up by simply writing incorrect IO information.

With the eb164 version of the X server it works, although for me it
doesn't restore the screen correctly on exit (I haven't bothered to track
this down, as the eb164 I have is only a temporary loaner, so I haven't
actually wanted to get too attached to it ;-)

> 3) Neither reboot not halt manage to correctly unmount the filesystems
> before they lock the computer so that a fsck is always required on
> bootup. Is this a common problem, or something unique here?

I haven't seen this one..
