>>>>> On Wed, 27 Mar 1996 11:33:25 +0800 (SST), Robert Allen Barker <rab@np.ac.sg> said:

Robert> Hello all,

Robert> Any EB164 people have any experience or ideas on the
Robert> following?

Robert> I have an EB164 running with Redhat 2.1 and the latest
Robert> milo test image from gatekeeper.dec.com. I am trying to do
Robert> some programming and am seeing a trend.

Robert> 1. Software that has been specifically ported to the
Robert> EB164 runs and can talk to the hardware. This includes MILO,
Robert> the EB164 kernel, and the EB164 X Windows port.

Robert> 2. Programs that don't talk directly to the hardware
Robert> run too.

Robert> 3. Other axp software in redhat 2.1 that needs
Robert> hardware access doesn't run and causes memory
Robert> violations. This includes the graphics libraries, programs
Robert> compiled with the graphics libraries, and even the io
Robert> portion of libc.a (with _inb() etc).

Robert> 4. This is due to the libraries not yet being ported
Robert> to the EB164 hardware environment?????

The current libc has the in/outb routines statically configured. This
wasn't a big problem so far because all ev4 based PCI machines use the
same I/O architecture (Jensen users may have a different view on this,
though... :-). The ev5 is different because of its larger physical
address space. The solution is to make the inb/outb routines
adapt dynamically to the system it runs on. I have implemented this
for GNU libc and it appears to be working. I also sent that code to
Jay, but I'm not sure whether he has had time to look at it already.
In any case you'd need to recompile the offending binaries. If you
already have the right libc installed, this should be fairly easy (rpm
--rebuild <rpm-filename> as root should do the trick; really
handy...).

--david
