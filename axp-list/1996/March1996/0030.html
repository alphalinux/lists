Jim> Hi, I have seen some discussion about how to get a MILO booted
Jim> kernel to reboot. Has anyone given any thought to getting an
Jim> SRM kernel to reboot? My gut feeling is that it is probably
Jim> easier since I believe that SRM stays in memory after the
Jim> kernel is running, but I really do not know. Does anyone have

Switching back to SRM is trivial and does work. Problem is: you may
not see anything on the screen... :-)

What happens is that the SRM makes some specific assumptions on what
the state of the hw is at the point the call_pal HALT is executed. In
particular, the graphics card needs to have the starting address reset
to the beginning of the memory (i.e., if you use hw scrolling and go
back into the SRM mode, the SRM console will write to the "middle" of
the screen, or worse, off-screen...). Also, the virtual page table
location would need to be reset (Linux moves the address of the
virtually mapped page-table, which the SRM doesn't seem to like).
Finally, I remember that not reinitializing the PIC (interrupt
controller) may also cause problems. This may have changed though
since I last used the SRM console.

In summary: it would take a routine

reset_that_dang_hw_the_way_SRM_wants_it()

and you should be all set. AFAIK, there is no documentation as to
what the SRM expects, so it would take some trial and error.

Getting MILO to work is probably more fun and of longer lasting value.
