(I hope that the Digital Guys concered also reads this list, as I'm "giving
up" on vger..)

Details: Noname, RH 2.1, kernels 1.3.59-66

The system used to be Milo 1.3.34.. I ran the test-images/noname/fmu.gz
binary from my old milo. It upgraded the system wo any problems. It also
works perfect! :-) ... But..

Reboot is a bit 'mal-functional'.. typing shutdown -rf now shutsdown the
system, makes a reboot (putting the graphics card in a non-sync mode for short
perdiod), getting me back to the milo prompt. But when I now types 'boot',
milo reads in the new kernel into ram, but as soon as its tries to execute
the kernel it just "crashes"... :-( Because it dumps tons of info on the
screen, and scrollback doesn't work, I don't have some many details about it..
But I get the following message every time I boot, maybe that can give a
hint..

#lca: machine check (la=0xfffffc00002084d0,pc=0xfffffc000025b494)
Reason: access to non-existent memory (long frame):
reason: 100000084 exc_addr: fffffc000025b494 dc_stat: 3
car: 60706541
A write error to b-cache occurred at address 15010d0
Attempted to access non-existent memory.

Another thing that I would like is that "reboot" automagically starts the
same kernel as last time (using the same 'bootoptions'), and that "halt"
gets me back to the milo prompt, and doesnt boot a new kernel.
I guess that booting automagically, is just a matter of setting ACTION = boot
, or whatever the correct syntax is... The problem with this is thoose few
times I've tried this, is that its have been "very hard" to 'break' the
automagic boot (for say booting different kernel).. The easist way I found
was to remove the scsi-cable...
