David Mosberger <davidm@AZStarNet.com> answers my question about ECC:
> At least, that's true for the Noname: ECC enabled, I have seen a
> *very* occasional ECC machine check (correctable, fortunately :-) and
> the PALcode does the scrubbing.

Great! I'm not sure we're in agreement on the definition of scrubbing,
though. I've always heard the term scrubbing used with regard to systems
in which the hardware will detect correctable errors during memory refresh
cycles. This guarantees that any correctable errors will be fixed within
the amount of time that a complete refresh iteration takes times the number
of banks that must be refreshed. Otherwise a memory location that was
accessed infrequently could accumulate errors over time.

For this to work, the hardware typically would generate normal read cycles to
one bank of memory at the same time as RAS-only or CAS-only refresh cycles to
the other banks. This obviously also requires that the refresh counter be
able to address all of memory, rather than just the maximum number of row
addresses in a bank.

The other thing that would be nice for them to do in the memory controller
would be to include the address bits in the ECC computation. This would
allow detection of addressing failures. Of course, even a single bit
address error is uncorrectable.

I don't have a 21066 hardware manual, so I don't know if they built scrubbing
or address checking into the memory controller.

I suppose the PALcode *could* do scrubbing, but it wouldn't be as efficient
since it would use memory bandwidth beyond what the hardware is already
using for refresh.
