Milo doesn't initialise that much of the HWRPB, just enough to fool
linux (or to put it another way, to allow linux to tell what system
resources it should use). I'm happy to add in more stuff so long
as I can deterministically find it out or hardcode it. Folks are
welcome to send me patches to hwrpb.c.

As for generic linux kernels. In the old days we had several
generic kernels (one per PCI/memory support chipset) and used the
HWRPB to differentiate between them. This gave an overhead to
all interrupt handling (each interrupt causes some decision to
be made) but not much else apart from image size. By the way, the
kernel is larger now than it was then but it's still not out of
hand. I have been kicking around an idea which goes along the lines
of this:

Milo would load a system specific interrupt "module" into a known
location,
This would handle all of the interrupt stuff, including request_irq
and so on. All this stuff is currently table driven and
so there is no extra overhead to receiving interrupts, just
to asking for them, disabling them and so on. That would
have to go through a jump table or some such.
This is not a burden for Milo - Milo _has_ to be system specific.
The result would be a set of system specific Milos and 3 or
so "generic" kernels (21066, 21064+Apecs, 21164+Alcor, Jensen).

Now that I've done the PCI-PCI bridge support maybe I'll spend some
time on this...
