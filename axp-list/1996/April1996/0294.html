> Can someone tell me what the status of glib.c for linux-alpha is?
> I have some C++ codes that need work.

Yes, I can. The latest glibc snapshot plus some additional patches by
me allow glibc to compile and work. There are still quite a few
header files that are either wrong (they should be based on the Linux
kernel headers, but instead define their own, sometimes incompatible
constants etc.) or simply missing.

The FSF maintainer of glibc should be back from vacation shortly and I
hope we'll be able to resolve the remaining header file issues soon.
(Note: as mentioned earlier, I'm buried with work until after first or
second week of May. Till then, I have practically no time for Linux
work.)

Emacs used to work under glibc but the newest pretest doesn't.
Similarly, ghostscript works worse when using glibc compared to the
old libc. I suspect memory manager problems, so that needs to be
debugged.

strptime() is missing; if somebody knows of a version that is *really*
in the public domain, let me know. Don't bother if it's not public
domain since FSF is rather picky about copyright issues. Even better
would be if it had been written by sometbody who has signed a
copyright assignment with FSF already.

Timezone support doesn't seem to work quite yet either. Even after
setting TZ and/or /etc/localtime to "US/Arizona", I either get GMT or
EST.

/etc/host.conf support is missing. I started implementing this and
made good progress, but see above comment regarding time... If
somebody has good documentation on what some of the options in there
really are supposed to do, I'd love to hear about it (the existing
code in HJ's libc is, shall I say, "suboptimal"). E.g., what are
"trimdomains" *really* used for?

Despite all these missing things, it is possible to compile
non-trivial C++ programs with glibc+libg++. The problem is that glibc
and the old libc are not binary compatible and the header files are
wildely different. Thus, putting together an environment that would
allow coexistence of the old libc and glibc would be quite
timeconsuming (e.g., consider /etc/localtime). glibc definitively
isn't ready to substitute libc.

Sorry for not having better news, but let's not forget that for a fair
number of developers Linux is still basically a hobby.

--david

PS: I don't want to sit on the patches I have. If somebody else wants
to take over, just give a holler. I have a long list of other things
that I'd love to hack on.
