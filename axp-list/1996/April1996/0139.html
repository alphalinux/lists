> We have three EB164's which all exhibit some very strange
> behaviour. We often find that simple commands like 'uname','tar',
> 'mv', 'id', 'cp' fail sporadically, even when invoked with
> trivial or no arguments e.g. 'uname','id','tar --v', 'mv --v'.
>
> When a particular command has decided to get itself into this
> state, invoking it repeatedly will cause it to fail with a memory
> violation at the same PC. Leave the shell alone for 10 seconds
> and try it again, and the command will magically work. Repeating
> it immediately will cause it to fail at the same PC again. Once
> into this state, the behaviour is totally repeatable!

Yes, I've seen this; during boot-up, "touch" repeatedly memory violates;
I'm assuming that it is *after* the first time when it ran correctly.
Also, the *first* time I "startx", the server comes up just fine, but if I
quit out and do another "startx", I get error logged messages saying that
".Xclients" has memory violated, and the server comes down (it's not
crashing, its just that its controlling process, ie .Xclients via bash, has
terminated). Note that this has been with the TGA server, as the S3 server
seems to work fine; go figure... :-(

> All our EB164's are Redhat-2.1 axp installed, and the problem
> exists with all kernel versions we've tried up to and including
> 1.3.85. All the kernels have been patched to disable DISCONNECTS
> in the 53c810 driver.
>
> The exact same binaries run fine on a Redhat-2.1 EB66+
>
> Can anyone shed any light on this please ?

The problem *feels* like one which shows up when executable files have been
cached or sticky-bitted or whatever, and have stayed around in memory, and
the *next* time they get used, use old addresses or data or something,
instead of all fresh stuff.

I have seen the above symptoms with 1.3.84; *HOWEVER*, a 1.3.81 kernel seems
to work fine; ie no "touch" or ".Xclients" problems at all.

So, maybe something in between changed. I looked, and Linus' notes for 1.3.84
included:

- alpha patches to support PCI-PCI bridges
- file locking is better
- kmalloc() is slightly faster
- new pages are appropriately marked dirty if written to (results in fewer
page faults on architectures that do the dirty bit in software, ie alpha)
- use atomic operations instead of disabling interrupts for skb free'ing.

His notes for 1.3.83 didn't have anything suggestive, and I couldn't even
*find* an announcement for 1.3.82, so... :-(

Anyway, that's as far along as I have gotten, maybe this will be helpful
for others, as the VM stuff is *not* my specialty... :-) :-)

Good luck.
