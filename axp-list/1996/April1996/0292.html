At 8:49 AM 4/17/96, Matthew Geier wrote:
> A user on my home system complained yesterday about their pop client
>bailing out. When I looked at thier mailbox, the above was exactly
>what I saw.

Well, we think we found the problem that both of us are having. Actually,
Chuck <alcorn@within.com> found it.

We suspected a locking problem. We are using sendmail as our SMTP server
and procmail for our local mail delivery. Chuck set procmail to log in the
verbose mode and noticed the following...

1. When procmail ran, it ran with the uid of the user who was receiving the
mail. This is due to the "-d" switch as setup in sendmail.cf.

2. Since procmail wants to create the lock files in the /var/spool/mail
directory and it is owned as (root,mail) and does not allow world to
have write access, procmail would fail to create a lock file (Aha!)

What is the recommended fix for this? As a temporary measure we gave world
write access to /var/spool/mail (we don't like this as a long term
solution). This allowed the creation of lock files and we haven't seen the
problems since.

So the real question is: what privileges are screwed up, and by whom? We
installed a standard RedHat install. Does the same problem happen on x86
versions of Linux?
