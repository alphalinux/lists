OK, I think the 53c7,8xx driver's DISCONNECT feature is indeed still
broken. The canonical test that _used_ to trigger the bug works fine
since we did the interrupt handling fixes, but yesterday I tried some
heavy/long builds on the Cabrio and every so often it would crash.
There were always a _lot_ of error messages from the SCSI driver. But
I suspect if you run X, you might not see those messages since syslogd
won't be able to write anything to the disk, once the drivers is in a
messy state. In fact, even configuring syslogd to log things to a
remote machine didn't help. I suspect that syslogd got paged out
during the build and then when the SCSI driver went berserk, syslogd
got active and the kernel attempted to page it in, but since the disk
is inaccessible, that doesn't lead anywhere...

In any case, I disabled DISCONNECT and did four or five glibc builds
without any problems. This does explain why my Alcor system never
suffered from this problem---it uses the qlogic SCSI adapter. Mikael,
does disabling DISCONNECT help on your system too?

As Craig (I believe) pointed out, another option would be to use the
ncr53c800 driver.
