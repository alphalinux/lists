Miquel van Smoorenburg <miquels@cistron.nl> writes:

> I've re-written clock(1) to use /dev/rtc instead of inline assembly
> (which only works on the i386 ofcourse). Now this works fine on the
> i386, but on my Jensen it lets the machine hang.

Actually, there is a version, which uses libc's i/o
functions. You can find a working binary on my web page.

> Even a simple "clock -r" locks the console. Console swithing is still
> possible, numlock works and even telnetting into the machine works.
> You just can't type anything on the console..
>
> About 5 mins. later the whole machine locks up.
>
> Any ideas?
>
> This is on a Jensen (DEC PC/axp 150)

The Jensen has its RTC located at a different place than all other
Alphas and the Intels. Its starts on port address 0x170, instead of
0x70. This caused to lock the clock program for a while, because it
waited for the RTC to turn some flag in an (almost) infinite loop. If
you do something like that in kernel code (/dev/rtc), this could lock
your whole machine, I suppose. Maybe /dev/rtc wasn't ported in a clean
way. You could use the RTC_PORT define, located in <asm/jensen.h>.

BTW, I thought /dev/rtc was just a timing device, not an absolute time
base? Thus I didn't bother to use it...
