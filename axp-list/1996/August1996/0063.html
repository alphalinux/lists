Martin Cooke writes:
>
> Can someone point me to a good backup program that will work on
> our alpha (multia). It is running the 2.0.8 kernel and the tape drive is a
> conner python 2 gig dat. I tried useing tar, but it choked:
>
> [root@beam linux]# tar -cvW -X/etc/tar-backup/exclude -f /dev/nst0
> tar: Read error on /dev/nst0: I/O error
> tar: Read error on /dev/nst0: I/O error
> [you get the idea..]
> tar: Too many errors, quitting

I have a similar problem. My config is:

UDB (w/ 350 internal)
DEC 2G drive (external case)
WangDAT 3200 (external case, same case)
Kernel's 2.0.0 through 2.0.10 ( compiled w/ gcc 2.7.1/ecoff )

The DEC drive is at the end of the chain. Everything works fine,
except the WangDAT. The WangDAT works just fine on my x86 linux box
w/ an Adaptec 1542. I have tried other DAT drives on the UDB (HP 4
gig DDS-2 DAT) and got the same results. As best I can tell the SCSI
controller/driver is having troubles talking to the tape drives. I
have also tried playing with the kernel configuration for both the
NCR7,8xx driver and the NCR8xx driver. I can effect different
behavior this way, but in all cases I eventually get I/O errors on the
bus w/ the DAT drive. In all cases mt works just fine to rewind the
tape, and do a status, although operations that require reading (like
fsf) fail.

I have come to the conclusion that the problem is somewhere between
the device driver and the NCR controller. I'm going to dig into this
further but it's not at the top of my list of things to do. If anyone
else would like a brain dump of the information I've amassed I'd be
glad to pass it along.
