I've been listening in on the Alpha listserve and your comments caught my
attention. I have a few question that I hope would be able to clarify
before I purchase a UDB.

You mention that you got better performance using the Digital Unix compiler.
I can undestand that as I would assume that Digital would know it's
processors better than GCC. Would it be possible/recommended to compile the
Linux kernal and take advantage of this optimization for the kernal?

You mention that the UDB was never intended for performance work. That's
ok, cause I just want a web server to act as a virtual web host. I see the
system demands being mostly TCP/IP and disk access. There may be databases
queries at some point in time but I dont see this as be very heavy. The
server software is multi-threaded not fork().

Do you feel that this would be an acceptable use of a UDB 266?
If I compiled the server deamon with the digital compiler, should I
get a boost in performance (vs. gcc)?

Would you compare Linux and Digital Unix? I'm looking for things like
performance, ease of administration, etc.

If I have the GCC libs, can I use the Digital compiler "out of the box"?
How is the alignment requirement handle?
Does the Digital compiler correct alignment errors or at least notify me
during compile?
Are there any hardware compatibility issues that I should be aware of?
Which TGA model does the QS1 motherboard have?
Does v3.2 come with an X server that supports the TGA ZLXp-E2 or E3?
And lastly, what is list price for v3.2c?

Thanks,

>There are a couple of relevant factors here:
>
> o gcc for Linux/Alpha is not as good as it could be. When I
> compiled the programs under Digital UNIX V3.2C with:
> cc -migrate -O5 benchi.c -o benchi -non_shared
> cc -migrate -O5 benchf.c -o benchf -non_shared
> and run them on my 'Jensen' (150 Mhz DEC 2000 AXP)
> they run in 17.0 and 14.1 seconds for integer and float
> versions respectively. About half of the time of a gcc
> compiled version running on a faster machine.
>
> o 21066 and 21066A have relatively poor memory bandwidth
> capabilities, compared to 21064[A] and 21164[A] chips.
>
> o The UDB's weren't designed to be high-performance machines.
>
>If you are doing performance critical work, a UDB is the wrong choice.
