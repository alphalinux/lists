> I don't understand the part about "wants to be IRQ16". As it's a PCI bus
> card, it should be able to be any IRQ that the OS wants it to be. It's IRQ16
> because you placed it in a specific slot, and the card is configured for a
> particular INT[ABCD]. Since the PC164 uses IRQ16 for slot 0 INTA, I assume
> that INTA is what the card asserts, and that it's in slot 0.
>
> Have you tried moving the card to the other PCI slots, especially the 32-bit
> ones? (From the above description of things you've tried, I expect so). Did
> you notice any change in behaviour?
>
> --Jay++

Yes, I have tried all slots and int lines A,B,C, and D. when MILO goes to
look for devices I always see the following line rerardless of which
slot/int lines:

scsi-ncr53c7,8xx : at PCI bus 0, device 6, function 0
scsi-ncr53c7,8xx : NCR53c810 at memory 0x1000000, io 0xb000, irq 16

if the card is NOT the last slot (64 bit) AND int line A then I get a scsi
device time out message and the system hangs. Thats why Isay the card wants
to be int 16 everywhere, though I admit I don't know what I mean by that.
This happens with both the EB164 aand PC164 and this is not really a problem
with MILO in that MILO, NT3.51, NT4.0, and DEC UNIX 3.2D all hang in some
fashion if the card is not set that way. The interesting thing is that the
the ARC console and SRM seem to work fine no matter where the card has been
placed making me think they are no using interrups for the scsi card. I
agree this seems strange but believe it has something to do with the card.
Its a cheapy from Taiwan and perhaps should be tossed but I don't because
none of my 9 spare scsi controllers in my drawer will support all 4 OSs.

When I mentioned a problem with MILO with the PC164 its becasue the card
sporatically casues a problem even when in the last slot of the PC164. I
have discovered that it usually hangs after a power down cycle but after a
soft boot seems to behave well.

I doubt if this problem has anything to do with MILO so the header to my
message was poorly chosen. I thought it might be of interest though as I too
thought MILO and later the kernel would lay out the PCI bus to their liking.
The only thing I can think of is that the designers have done something
funny with the interrupt number register in the cards configuration space.
