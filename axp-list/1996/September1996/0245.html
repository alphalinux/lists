On Tue, 10 Sep 1996, Abhay Manusmare wrote:

> I finally have a nice working 2.00.17 kernel thanks to this group,
> however I have a few questions about some boot-time errors. I was
> hoping that someone could explain the errors to me and how to fix them.
> I'm running Red Hat 3.0.3 with updates to 9/6/96 on a Jensen with 80MB.
>
> Error 1:
>
> PPA: unable to initialise controller at 0x378, error 1
>
> I was wondering what PPA is (so I'd have at least a starting point). I
> know it must be something I added with the new kernel since the old
> kernel (1.3.72) doesn't show this message.

The PPA is the Parallel Port Adaptor. This is the parallel port SCSI
adaptor used by the Iomega Zip drive and others. 0x378 is the address
of "lpt1" on most systems. It sounds like you have compiled in support
for the zip drive and the system is complaining that it does not find it
(at least on that port).

>
> Error 2:
>
> fcntl_setlk() called by process 50 (syslogd) with broken flock()
> emulation
>
> This message doesn't show up under kernel 1.3.72 and prevents the
> syslogd daemon from loading.
>
All I can remember about this was that it was a common problem with
newer kernels in the intel world. Check the notes in your new kernel
sources and you will probably find some reference.

> Error 3:
>
> /etc/rc.d/rc: /etc/rc.d/rc3.d/S99local: Permission denied.
>
> This one really stumps me. I've checked everywhere (rc, rc.sysinit,
> rc.local and on and on...) but haven't found anything to fix this. I
> re-linked S99local to rc.local and checked all the scripts I could
> find. This shows up under both kernels and I think it may be related to
> the upgrade to Red Hat 3.0.3 that I did.

Not sure about this but on the assumption that S99local is a script
of some sort be sure that it has both read and execute permission.
Unlike a binary executable a script has to have read permission as
well as execute since the shell has to "read" it before it can run.

Hope this is of at least some miniscule help.
