>I really don't think that is going to work. It would require that a lot of
>stuff like libc, and libX* use identical data structures etc.

This sounds a lot like the challenges we faced when we wanted to be
able to run SVr4 binaries under Intel Linux.

Given that the machine is basically syscall compatible, you basically
don't need to mess with the kernel at all. This was the first challenge
we faced (the SCO binaries and SVr4 binaries use a very similar interface,
so to run static linked SCO binaries the kernel end of things needed to be
done anyways).

The libraries aren't really that big of a problem.. The problem is
that you need to have a publicly available dynamic loader that is ECOFF
compatible where none currently exists, and for this, someone needs to sit down
with the ECOFF documentation and a bunch of ECOFF libraries and figure out
exactly what is needed.

Technically, the next step is to fix binutils so that it can build OSF
shared libraries, but this is no 100% required, at least to start with. When I
first started with the SVr4 stuff, I was able to bootstrap things using the
SVr4 linker running under Linux. Yeah, you need the license, but if you
already have it, you probably should be able to bootstrap things in this way.
If everything else falls into place, then yes, you would eventually need to
fix binutils so that you can generate an OSF shared library.

Once you have a dynamic loader, the next step is to come up with
a libc that works. When we did the SVr4 stuff, the hardest bit was stdio,
since it was basically pretty ancient compared to anything else we had
ever used. Someone sat down and wrote a stdio implementation that was SVr4
compliant (and interestingly enough it outperforms the current versions).

The main place you are limited in terms of the stdio implementation
are in that stdio.h defines a bunch of macros to operate on FILE * structures,
and whatever you use must be compatible. If OSF doesn't use macros but uses
functions instead, then you have no problems at all in this area.

If someone considers this a worthwhile project, I would be happy
to give the benefit of my experience. If I were to attempt to do it myself,
it would probably take far too long as I have too many other things I am
trying to do. I do have access to OSF at the office, so I could poke around
on my own if I really needed to.

-Eric

In passing, I want to toss out one other idea. It *may* be
theoretically possible to convert an OSF binary to ELF format. Internally
they aren't that much different in a number of places, so it might be possible
to do this. If this can be done, then the need for a publicly available loader
can be finessed, and all you would need would be the libraries.
