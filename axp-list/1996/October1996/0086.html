Over the past few days the recent kernels that I tried on my
EB164 crashed several times, often just a few minutes after
booting and running X (but not always after a big compile job).
The machine had been running nicely for 6 months with 128 MB,
NCR53c810, S3 VGA, 21040 Ethernet.

This morning it would boot only transiently, sometimes booting
OK to Milo, then stopping at the point where the kernel unzips in
memory, or sometimes rebooting Milo over and over. A few times,
it went as far as booting the kernel. But I've spent several
hours today rebooting and trying different things...

I connected a terminal to ttyS1 and can boot the minidebugger
nicely by jumpering (#11 as in milo.howto). The memory test
("walking 1's") checks out OK, and I've tried swapping memory
with other machines and other sizes (32 MB, 128 MB, 256 MB) so
the RAM seems OK. I can read the image headers from NVRAM OK.
Earlier in the day when I could still run "fmu", I reprogrammed
Milo into NVRAM several times, wondering if maybe the NVRAM image
might not have held.

When I set bootopt=1, Win NT ARC runs as it should, but when I
use it to boot through linload.exe to Milo, it stops before Milo
comes up and it goes back to reboot to ARC. When I set bootopt
to 131, Milo in NVRAM will occasionally boot but now mostly I
just get a printout on the serial port about checking the BIOS,
which normally happens just before Milo comes up. However there
is often 1 garbage char after the BIOS printout, and when this
garbage char appears, Milo does not come up and the system
reboots through the same sequence. The boot through Milo in
NVRAM is very intermittent, sometimes working once in 10 power
off/on cycles.

Evidenly both RAM and NVRAM and the hard disk are all working,
but from the irregular boot pattern, it seems like there must be
some mechanical intermittent connection... I have wiggled the RAM
and cache SIMMs and some of the other socketed chips since these
are often weak links in a big system (especially after a few
months). Sometimes this seems to work, after 10 boot attempts,
sometimes it will boot after wiggling some of the RAM.

The problems appear to have started after I upgraded to the
recent Milo for EB164, so I've been trying to eliminate
"software" as a problem by going back to the older Sept 6 Milo.
So far I haven't been able to get the old Milo back in NVRAM since
in the current state Milo won't boot at all.

So it sounds like hardware, but the Minidebugger seems pretty
reliable. Maybe it's NVRAM, but Win NT ARC runs OK when it
boots. Is it possible that NVRAM could give transient errors
but then come back OK once in 10 tries ?

Anybody have some advice on this ?

Ideas about what to try or what the problem most likely is ?

Thanks,

Rob Smith
