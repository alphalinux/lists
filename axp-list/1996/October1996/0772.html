Sorry for continuing on endlessly, but I need some more help. I never
thought I would actually consider downgrading back to RH 3.0.3, because I am
having *unbelievable* troubles using RH 4.0. (BTW, I did a full reinstallation
twice. Bug: some packages, e.g., libg++-dev are installed even if they are
not requested.) Here are a few things which I have not been able to resolve:

- Can't compile *any* post 2.0.18 kernels (only the 2.0.18 RPM works)
(machine=UDB; some includes (timespecbits.h) are missing, parse errors
with sound driver, etc) I tried moving some of the missing includes over,
but the it still dies with various parse errors upon compile. Do I need
the latest axp-diffs? Is there a HOWTO on axp-diffs? With 3.0.3, I never
had probs compiling any kernels.

- Upon reboot/halt, sda3 fails to umount, because it's busy. Also,
rc.sysinit fsck's sda3 *every* boot. Is this because sda3 can't
unmount in the first place? Any fixes?

- While networking appears to be better (with nsswitch.conf), it is no
where near as usable as RH 3.0.3 was. I'm only using a static PPP
(not dedicated) and my nameservers fail to be accessed with some programs,
e.g., the Mosaic RPM doesn't recognize names, only IP numbers. I'm
convinced something is fishy here.

- XQuake core dumps. It used to segfault upon cosole initialization due
to a networking misconfiguration, but that's been fixed. Now it crashes
fvwm completely during color map initalization and sometimes when
allocating mem. I tried multiple downloads of v101, both xquakes, and
fvwm & fvwm95...no dice, same crash. XQuake ran flawlessy in 3.0.3.
Is this still a networking problem?

After spending $100 on 4.0 (I didn't even know about the upgrade price),
I'm starting to regret it. Granted Alpha ELF is fairly new, but these 'bugs'
seem a bit too obvious. Am I the only one with these problems?? I would
really love to avoid going back to 3.0.3, but at my current rate, it seems
very probable. Thanks for any help.

Jonathan Zimmitti
