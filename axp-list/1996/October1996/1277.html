>
> Sorry if I'm overlooking something obvious, but how do I get f2c to
> map INTEGER*4 to 4-byte integers on Linux/AXP?

This is broken. I was posting in the past on the topic but without
a trace of a comment from anybody. It may, or may not, have also
an impact on a performance. AFAIK nobody was doing testing.

Why exectly you need that? There are some workarounds.

>
> >From the f2c man page:

I am afraid that the manpage does not correspond to the current
reality, although obviously it should.

>
> Also, f2c.h has
>
> typedef long int integer;
> typedef short int shortint;

This was likely done so things will compile under MS-DOS and is definitely
incorrect for Linux/AXP. Note that if you change that then f2c libraries
will need to be recompiled and if they recompile cleanly and correctly
remains to be seen. If you have around some other object files/libraries
they will become incompatible with whatever you will compile with a
"new" f2c. This will make you also out-of-sync with Red Hat rpm system,
which means that the next update will likely clobber your efforts if
you are not careful.

I run into the problem in the past when trying to port 'vg' graphics
library, written in Fortran and C, to Linux/AXP. I did the port
but it required some hacks.

I did not try to pursue f2c issues as personally for me they are not
really relevant and I do not have an easy access to Linux/AXP at the
moment. Still this should be fixed but I did not notice any interest
in that in the past.
