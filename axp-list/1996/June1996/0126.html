I'm currently using Milo-1.99.3, Linux-2.0 (copied to a eb164.img
image based disk) and Redhat-3.0.3 and the following boot commands:

boot fd0:vmlinux.gz ramdisk=1 root=/dev/fd0 floppy=0,4,cmos
boot fd0:vmlinux.gz ramdisk=1 root=/dev/fd0

In both cases when I switch back to the boot disk (as directed) and
press enter, I get a stream of errors along this line:

floppy0: disk absent or changed during operation
end_request: I/O error, dev 02:00, sector 2
...

where there are a variety of different sectors listed, the grand
finale is:

Kernel panic: EXT2-fs panic (device 02:00): ext2_write_inode: unable
to read inode block - inode=122, block=20

Now, what you're probably saying to yourself is that I'm suffering
from a case of the rotten floppies, but I've used several different
brand spanking new floppies, low-leveled them, fsck'ed the ext2
images, etc. Nothing indicates any physical or logical flaws in the
file system. I tried various kernel revisions, to no effect. I doubt
it's the floppy drive itself, or the controller, I've been using it
without any problem for quite a while now. Just for the record I have
an Aspen Timberline system (EB 164) with 2mb Cache, 64 mb RAM, 1 2 gig
and 1 4 gig disk, a 1.44 floppy (Although linux seems to think that
it's a 2.88, unless I specify floppy=0,4,cmos), an ISP 1020
controller, #9 S3 764X based card and a DE-500 network card. I had to
move the network card down to the second PCI bus in order to make
Linux recognise it. If I've missed an important detail, please let me
know. Thanks for your time.

Roman
