Linus Torvalds wrote:

> Umm.. You don't need a floppy: you can create a msdos filesystem on a disk

I had to do the usual long to int fixes to get get mkdosfs to work on the
Alpha. It was trivial to do but if anyone really needs a copy I can put it
on my FTP server once my T1 line gets fixed. Hmmm... this is so simple
maybe I should use it to learn how to build an RPM pacakge.

> Without knowing anything about the UDB, I suspect the problem is that it has
> no "fallback" bootup method. So if something goes wrong with the flash,
> you're screwed. With the development motherboards (at least the Cabriolet),
> you can set a jumper to boot the thing into a known state (a _awkward_ known
> state, but still..).

The UDB and Multia are alleged to have the recovery mode jumper you describe
(they certainly have plenty of jumpers), but since there seems to be no
documentation available I don't know which it is, or what any of the others
do.

Could somebody *please* prove me wrong and tell me where to get hardware docs?
DEC seems to have taken an admirable position in supporting the development of
free software, even to the extent of offering a machine specifically intended
to run Linux (the UDB), so I don't understand why they are so reluctant to
make even the most rudimentary hardware docs on that machine available. Yes,
I know it is very similar to a Noname, but not so similar that no docs are
required to explain the differences. It is very frustrating. I am beginning
to think that buying these machines was a mistake and that I should have
bought Noname boards instead. (But the Multia/UDB packaging is so awesome...)

I've been thinking about hacking some code into fmu to let me dump the
flash contents for safekeeping so I can experiment with flashing Milo, but
until I know where the recovery jumper is and verify that it works I don't
want to mess with the flash.

Cheers,
Eric
