Vareck Bostrom <bostrov@teleport.com> writes about the UDB:
> I'd been told I should expect "about pentium-66 or -75" level perfomance,
> and am getting closer to pentium-120 or -133 performance.

My Pentium Amateur 133 MHz systems is noticably faster for non-floating-point
applications than my 233 MHz Multia, so I would guess that a 166 MHz Multia
would be a little slower than a 90 MHz Pentium Amateur. Of course, a good
bit of the performance difference is that GCC generates better code for
the Pentium than for the Alpha. As the Alpha GCC matures I'm sure the code
quality will improve substantially.

> Do the cores vary between the 21064 and 21066/21068?

The EV4 and LCA4 are nearly identical, as is the case for the EV45 and LCA45.

The -45 cores have some improvements over the -4 cores. In particular, the
floating point divider has a shorter latency and produces the IEEE inexact
result exception. The 21064A has larger caches than the 21064, but the
21066A cache didn't grow.

> and gather that the "EV45 and EV56" are optical shrinks from the
> EV4 and EV5

They are not optical shrinks. There was apparently an optical shrink
from 0.75 u to 0.68 u for the 21064, but AFAIK it was still called an
EV4.

> Are there different revisions of the Alpha instruction set (similar to
> MIPS-I, MIPS-II, MIPS-III and MIPS-IV)? If there are, what are they?

I vaguely remember reading a reference to a particular version number of
the architecture in some edition of the Alpha Architecture Handbook or the
Alpha Architecture Reference Manual, but naturally I can't track it down
right now.

Several changes seem to have been made to the architecture will little
fanfare between the publication of the first and second editions of the
Alpha Architecture Reference Manual. For instance, the WMB instruction was
added in 2nd edition, but the note at the bottom of page C1 of 1st edition
which specifically reserved not just the 18.4400 code point later used for
WMB but also the 18.4800 and 18.4c00 for other weaker memory barriers. Does
this mean that new implementations are no longer required to treat 18.4800
and 18.4c00 as memory barriers?

The EV56 adds some new instructions including byte store. About d*%$ time,
if you ask me. I've always been amused by the claim in the Alpha Architecture
Reference Manual that one of the reasons for the lack of byte store is that
"The Alpha AXP approach can make it easier to pipeline multiple byte
operations." In other words, by omitting a hardware capability, the programmer
might be encouraged to come up with clever algorithms that handle multiple
bytes at once. Apparently the authors feel that if a byte store instruction
were available, no one would bother to spend time figuring out how to speed
things up. In reality what more frequently happens is that a lazy programmer
(or compiler) who doesn't have a store byte instruction will end up with
really bad sequences of instructions. Sure, writing better code would be the
best solution, but having a byte store instruction would at least be somewhat
of an improvement. I guess they finally figured it out. Of course, it's
not going to help those of us with Multias, UDBs, and Nonames.

> What Operating Systems can the UDB run? I know NT 4.0 runs on it (running

heard of anyone trying OpenVMS, and can't imagine why anyone would want to.
