Jay> Now, as I remember, you're booting via SRM; I don't believe the
Jay> kernel actually exits back to the SRM console correctly, and
Jay> thus you're left with an (essentially) dead machine upon

Jay> I don't know how hard this would be to fix; anyone else?

The kernel does exist back into the SRM properly, but SRM is a little
picky w.r.t. what state it wants the machine in when this happens. In
particular, it (a) does not want to reset the graphics card and (b)
expects the virtual page table to be setup the way it did when the
system powered up. The Linux loader (MILO or aboot) moves the virtual
page table so (b) isn't satisified. This typically results in
machine-checks when trying to do anything in the SRM console after a
reboot. Also, since the Linux console uses hardware scrolling, the
SRM often is unable to print any messages on the screen. This
sometimes can be worked around by switching to a pristine (unused)
virtual console right before the system halts. This info should be
true at least for Noname and Cabriolet systems. There are differences
among the various SRM implementations, so your mileage may vary.

These problems are not unsurmountable, but AFAIK, nobody has bothered
so far. But I'm sure there are hackers that could be bribed into
fixing this---a bribe will be necessary since this isn't exactly a fun
thing to do (but I'd be _very_ happy if somebody proved me wrong on
this point... :-)
