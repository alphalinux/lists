...this is just a follow-up to Harald's report of a problem
he was having with g77-0.5.17 (Craftworks' version, I believe).

At first I thought the only "problem" was that he hadn't
specified -finit-local-zero, as the crash was due to trying
to print an uninitialized floating-point value, that happened
to be a denormal.

Then I noticed he mentioned this in the header, and also pointed
out a problem in earlier printouts, in that the value printed
for a SQRT(AMAX1(...)) result could be right or wrong, depending
on what -O level was used.

I've investigated that problem, using the g77/Alpha development
version I've been working on, and it appears to be a gcc back
end bug. The cmpt*/fcmov* pairs just prior to calling sqrt
are incorrectly coded, causing incorrect results from the
AMAX1 intrinsic. (Fixing these pairs in the generated .s
file and assembling that file produces a program that works
properly.)

I've reported this problem to two of the gcc maintainers, and it
turns out they might already have a fix, if not, it seems likely
to be easy to fix.

In any case, I just wanted people to know that g77/Alpha work is
indeed proceeding. Aside from this outstanding problem, which I
expect will be resolved shortly, I have no known outstanding bugs
in the development version of g77/Alpha that I can reproduce.
(There are two bug cases reported on g77/Alpha using an OS other
than Linux, but I cannot reproduce them -- I might look into these
further.)

Of course, testing is being done by a very small number of people,
and it's as yet unclear how or when the development version will
be made available for public use. Further, I haven't investigated
any of the performance issues yet. (I did notice that the buggy
cmpt*/fcmov* pairs could each be reduced to one instruction; maybe
I'll get into writing peephole or other optimizations for gcc's
Alpha support down the road.)

At least, thanks to Craftworks, Carrera, and DCG, for lending me
this Linux/Alpha workstation, there has finally been real progress
getting g77 to work on this machine (and, perhaps, resulting in a
fix to gcc that could make C code using floating-point work as well).

tq vm, (burley)
