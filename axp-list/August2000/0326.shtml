Alan Keane wrote:
>
> <disclaimer>
> It's been a few months since I messed with BSD style disk labels under
> fdisk, so this info may be incorrect...
> </disclaimer>

It is.

> OK, looking at the partition table you have a partition "c:" which
> starts on cylinder 1, this causes problems as the lowest cylinder that
> should be used should be 2, despite the documentation.

No. If you have "unused" c partition, for a compatibility with DU
partitioning, it should cover the whole disk or there is no point
in creating it at all. Partitions with data cannot start on cylinder
1 _if_ you plan to a write a boot block on this disk. You simply need
some space for it.

Linux does not require this "unused" partition to be present but
does not have any problems if it is there. Regardless of other problems
a partition table as shown in the original message is valid.

> The partition structure once we've finished
> partitioning looks like this:-
>
> 3 partitions:
>
> # size offset fstype [fsize bsize cpg]
> a: 8578710 16065 ext2 # (cyl. 2-535)
> b: 273105 8594775 swap # (cyl. 536-552)
> c: 8883945 0 unused 0 0 # (cyl. 1-553)

For many reasons, which were explained a number of times on this list,
it is _extremely_ unwise to dump the whole Linux into one big partition.
In particular do not be surprised if you will find out one day that
you have troubles with booting. If you think that you would _really_
need a big continuous disk areas, and you do not want to play with
partitions resizing, then split your disk between a root partition
(300-350 MB should be plenty) and the rest.

>>From a point of view of a long range exploatation, in nearly all cases,
if you cannot boot only a minimal system which mounts only a root
partition and NONE of your "work data" then your system is broken as
installed.

It seems to me that current versions of an installer from Red Hat
do reasonably decent job in partition a "virgin" disk if you are
not sure what you want (with SRM as well).
