> Yeah, but sneaking in weird stuff is so much fun. I've had to change the
> source of several different packages to get them to compile. I've always ended
> up re-tar.gz'ing (<- now that is an interesting word) them and having to start
> everything over from scratch. It would be nice to selectively move from
> compiling, to installing, and finally to wrapping a binary package one step at
> a time.

Yes. This would be a vaulable extention indeed. Especially when you're
packaging
something that takes a long time to compile like for example gcc.

 
> > It is possible to create and use another spec file which will be doing
> > mostly nothing until an installation phase. Like one for Netscape from
> > DU. :-) Techically this will be a different rpm archive then the one you
> > tried to build in the first place. Either redefine prep to do mostly
> > nothing or prepare "sources" from your current compilation tree.
> > Do not unpack on the top what you compiled or you will be starting
> > from scratch again. The first action in a normal prep is 'rm -rf ...'.

That's one way of doing it though I would like to keep things as simple
as possible. Having a couple differnet versions of spec files that have
different purposes is kind of a pain.

> What is with that 'rm -rf' thing anyway? I can understand the need to delete
> old binary trees, but why not do the same as the removing of the /var/tmp
> build roots. Leave it as a command after the prep. Basically without that you
> could simply comment out the 'rm -rf' and not have to rebuild all the object
> files. Also, can I changed the default behavior of the %prep macro?

Indeed. Passing a command flag that will change the default behavior of
%prep
would be ideal. That or you 'could' define your own macro to use instead
of %prep
that allowed such things in /usr/lib/rpm/macros.
