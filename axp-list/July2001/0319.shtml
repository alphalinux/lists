The Kernel does have code which does:
        print out unaligned access errors
        does the 'right' thing (albeit a bit slowly)

In the event that a ldwu instruction is accessing a misaligned
address. The 'gotcha':
- Unless the code in the application is explicitly using the
  ldwu instructions, the code executed will read the content
  correctly from a misaligned address. The only way to get
  those instructions is to compile with -mcpu=ev56 (or later)

If your machine supports ev56 instructions, you _really_ want
to use them. Your code size will shrink (good for performance)
and your code will automatically speed up - I've seen speedups
ranging from 10%-20% by just using the new instructions.

I hope this helps,

        Rick

I have noticed some problem with accessing unaligned data.

I have an application that has a bunch of accesses to shorts in a byte
array. Not seeing any more alignment errors from the kernel I assumed
that all was ok due to the fact that althought it is a byte array it
seemed to be short aligned...

Well this turns out to be wrong. forcing alignment and doing memcpy's
made my problems go away.

I was under the impression that the:
1) OS/Palcode handled unaligned accessed as a fault and although there
is a performance
   penalty that is all.
2) The kernel generated an error message for unaligned accesses.
