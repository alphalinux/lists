>Linux does not know how to mount an NTFS partition.

Red Hat Linux does not come with ntfs support because it is not
stable in any sense, however the Linux kernel does have NTFS
support both read-only which many people use with minumal
problems I believe, and write support also, which most
definately will hose your NTFS partitions. You would need to
build the ntfs module or a new kernel but I do not recommend it
for day to day use. It is much better to have an additional
drive or partition formatted with a filesystem which is common to
both OS's such as FAT or FAT32 perhaps and use that as a go
between. Better yet do it between 2 networked machines.

>I have heard that there are file systems that will allow you to
>read, but not write NTFS, but they are still experimental.

Both read and write, and yes, very experimental.

>DEC Unix ( or OSF/1 or TRU64 ) also do not know about NTFS.

They know FAT though no?

>I believe that you should have used FDISK, because any drive that was used
>with DEC UNIX should have a BSD disklabel. FDISK can write either BSD or M$
>type disklabels ( partition tables ), but Disk Druid only knows about M$
>type.

Not sure what FS DEC UNIX uses natively, but Linux supports
numerous filesystems itself, and you should check the exact
filesystems and partition table types in the kernel source to be
certain.

>Disk Druid MAY have screwed your BSD disklabel to the point were it will
>take some SERIOUS hacking to get your data.

I believe at least in the past that Solaris partitions wuld get
nailed as swap space, but I think that might be fixed in rawhide.
Dunno if that applies to BSD too or not though.
