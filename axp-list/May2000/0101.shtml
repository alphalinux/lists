Mohan wrote:
>
> I had an earlier post regarding my dual boot problems,
....
> CONFIGURATION:
> -------------
> Alphaserver DS20 dual cpu. 4 Harddisks and they are:
>
> [Tru64 Unix BOOT DISK] DKA0 RZ2DA-LA N1H1
> [Tru64 Unix] DKA100 RZ2DA-LA N1H1
> [CDROM] DKA500 RRD47 1206
> [LINUX] DKB0 RZ2DD-LS 0306
> [Tru64 Unix] DKB100 RZ1CB-CS 0844
....

> Here is the screen shot.
> ------------------------
> P00>>>boot
> (boot dkb0.0.0.2001.1 -file 2/boot/vmlinux-2.2.14smp-001 -flags
> root=/dev/sda2 console=ttyS0)
         ^^^^^

In other words you are telling your Linux that your root file
system in on the second partition of the first SCSI disk and,
accordingly to what you wrote above, this is really not the case.

.....
> [MS-DOS FS Rel. 12,FAT 0,check=n,conv=b,uid=0,gid=0,umask=022,bmap]
> [me=0x0,cs=0,#f=0,fs=0,fl=0,ds=0,de=0,data=0,se=0,ts=0,ls=0,rc=0,fc=4294967295]
> Transaction block size = 512
> UMSDOS: msdos_read_super failed, mount aborted.
> [MS-DOS FS Rel. 12,FAT 0,check=n,conv=b,uid=0,gid=0,umask=022,bmap]
> [me=0x0,cs=0,#f=0,fs=0,fl=0,ds=0,de=0,data=0,se=0,ts=0,ls=0,rc=0,fc=4294967295]

Here Linux is trying very hard to find a file system to mount from
a location you told it by checking various, known to it, possibilities
(MS-DOS, at last).

> Transaction block size = 512
> Kernel panic: VFS: Unable to mount root fs on 08:02
                                  ^^^^^^^^^^^^^^^^
     And, predictably, it fails as there is nothing there of a sort
     you promissed.

If you physically remove two first disks then device assignments change
and your root file system indeed shifts to /dev/sda2 (I assume that this
_is_ sda2 as you write that you succeed there) and your boot command
becomes a correct one. From what you write it sounds that in a "normal"
configuration you should use 'root=/dev/sdc2'.

The fact that Linux checks partitions on Tru64 disks is irrelevant and
harmless. Remember that you do not have _any_ OS until you booted
it. It is just a computer program after all.
