Brian Rerecich wrote:
>
> > So where does 'true-parity' DIMMS fall in terms of bit usage.
>
> OK, so heres how I understand it....ECC needs 1 bits correction for 8 bits
> data. thus 8 bits for 64 bits data. Parity was used to know if there were
> an even or odd number of 1's in the data sequence. So, if I had 64 bits
> data, I would need a 65 bit wid ememory. (64+1). If i had 32 bits data, I
> would need 1 bit parity for 33 bit memory (32+1). Regular RAM of course has
> none of this and thus 32 bits data = 32 bit wide memory. Hope this answers
> the question....

        In the old world of SIMM technology, true-parity SIMMS *were* 36 bits
wide, or 1 parity bit for every 8 data bits (or least this is how they
were marketed). My older Alpha XL300 uses '36bit true-parity memory'
according to its manual. It was my understanding that SIMM ECC memory
used 4 parity bits for every 8 data bits, therefore making it more
appropriate for "mission-critical" applications.

        Can anyone confirm/deny this, as it might clear up some confusion about
what RAM is required for their Alphas (both new and old)?

        I am sorry to be a bit off-topic here, but this is apparently a source
of confusion for many people. Also, I am not trying to start a 'war' by
any means. Just a slight digression into the uppermost layer of memory
technology terminology.

        Is it possible that the amount of parity bits required in a mem config
is not a purely linear relationship with the amount of data bits?

        Chris
