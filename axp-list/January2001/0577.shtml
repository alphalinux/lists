ppetrakis@alphalinux.org wrote:
> So there would need to be full DEC_THREADS support. which means a fully
> POSIX compliant thread model of which the linux kernel does not have.

Much/most of pthreads should be handled in user space anyway. Any
reasonable implementation needs some kernel support, of which Linux
provides a minimum with clone().

> http://kt.linuxcare.com/kernel-traffic/kt20000911_84.epl

This thread starts focusing on thread create latencies, which is kind of
pointless. Good threaded applications create a bounded number of
threads, for which the total latency is slight. Context switch times
are important. Synchronization primitives are very important, which is
why all performance-critical threaded apps seem to supply their own
primitives anyway, based on test-and-set etc.

Tru64, I believe, supports a m:n thread model which requires some extra
kernel support. There is a potential performance benefit which a cost
of greatly increased complexity.

Linux takes a minimalist approach with a 1:1 model. Simple and
reasonably fast. Not a bad compromise IMO. If it were a serious
problem you can bet the commercial vendors would be complaining.

As one poster says, pthreads are really inadequate for complex
applications anyway, so most end up with their own thread libraries,
rendering pthreads as somewhat of a failure. I concur with this.
