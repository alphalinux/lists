On Wed, 17 Jan 2001, Paul Allen wrote:

: The ext2 filesystem is designed to automatically keep the blocks of
: a file near each other. It will never become fragmented the way a
: Windows disk will unless you defeat the mechanism by setting minfree
: to zero and running the disk near full. A "fragment" is a subset of
: a filesystem block that is allocated at the very end of a file. The
: fragmentation percentage reported by fsck is talking about these ends
: of files and does not indicate a performance problem.
:

While all this is so very true, SteveCraft probably meant the
"non-contiguous files %" that fsck reports. That number actually is the
(internal) fragmentation, i.e. the percentage of files that have at least
one jump in their sequence of blocks. As this number says nothing about
how large these jumps are, it doesn't really tell how severe fragmentation
is. Ext2fs tries to avoid jumps, and tries to keep them small if
unavoidable.

This number therefore does not mean the same as the equivalent number a
certain less refined OS from Redmond reports. In practice you shouldn't
worry unless the percentage gets larger than 10 - 15%. You won't notice
fragmentation effects when loading binaries (page by page) to memory in a
multitasking, multiuser environment. You may notice the effects if you
sequentially read large files, though.

Note that, when using the default 1k blocks, having 8192 blocks per group
means that every file larger than 8 MB extends beyond one block group and
will be non-contiguous according to fsck. At home, I have a backup
partition containing large tar.gz's only. It's 100% fragmented ;-)
