On Tue, Feb 06, 2001 at 02:58:01PM +0100, Georg Wittig wrote:
> The following program doesn't compile with gcc-2.96:
>
> ---------------------------------
> #include <stdarg.h>
> #include <varargs.h>
.....

Eh? Why it should? This is not a valid C! You are trying to mix
a standard header <stdarg.h> with an obsolete <varargs.h> which is
provided only for backward compatibility for old sources.

In other words you attempt to use at the same time two different,
incompatible, implementations of variable arguments functions and what
you can possibly get out of there is clearly undefined. It may even
work if you are (un)lucky enough. OTOH you cannot predict that it will
not launch 'nethack' if it will compile. Standard documents explicitely
say that "using #undef on va_start and va_args macros leads to
_undefined behaviour_". They are indeed numb on attempts of a
redefinition but this is effectively an "#undef" as well. What is even
worse results _may_ be different if you will switch an order of these
two "#include" lines above.

It is quite possible that some compilers may swallow that but I can
only be happy for the one which will barf on this obvious programmer
error. If you want a code which is acceptable to pre-Standard
compilers then use some configuration means and provide two different
implementations compiled conditionally.
