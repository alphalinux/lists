On Fri, 24 Mar 2000, Ivan Kokshaysky wrote:

> Anyway, it won't detect spurious interrupts from PIC 2 (IRQ 15).

 Right.

> It _is_ configured to IRR at least on lx164 and sx164, so that
> code is wrong. And a generic kernel code seems to be relying
> on the status read returning IRR - see isa_no_iack_sc_device_interrupt().

 Hmm, I don't have 2.3.99 handy and 2.3.42 have no such function. But why
would generic code handle 8259A -- the chip is not generic at all. Oh
well...

 I checked docs meanwhile and IRR is really the default state, BTW.

> So I think the patch I posted is correct because
> - spurious interrupts should be harmless for current kernels

 Correct.

> - duplicating i386 code for spurious interrupt detection
> would buy nothing except the warning about broken hardware

 I am not the one to decide whether it's useful or not. You asked for an
explanation for the code, so I provided it.

 Warnings of broken hardware can prove useful as the recent BP6 (that's a
dual-Celeron board) case showed. Enabling of APIC error interrupts let us
know the board works under marginal conditions, probably due to a design
error. We can now safely assume any strange behaviour of such systems is
not a Linux's fault.

 On the other hand current Alpha hardware is of better quality than PC one
and the situation isn't going to change any soon, I suppose (i.e. prices
will remain high).

> - it would be a pain checking such code on an each possible
> platform/firmware/kernel combination

 Why would it?
