>
> I think this may be a pure userspace problem... *If* you use the rtc driver
> and hwclock, all will be fine. hwclock mis-detects in not-up-to-date versio=

> the epoche when accessing the RTC chip on older machines via inb()/outb().

The current versions of 'hwclock' have this interesting property that
with -A flags on UX they will set you back by twenty years. OTOH
Red Hat startup scripts, at least from 6.0 and 6.1 distributions try
to second-guess you and upon detection of a milo (UX has to boot with
milo) add this -A. The net effect is that does not matter what you
put in /etc/sysconfig/clock your time will wrong until you will edit
startup scripts (you cannot delay that to local stuff as a number of
time stamps will be already wrong).

> This should be fixed in the most recent versions.

This will be not fixed as the hwclock author insists that this is
exactly like it should be. If you added -A flag where your clock is not
really ARC clock (technically this is correct as UX is using AlphaBIOS
and not ARC and a clock behaviour is different so this is not ARC clock)
then this means that you really **wanted** some funny offset. Why back
by 20 years - beats me but this is as good any other one. :-)

I mentioned that on this list a few months ago.
