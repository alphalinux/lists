According to the gods of SMTP, Peter Petrakis wrote:

> I started this thread on the AlphaNT list and I was just wondering if
> there are any x86 assembly Gurus around who can verify if this is
> possible or not.

Well, I'm not an x86 assembly guru, but I did migrate to Linux from OS/2, and
this is "common knowledge" in that world. :)

> I saw the coolest thing in Assembly class today. My prof. explained this
> method though not the specifics of tricking the Intel processor into
> thinking it has rebooted so it could change modes or something.

He was probably talking about the trick that one of the OS/2 developers at IBM
came up with. Originally, when OS/2 was in early development, it was being
aimed at the 286. The 286 has a protected mode (i.e. mode with hardware memory
management enabled) but I think during the boot process OS/2 needed to be able
to hot-switch between real and protected modes.

The 286 did not support this (once you went into protected mode, there was no
going back) so somebody at IBM came up with a way to fool the CPU into thinking
it had rebooted so it could be switched back into real mode. This was
described as "turning the engine off and then back on again at 55 mph." They
actually patented this method.

> Intel claims that you have to reboot the computer to disable/enable
> this privacy thingee. Soooo given the facts I previously stated it is
> theoritcly possible to trick the processor that it has rebooted and let
> you enable/disable the privacy thingee.

The 386 and later processors support hot-switching between real and protected
modes, so the particular hack I describe above was rendered obsolete by later
Intel processors. I.e. it was specific to the 286, so the fact that it worked
once doesn't mean that it works on current Intel processors. In fact, Intel
may have "fixed" this particular "feature" to prevent it being exploited by
e.g. virii.

Of course, I could be wrong. I'm not Intel expert, either. If you can develop
a work-around once, you can do it again. :)

> He also told us that all of the Intel chips including todays still have the
> instruction code in 'em back from the 8086 for backwards compatibility
> (Why?) and that you could get to those instructions if you wanted to.
> Then a light bulb in my head went off :)

Well, anything that's called an "x86" CPU has all the original 8086
instructions. I would imagine that this is only at a functionality level:
i.e. the Pentium III supports the 8086 instructions, but does not actually
include any of the actual silicon guts of the 8086. So, it's like the POSIX
API: every UNIX supports fork(), but each implements it in its own way. The
exploits that work vs. one UNIX's fork() aren't necessarily going to work
against another's. Every x86 supports the 8086 instructions, but the exploit
used to do the hot-reset of the 286 may not work on later CPUs.

My US$0.02...
