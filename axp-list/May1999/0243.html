On Tue, 11 May 1999, James Bremer wrote:

> Dunno - mine is running an Exabyte-8200 fairly flawlessly since compile
> time. It did have to get a different mt (mt-st, from an old slackware
> build) to access it properly (mt for SCSI seems to be broken in 5.0-5.2).
> What kind of "flakey behavior" are you seeing? Do ALL the commands return
> an error (rewind, erase, format, etc.), or just randomly when doing a tar?

I have a Seagate TapeStor 4/8GB SCSI 2 type tape.

Detected scsi tape st0 at scsi0, channel 0, id 1, lun 0
  Vendor: SEAGATE Model: ST39140N Rev: 1281
  Type: Direct-Access ANSI SCSI revision: 02

With the 'st' device compiled as a module, it simply does not work. I
boot with the tape in the drive, and it takes a *long* time for it to
finish winding, like 3-5 minutes. It is winding during boot, when
redhat's init scripts try to force the 'st' module to be loaded. That's
when I get the "cannot get major 9" error.

If I compile it into the kernel monolithically, it mostly works. I can do
a 'tar', though I'm not using it as such, and have been using mostly
'amlabel' until I can get amanda raw access to my disk partitions.

I think I may have used 'mt' to rewind the tape; I suspect this is where I
started to get the I/O errors. Once an I/O error occurs, I have no choice
but to reboot the system - that's why I wanted to try a modular approach.
But I now see that that doesn't appear to work at all.

I will get the SRPM for mt and the latest version of mt, and try my hand
at compiling/upgrading an RPM package. In the meantime, I'll just not use
it.
