When/if DEC releases their compilers for Linux, that'll do away with the issues
I suppose. Well, maybe not, this raises another question though. If I buy the
new DEC Linux compilers and statically link libs, I'll then be able to move the

Craig P Prescott wrote:

> Matti Aarnio wrote:
> >
> > Duccio Medini <Duccio.Medini@PG.infn.it> asked:
> > > The difference that I experience between fortran code compiled
> > > under DU - and - under Linux with g77 AND libffm or libcpml is still
> > > remarkable.
> > > I saw at alphalinux a detailed set of instructions about running DU
> > > Netscape under Linux. Is there anything similar about running DEC-f77
> > > compiler under Linux (a compiler should be, in some way, a binary like
> > > the others)?
> >
> > For various reasons running the DEC Tru64 *compiler* at Linux is
> > likely doomed to fail. Linux doesn't supply license management
> > calls for example, which seem to be needed.
> >
> > However, running compiler at DEC Tru64, producing *static*
> > binary, and then running that binary at Linux is very likely
> > to work, and even (barely) within license rules.
> >
>
> Running static DU binaries under AlphaLinux certainly works...
>
> ... but is still a violation of the license agreement without a DU
> license. In a static binary, you are still copying over
> encumbered objects (in your executable). Below are two snippets
> from postings Jim Paradis and Jay Estabrook made to the list last
> summer. If the status of running DU static binaries on an unlicensed
> box has changed... well, I'd be very interested to know it!
>
> Hmmmm... is the licensing of Tru64 the same as it was with DU?
>
> * <begin snippet 1>---------------------------------------------
> *
> * From: Jim Paradis <paradis@linux01.hlo.dec.com>
> * Subject: Re: DU Licensing ramifications
> * To: axp-list@redhat.com
> * Date: Mon, 13 Jul 1998 15:21:15 -0400 (EDT)
> *
> * > 1. If I link a program statically against the DU libraries,
> * > am I allowed to run it on an unlicensed CPU? Probably not,
> * > I'd expect, though it seems that that may not be stopping
> * > some Beowulf users that own DEC compilers, unless they all
> * > have DU site licenses.
> *
> * Correct; legally you may *not* run statically-linked DU executables on
> * a non-licensed system. While it's highly unlikely that anyone in a
> * position to turn you in to the Software Police will *notice* that you're
> * doing so, it's still a technical violation of your Digital licensing
> * agreement.
> *
> * <end snippet 1>---------------------------------------------
>
> # <begin snippet 2>---------------------------------------------
> #
> # To: axp-list@redhat.com
> # From: Jay.Estabrook@digital.com
> # Subject: Re: DU Licensing ramifications
> # Date: Mon, 13 Jul 1998 15:33:38 +0500
> #
> # >>> Bill Rugolsky Jr. said:
> # > Two questions:
> # >
> # > 1. If I link a program statically against the DU libraries,
> # > am I allowed to run it on an unlicensed CPU? Probably not,
> # > I'd expect, though it seems that that may not be stopping
> # > some Beowulf users that own DEC compilers, unless they all
> # > have DU site licenses.
> #
> # No, you are NOT allowed to run it on an unlicensed CPU; a statically linked
> # binary will still have C library routines bound into it, and it is those
> # routines which are encumbered.
> #
> # <end snippet 2>---------------------------------------------
>
> > (Copying shared libraries to a machine without DEC Tru64 license
> > isn't strictly speaking allowed...)
