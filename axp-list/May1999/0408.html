>>> Rich Payne said:
>
> I think it is set to VMS already, so I don't think that's the problem.
> Though I will check it out when I get a chance.

VMS PALcode and OSF PALcode are *NOT* the same. You must be running with
os_type set to UNIX...

> The other thing I noticed (and could find no info about in the SRM
> reference manual) was that the SRM console prompt was usually P00>>>
> but sometimes it would come up as P01>>> and the interesting part was that
> every boot from P01>>> would fail. aboot would report Switching to OSF/1
> PALcode version 0.0 (instead of 1.44) and the initialization of the SCSI
> controller would fail every time with a time out. Is this something simple
> I've overlooked?

P00>>> indicates that the message is from CPU #0, so P01>>> would be CPU #1,
which would cause problems since the boot CPU ID must be 0 for the kernel,
IIRC. Also, interrupts are all routed to CPU #0, only timer ticks come
through on the others... :-\
