> Tru64 - or whatever OSF1 is called this week - can store passwords
> in several formats, as debian and, I assume, redhat can. Plain passwd file
> will probably be compatible; they can be stored in dbm or dbz or similar
> databases, and these are more unlikely to be directly readable from Linux.
> You could try converting from dbm to plain file in the DU box, and then
> using the file in Linux (or converting that again to db* from Linux).

  The /etc/passwd.{pag,dir} is trivial db version of the /etc/passwd
text file -- each entry is there twice: once keyed with character string
(username), and once with UID. (getpwnam(), getpwuid())

The PASSWORD encryption, however, might be a difficult case.
Tru64 has at least two ways to encrypt passwords:
        - classical UNIX crypt()
        - SecureWare bigcrypt()
I don't recall having seen BSD popularized md5crypt() in the repertoire.

Passwords encrypted with the classical one are trivially copyable over,
but passwords encrypted with SecureWare are definitely incompatible.

In SECURED (C2) mode, the system uses sort of 'shadow' password system,
and passwords are stored under "Trusted Computing Base" subsystem; under
"/tcb/files/" directory...

So, if you have passwords stored into /etc/passwd *file* for everybody
to copy, you can copy that file very likely straight over to Linux.
However if you have secured environment in running, you can't use those
passwords. (This is based on assumptions about system not using
bigcrypt() without C2 being enabled... I could be wrong, I am not
fulltime Tru64 ISSO -- Security Officer.)
