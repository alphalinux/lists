        I think the real issue with VMS and Tru64 support is a matter of
the operating systems don't support most of the inexpensive hardware or
much hardware without the compaq firmware.

On Sat, Apr 07, 2001 at 01:18:59PM -0400, Speedracer wrote:
>
> I'd thought I'd share some comments, mostly in regards to the first system
> you set out as I think this is the most viable option.
>
> On Fri, 6 Apr 2001 KirkE@paccessglobal.com wrote:
>
> > Enthusiast/Developer/Student (emphasis on Linux):
> > Small footprint ATX deskside chassis
> > space for one internal 3.5" 80GB 7200 RPM ATA-100 drive, 1 combo MKE
> > DVD-RAM/DVD-ROM/CD-ROM. Optional support could be provided for a
> > standard floppy or an MKE LS-120 ATA-33 "floptical" drive in a
> > vertical position adjacent to the CD/DVD drive. 300 watt ATX power
> > supply ATX compliant motherboard
>
> I think it would be important to make the Board 100% standard ATX
> complient so that any ATX case can be used. Make the board available to
> system resellers! This will allow whoever produces the board to have a
> higher production run because the same board would be used in internally
> produced machines as well as sold to vendors.
>
> Perhaps a smaller power supply requirement? 250w? I'd like to see future
> boards not quite so power thirsty. Just a thought :) VMS compat. isn't
> as critical in this product IMHO. If some savings can be had while by
> sacrificing VMS support that may be ok. I'd say the next step up in price
> VMS becomes more important.......just a thought. Although if VMS support
> can be kept then "hurray!"
>
> > 700 MHz EV68 or EV69 CPU/4MB DDR HSTL cache in either
> > non-upgradeable "soldered on" or upgradeable socketted configuration
> > AMD IG-4 761 chipset (VIA chipset may become a viable option also)
> > 256MB 184 pin PC266 DDR SDRAM, 4 sockets (2GB max using 512MB
> > unbuffered ECC modules)
>
> The important thing is to use memory that is pretty standard and cheap.
> With ALPHA's RISC design memory is probably more important than clock
> speed for many applications. The ability to expand memory cheaply
> gives these MB's a chance to live a LONG life. This is attractive for
> educational buyers who generally have a longer product replacement cycle
> than many corporations (5-7 years opposed to 3-5)
>
> > Nvidia GeForce 3 MPU, AGP 4x (integrated as a soldered on AGP
> > device, no AGP slot is provided) with 32MB DDR
>
> I'd substitute "Any GOOD 3d AGP chipset that has at least 24mb of on board
> video ram and is currently supported by true64 and Linux"
>
> > Single port Intel 82559 Fast Ethernet
>
> Again substitute "Any on board 10/100 ethernet but it MUST be supported by
> True 64 and Linux"
>
> > ALi Southbridge I/O chip
> > ATA-100 support (via ALi southbridge, treated as a SCSI controller
> > by the SRM console)
> > Integrated USB, IrDA wireless, parallel, dual 9 pin serial, PS/2
> > keyboard and mouse ports (via ALi southbridge)
> > Integrated PC97 accelerated audio (also via ALi southbridge) for 16
> > bit, 48KHz stereo audio
> > Integrated dual port TI IEEE 1394c "Firewire" for external devices
> > 1 64 bit PCI slot supporting 66MHz 3.3 volt operation, isolated from
> > internal PCI bus (where the Firewire, ATA-100 and Fast Ethernet stem).
>
> To this I would add a single 32 bit PCI slot also (maybe 2) Just so there
> is SOME room for expansion in the future. ATA 100 support is important,
> and a ATA chipset whose BIOS can be upgraded later (ATA 133?) on might
> also be useful (we are all dreaming here, so cut me some slack on this one :)
>
> > There are of course several technical issues to overcome: Firmware console,
> > SROM and operating system/SRM drivers would have to be written more or less
> > from scratch, to name just a few items. The system described here does bear
>
> Couldn't we use LINUX in flash the way API is now? This may/would save
> development time, and I'm quite sure you could teach linux to load Tru 64
> unix. Since this is aimed at the lower end keeping the development costs
> down is important. Also the ability to be able to correctly identify
> future hardware that is plugged into the thing should be easier. (Grab the
> source recompile linux firmware xxx and load it up) The ability to use a
> more "open sourced" firmware loader has been discussed here before. I
> think it may have a good fit in the praticulair application.
>
> Also DEC was pretty darn smart with the fail safe boot loader available on
> many Alpha motherboards. I would say with the market that this is aimed
> at (hobbiest and people that like to poke around) any motherboard produced
> should have this option. MANY an intel based board has had to be sent
> back because a flash bios upgrade failed. So far EVERY alpha I own (6 of
> them) has been able to recover from a failed BIOS updates.
