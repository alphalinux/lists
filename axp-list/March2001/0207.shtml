On Thu, Mar 15, 2001 at 01:36:37AM -0500, Jeff Sturm wrote:
> Alan Young wrote:
> >
> > I believe that's due to Linux not supporting certain kernel functions
> > that the Tru64 kernel has and the threads library is looking for them.
>
> Right. The point I didn't make is that Linus really only implemented
> the syscalls he needed to get a self-hosting system, and Linux/OSF
> syscalls have probably diverged since anyway.

Not really, AFAIK. You can look into kernel sources and see for
yourself. Over time more of syscalls from Tru64 got their emulation in
Linux kernel but it is true that not all of them. In some situations it
may be not even clear what their interpretation should be. Luckily a
number of these can often be simply ignored. If you will see in your
logs "<sc, ..." with a bunch of numbers this is a recorded such ignored
system call. But indeed not every Tru64 binary will run under Linux;
generally you will miss required libraries and not system calls.

> You can bet that Linux
> also has syscalls that Tru64 does not.

So? Missing calls the other way would be important only if you would
try to run Linux binaries under Tru64. I never heard about such
attempts. First Tru64 does not understand an ELF format of "native"
Linux binaries, as it is using COFF instead, so further questions are
academic. Nearly always if you are interested you can simply recompile
an application in question on Tru64 and that is definitely much simpler.

> I don't use Tru64, but I'd wager that its thread APIs are based on Mach
> tasks, which could be very difficult to emulate (feel free to correct me
> here).

Some Tru64 programs which, at least nominally, use threads can be
satisified with libpthread.so and libpthreads.so which are simply links
to /usr/shlib/libc.so as provided by Tru64-compat package. Quite
possible not every one but at least some can be "cheated" that way.
