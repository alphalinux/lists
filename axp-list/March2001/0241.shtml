On Fri, Mar 16, 2001 at 04:59:58PM -0800, Paul Bunn wrote:
> Sorry, but you've been given bum info.
> I have 16 years experience of dealing with SCSI, coupled with writing both OS
> drivers and device firmware. In all that time I've never come across a device
> that has failed a FORMAT UNIT command
....

> Most devices do support various block sizes (I don't recall off-hand
> which commands are used to inquire as to what is supported, but I
> think it's READ BLOCK LIMITS). However, if you do successfully format
> a drive with 1024 or 2048 byte blocks (sectors) don't be surprised if
> your OS freaks out, as it has to be coded explicitly to deal with that
> block size.

Years ago when I got myself for NeXT machine a WREN full height SCSI
drive with a whopping 700 MB, or so, capacity (it still works even it is
not in use right now) I immediately changed its block size to 1024.
This was a popular move on NeXT, and tools for that were readily
available, as a drive after such change was performing better and was
gaining an extra usable capacity and one would not have to worry about
some legacy junk from M$. Quite a bit later I added there a Quantum
drive (it died some year ago). You could try to change on it a sector
size and that command was not failing but also it was not doing anything
useful. A sector size on that drive was hardwired to 512 bytes because
this is what "everybody" is using. Right?

> I'm not sure if linux has any such provisions.

BTW - Linux can mount this WREN if a support for a proper variant of UFS
is present in a kernel so, apparently, it is not fazed by its sector
size.:-)
