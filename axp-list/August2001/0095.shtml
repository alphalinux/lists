>
> Does Linux expect it in the Arc/NT format (historical from the MILO days)?

No, it does not.

> Possible I guess but I'm not familiar with that part of the Linux kernel
> code.

There is more than two possible formats in which hardware clock stores a
time information on Alpha boards. Current versions of 'hwclock' attempt
to guess which one to use and, in most cases, they are doing pretty good
job with that so actually _not using_ flags like '-A' is preferred.

OTOH it is possible that in some situations communications between
hardware, kernel and 'hwclock' are going awry. For that 'hwclock' has a
number of flags, described on its manpage, like '--funky-toy',
'--bad-year', '--epoch' and so on. It is a good idea to actually
look at this documentation in case of problems. I do not know if every
possible conflict can be resolved but one can do at least a credible
effort in this direction.

  Michal

p.s. One of bugs in older startup scripts from Red Hat was that on
Ruffian (UX) they were doing "aha, you are booting with milo so you
should use -A option" regardless of what was specified in
/etc/sysconfig/clock. This resulted in shifting your clock by twenty
years as a format used by Ruffian hardware clock for some reasons is not
ARC. Very annoying; especially during an installation when you did not
have an opportunity yet to apply corrective measures. A time related
information in rpm database was also messed up and made hard to use
in a future system administration.
