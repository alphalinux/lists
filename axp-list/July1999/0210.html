"Alexander L. Belikoff" wrote:
>
> Uh... I'm actually on my way to check the Steel book on that, but
> isn't it the case that the optimizer should just drop certain
> optimizations in case the programmer stands on the way - like in the
> situation above, shouldn't it just make a temp. variable and use it
> instead of a register? Besides, in case I have an optimized version of
> put_f8 in a separate file (with v passed in a register) and I try to
> call it from an unoptimized module - how would the compiler figure out
> the calling convention during the compilation stage?
>

Don't know where that optimization is performed. One possible place
is for the linker to adjust both modules to "fit", assuming there
is argument info in a *.o file. Lot's of my code is in different
source files so maybe that is part of the problem too. Don't know.
I can guarantee I optimize all code at the same level and still
see the problem.

You're welcome to code any way you like. I do my best not to
use code styles that will break my code when I port to other
architectures. This code style can break so I don't use it.
Much easier to make the change than to complain to the compiler
writers and ask them to fix it. Good to let them know the
problem exists though since it does cause headaches/time to
track down.
