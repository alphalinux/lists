> Alexander L. Belikoff wrote:
> >
> > "Wes Bauske" <wsb@paralleldata.com> writes:
> >
> > > Try modifying your code to pass the address of a double rather
> > > than the double itself and see if you still can create the
> > > problem. I've seen problems passing floats/doubles to functions
> > > rather than their addresses. I just pass the address instead.
> >
> > As you may have noticed, I didn't talk about workaround - there are
> > many of them. The one, I'm using now is that I have a temporary
> > volatile double v2 in put_f8() and I copy v into v2 and use the
> > latter. Other possible workarounds include, compiling put_f8() with
> > optimization turned off (it is not time-critical), or using manual
> > 'for' loop, rather than memcpy().
> >
>
> YEP, but actually I'd say this is a feature, not a bug ! If I remember
> correctly written
> somewhere in C standard, a temporary double arg, that will be passed in
> a *register*, as you **will** like it to happen at maximum optimization,
> simply has **NO** address to pass to memcpy. At best, the compiler can
> warn you and automatically generate one of the above workarounds you
> mention (with the SAME drawbacks), I don't think/expect there to exist a
> compiler smart enough to replace your case with one single optimized
> inline register to memory store or whatever. Or ???

Thank you for a very clear explanation. May I hope for a reference in
