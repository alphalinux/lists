Frank Peter Rival USG wrote:

>Nikita Schmidt wrote:
>
>> On Monday, 18 Oct, Barrett G. Lyon wrote:
>> >
>> > SRM is licensed and Compaq wants the money to use it. If SRM was more
of
>> > an open standard life would be grand.
>> >
>>
>> As a standard, SRM is reasonably open - it's described in the Alpha
>> Architecture Reference Manual available from bookstores. What is really
>> bad is the highly restricted availability of the bits which can't be
>> built independently.
>
>Yup. I have been checking internally and someone pointed out to me the
fact
>that, off of the AlphaOEM section of www.digital.com, there is a link to an
>orderable part number for the DP264 SRM SDK.

As I understand it the major benefit of SRM over ARCBIOS is that it
provides a PALcode image, and that milo is required to provide the
PALcode when ARCBIOS is used.

>
>>
>> Take the PALcode, for example. I really don't understand why it's not
>> available. PALcode is basically what other processor manufacturers put
>> into their chips! I know it was a good idea to move it into general
>> purpose RAM - flexibility and stuff, but well, can we the processor
>> users please have it?
>>
>> My understanding is that Digital assumed nobody would need an Alpha CPU
>> without one of Digital-supported operating systems: Open VMS, Digital
>> UNIX, or Windows NT. Their attitude towards firmware and PALcode has
>> made Alpha CPUs actually unusable without one of those OSes - the
>> situation I believe no other processor has ever been in before. They
>> (other processors) can all be usefully programmed in machine codes,
>> using just the architecture specification. The Alpha can't - unless you
>> want to dive into nitty-gritty details of a particular implementation.
>>
>
>How did the milo/aboot folks get everything going? I'm being serious - I
came
>into the Alpha game way after most of that had been sorted out, I believe.
>Either (then) Digital slipped someone a CD and said "shh...don't tell
anyone"
>or someone managed to use the PAL from the SDK. Or something else that I
>don't know about. ??

The milo PALcode looks like it was obtained from the old Evaluation
Board SDK. This is now called the Compaq Alpha Motherboard SDK. I
got version 4.0 last May from Dick Kreischer, (Dick.Kreischer@compaq.com).
It contains modifiable and redistributable source for Debug Monitor
firmware for the 164LX, 164SX, and 264DP. This includes SROM code and
PALcode source. The PALcode is not the same as that used by SRM but it
may be okay for linux.

>
>>
>> I'm not even asking for the full SRM source code (though it would be
>> really great if it could be made available). All I'm asking is that all
>> platform specific code (initialisation, PALcode) be provided with the
>> hardware systems so their users can run whatever they want. There is
>> no logic in holding on to it.
>>
>
>You'd be amazed at what seems like logic depending on who you ask what
>question. I'm trying to track down the responsible people for this
internally
>so we can get a final "yes/no/gimme a week" answer to all this. Rest
assured
>that above all else there are lots of people in Compaq that want what's
best
>for Linux/Alpha - we're just trying to move it along a path that won't get
>us/Compaq in trouble down the road.
>

Getting the low level SDK's for alpha is like pulling teeth. Nobody seems
to know what they are or what is on them. The last two updates I got were
on CDR disks and one had a hand written label. For what it's worth,
I have heard that API may someday release a modified version for their UP
motherboards.
