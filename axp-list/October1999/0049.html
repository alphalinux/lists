I normally do more lurking than posting, but I just had to contribute
my $0.02 to this thread.

Like many on this list, my disappointment with RatHed grows with each
release, since it seems that more new bugs are introduced each time than
are fixed. As others have said many times on this list, it's clear that
RatHed does little if any testing of their alpha release. I once sent
email to one of their folks, begging them to start some kind of formal
beta testing program, but, if they ever did it, it's not working.

As a biologist, I tend to see everything in evolutionary terms. Open
source software in general, and Linux in particular, is a self-organizing
and evolving system. The problem with RatHed for the alpha is a lack of
selection pressure. We need a distribution maintained by people who
regularly use and depend on alphas, and who respond to bug reports and
rapidly incorporate contributed fixes. From reading Ron Farrer's recent
postings, and from perusing the debian website, debian is looking more and
more like the most promising (although not perfect) alternative. I
suppose I shouldn't praise debian too much yet, since I've not yet tried
it, but I will be doing so in the next few weeks.

I have recently been experimenting with the alpha port of FreeBSD. FreeBSD
has a lot going for it, including much better virtual memory algorithms
than Linux, an NFS implementation that reputedly is second only to Sun's,
and a really well-thought-out and relatively painless upgrade system based
entirely on make files and a central source code repository (on the net)
under CVS control. In essence, whenever you want to upgrade your system,
you type "make update", and the system fetches only the *changes* to the
source code since your last update, and recompiles your system. There's
never any concern about incompatible shared object libraries, etc., and
the source code is easily available for when you need to fix or tweak
something. As with Linux, FreeBSD development takes place along 2 tracks,
a stable and an experimental. If you have the stable system, doing a
"make update" really just gets you the latest bug fixes, whereas in an
experimental system it gets you the latest bleeding edge new features.
As much as I like this approach, I probably won't end up switching to it,
since my number crunching needs really benefit from cpml, and I doubt that
will be available for FreeBSD. Another problem is that FreeBSD on the
alpha, at least for now, also seems to have insufficient "selection
pressure", just because the community of users is so small.

I only mention it as an example of what's possible in the way of
"self-organization". It would be nice if a consensus could emerge from
our group about what distribution is most responsive to the needs of alpha
users. Maybe that's too much to hope for from such a diverse group, but,
if it did happen, and if Compaq threw their support in that direction, we
might be able spend less time on the kinds of problems that now fill this
list, and more on the work for which we originally bought our alphas.

JMHO,
Jack Wathey
