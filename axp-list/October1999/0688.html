Warning: quibbles ahead! You have been warned.

> I
> suppose the most common approach on the front end is to setup your servers
> and use Round Robin DNS.

No. NAT is much more common, and produces much more reliable systems, too.

> If you're serving mostly static content (rootshell.com, gnome.org) I don't
> think Alpha would give a noticable gain over a Dual PII/450.

If you're serving static content, then the cost of your bandwidth swamps the
cost of the computer. Have you ever priced a megabit of connection to the
Internet?

> But
> if you're
> running some sort of dynamic web site (Google, Amazon, Lycos, mp3.com)
> wouldn't you benefit from a machine that has more cache, better FPU, and
> 64-bit virtual memory space?

Maybe. None of the above sites uses floating point. mp3.com serves mostly
static content. Google, Amazon, and Lycos are large database apps, and
building large databases is a bit of voodoo that defies generalizations
about "more cache" and "64 bit VM space".

> The back end is where "clustering" comes in..

No, the front end is a cluster, too.

> Running applications like
> Oracle and Sybase.

Which, for problems without locality (most databases), run best on big SMP
boxes. Quite frankly, if IBM could only turn their grungy mainframe OSes
into something reasonably easy to administer, they would own the big
"database appliance" market, because their stuff is actually cost effective
(!)

> Is it just me or is this where Alpha really
> shines over
> Intel?

The Alpha's database benchmark numbers aren't exactly crushing the
competition. Of course, those benchmarks have some systematic problems, but
even if you correct them, I don't think you can say the Alpha is the clear
winner.

> Of course, many are still buying Sun Enterprise 10000 clusters
> (except for ebay)

I thought eBay had a Sun Enterprise somthing-or-other, and is about to
implement a HA cluster?
