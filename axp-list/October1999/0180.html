On Sun, 10 Oct 1999, Michal Jaegermann wrote:

> > >BTW - the current version of Gnome, although not totally problem free,
> >
> > Are there any Gnome problems that need fixing,
>
> Yes, definitely. :-)
>
> > Alpha-specific or otherwise?
>
> I do not know about "otherwise" part as I was hacking Gnome only
> on Alpha. Recently I submitted an "official" bug report
> (after a font change from a theme selector in CC the whole X locks up;
> a power switch time if you cannot telnet in).

This is a bug in either the X font server (xfs) or egcs which manifests
itself when xfs is compiled with optimization. As a result of the xfs
crash, the X server locks up.

The latest XFree86 packages in Rawhide build xfs without optimization on
the Alpha.

> Recently I was exchanging some e-mail with Miguel de Icaza and I
> mailed him the whole list of problems of varying degrees of severity.
> For example - I can crash an address book from gnome-pim on demand.

> He asked me to submit all these "officialy". Maybe one day if I will
> find some time. This alpha box is not on-line and this is not that
> easy as it sounds.

Forgive me if I can't sympathize - I don't experience any obstacles myself
in sending an e-mail to the bug system. :)

If you have suggestions on making it easier to report bugs, please bring
them up on gnome-list. The gnome hackers want to receive as many bug
reports as there are bugs...

> The other problem which I did not mention anywhere
> yet: docs claim that Help Browser should be able to retrieve documents
> with URLs like http://somewhere.net/.... I am getting **always** that
> such document cannot be found. I did not investigate that closer yet.

If those docs exist they are inaccurate and need to be fixed. The XmHTML
widget currently used by the help browser can't handle most of the HTML
out there, so HTTP was turned off.

A better HTML widget (hopefully Mozilla) will be in place for GNOME 2.0.

> Help in applications is using "default" URL handler which tries to start
> 'scrape and nothing else (or at least it is not clear how to modify a
> behaviour of gnome-moz-remote). This is dumb, to put it mildly.

I fixed this before GNOME 1.0.40 (see
http://bugs.gnome.org/db/14/149.html).

> On the top of it 'gmc' tries to retrieve broken URLs for help.

Need more info. If it's saying "can't find help file" it may be because
you don't have the user's guide installed.

> In general various headears are skipped

> and one does not get prototypes for various string functions
> ('memcpy') or 'alloca' and things like that.

I've not seen these. gnome-libs and gnome-core compile pretty much
warning-free with -Wall. There are also some warnings that are correctly
left unfixed (e.g. in the code generated by orbit-idl, "uninitialized
variable" stuff).

> Even with that fixed there is way too much casts between ints and
> pointers.

> Most of these are probably just lazy programming and harmless ("lets
> store this int value in a pointer and retrieve that later") but a
> noise they create makes hard to find real bugs of that sort.

I haven't seen any 64-bitness problems in Gnome itself. Storing ints in
pointers is done using a specific set of macros that avoid warnings, and
storing pointers in ints gets fixed quickly, seeing as at least a few of
the developers (e.g. panel author) use an Alpha.

Please make sure you're using the latest Gnome stuff from ftp.gnome.org -
at least one of the problems you mention has long been fixed, so odds are
the other fixes have been made as well.
