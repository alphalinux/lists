lucier@math.purdue.edu said:
> In my (limited) experience, this type of warning almost always
> indicates that someone is (a) trying to store a pointer in an int
> variable without (b) knowing that an int is 32 bits and a pointer is
> 64 bits on alpha linux.

IIRC, this is true to some extent, but not always. For example, one could
imagine a struct like so:
struct _foo {
  void *stuff;
};
which you would use to pass various data around. If you do things like:
  struct _foo f;
  int bar;
  f.stuff = bar; /* Get a warning */
  ...
  bar = f.stuff; /* Get a warning */
you should be ok. I'm not saying this is good programming practice BTW ;-)

lucier@math.purdue.edu said:
> Unfortunately, the build of glibc-2.1.2 got hosed because of clock
> skew (of about 20 years, I believe ;-), and another rpm didn't build
> to completion.

Didn't it get hosed because you have a recent 2.2 kernel tree installed ? The
math stuff in recent (not sure from which version) kernel causes problems when
compiling glibc-2.1.x... Also, I'm not sure stock gcc-2.95.x can deal with
the complex numbers from glibc-2.1.x, it needs a patch IIRC.

Anyway, I'm running on an alpha on which I compiled and installed a lot of
SRPMS from 6.x, and I don't see that many problems... ;-)
