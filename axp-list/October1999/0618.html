Greg Lindahl wrote:
> > Put a cluster of those
> > together and you'd
> > have a kick ass cluster! On the x86 side the closest thing is a
> > Xeon 2MB Cache
> > CPU which is $2k just for the CPU...so you lose out on
> > price/performance versus
> > the Alpha.
>
> No. Many web serving tasks are embarrassingly parallel, so you could go with
> clustered dual x86 boxes and the relatively cheap non-Xeon cpus, which would
> probaby be about twice as cost effective.

That's true for simple web sites... just about any modern hardware can
saturate a network interface with static requests.

If a web resource is compute-intensive, you might benefit from the Alpha
to shorten response times. Or if you are handling massive amounts of
data (e.g. large search engines) the 64-bit virtual memory space could
be a benefit. Otherwise you might as well stick with ordinary x86
hardware (as impressive as the DS10 may be).

Response times are critical for web applications. For several of the
sites I manage, we use a large memory cache of database results to avoid
reissuing expensive SQL commands on each page. The hash index can scale
up to several GB if necessary... in that realm we'd certainly be
considering Alpha to get us past the 32-bit barrier.

Web servers scale up using SMP and/or a server farm, connected behind a
load-balancing router. That's the norm for any platform.

> The Alpha's major benefit today is for floating point, and tasks which
> benefit a lot from high single CPU performance.

Does the Alpha still win on memory bandwidth too?
