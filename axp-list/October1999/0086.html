>The EBSDK CD does contain SRM binaries, but not SRM code. It contains
>source of "sample" PALcode, called EBSDK PALcode, which is NOT the
>same as OSF PALcode, and it is this source PALcode that is used in
>MILO.

It would be nice to use SRM for everything and not NEED MILO at all except
for older systems with no other options. The UP1000 and UP2000 do not have
an SRM either, it may be a good idea to provide one so these new systems
are not alienated by the death of AlphaBIOS. Just a thought.

>> By the way, I've seen conflicting documentation on what SRM stands for, is
>> it: System Resource Monitor or System Reference Manual? The DEC and Compaq
>> websites have both listed, which is a fellow to believe?
>
>System Reference Manual.

How authoritative is this? Is this the "modern" definition? I just when
to the Compaq Linux website and found the other definition (it was mere
months old and recently updated). I'm still confused as to believe under
the contradictory circumstances.

>The console provides an interface which follows the spec present in
>that manual. Note that most of it is published as the Alpha
>Architecture Reference Manual, but there are appendices in the SRM
>that are company confidential because they contain information about
>specific implementations.

Out of curiousity, what effect will any of this having upon releasing SRM
to the greater market? I have heard rumors that AMD has decided not to use
Slot B, mayhaps this is the reason console firmware is not going to be
released to the clone markets? Do you realize the effect this will have on
the once hoped for commoditization of the Alpha over mass produced,
platform generic, inexpensive motherboards (from multiple sources no less)!?

My hope is diminishing, FAST.
