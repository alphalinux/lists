On Wed, Nov 14, 2001 at 09:48:33AM +0800, Davies, Huw wrote:
> > From: Brad Stockdale [mailto:bradsprt@greenepa.net]
>
> > I'm working on installing Red Hat 7.1 on one of my DEC
> > UDB's (166MHz,
> > 160 megs of RAM, 4 gig external SCSI drive, etc etc)...
> >
> > I am however running into some problems...
>
> As far as I understand, problems with the SRM PALcode mean that it's not
> possible to boot a "standard" kernel on a UDB. I spent some time working on
> this and got an e-mail from Philip Ezolt which read in part:
>
> "This is the result of a buggy SRM, and unfortunately, it isn't fixed in the
> final digital SRM release for the UDB.
>
> When entering the trap handler for an illegal instruction, the pc is not
> advanced properly. When the kernel hits a floating point error which is
> normally handled in the kernel, the pc never advances.
>
> The problem is handled, but since the pc isn't advanced, the program
> rexecutes the same instruction, and BAM, trap. This repeats infinitely.
>
> Workaround: Boot from MILO. It has the PALcode fixed."

Well, that *used* to be true; current 2.4.x kernels have the appropriate
handling that makes up for this probelm with the last SRM for UDB. You
will see a message during the bootup that indicates whether the kernel
detected the problem and is going to work around it.
