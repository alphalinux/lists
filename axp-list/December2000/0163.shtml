Greg Lindahl wrote:
>
> > "ML values are represented as 32-bit quantities, so all code and data
> > has to live in the first 2^32 bytes of address space, or things break.
> > The run-time system is actually designed to support a 64-bit
> > implementation, but the compiler needs substantial reworking to do
> > so."
>
> This is exactly what -taso fixes. [...]

OK, I've linked my program with gcc -Wl,-taso ... and yet I'm seeing
addresses larger than 2^32 with gdb:

(gdb) n
186 bibop[start++] = aid;
(gdb) p bibop
$4 = 0x20000010000
(gdb) p start
$5 = 33554433
(gdb) p bibop[start]
Cannot access memory at address 0x20004010002.
(gdb) n
 
Program received signal SIGSEGV, Segmentation fault.
0x1201dda0 in MarkRegion (bibop=0x20000010000, baseAddr=0x20000010000,
    szB=655360, aid=0) at gc-util.c:186
186 bibop[start++] = aid;

bibop is declared as a pointer to data.

Unsigned16_t *bibop;

Anyone knows why bibop isn't pointing to the lower 32 bits if I have
linked with -taso?
