Too many misconceptions to let go by:

Building all the software to be big endian is insufficient,
I believe. There may be hardware assumptions make by systems
designers as well (motherboard hard-wired to be little-endian)

But that's _completely_ irrelevant in this particular case.
If you have big-endian data you want to use on a little-endian
machine, you do NOT have to do byte swapping on every single
access, unless you are actually trying to run a big-endian
program (for example, binary translated SPARC or mips executables)
Even then, there is a theoretical trick (called "the Martin maneuver"
after the guy who thought it up) to avoid a bunch of swapping.

All you really have to do is to swap the data at file I/O time
if you want to preserve the endian-ness of the data.
Some compilers (the compaq fortran compiler) have switches which
will make this behavior happen transparently. The switch for
that compiler is '-convert big_endian', and it has other conversion
switches for cray data as well.

If the problem is that there are big-endian assumptions in the
source code, then those would need to be corrected as well.
(C code knowing to access byte 3 instead of byte 0 of an int, etc)
