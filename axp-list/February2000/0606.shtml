> > 2. handling pre-fetch code as produced by CCC

> The kernel itself will handle this. Added quite some time ago.

For those interested in the details:

1. Richard's patches are in both 2.2.14 and 2.3.48 and
   later (and almost certainly in some earlier versions as well)
   arch/alpha/mm/fault.c, so using any of those with
   CCC-generated "pre-fetch" instructions should just work, albeit with
   a performance hit on those machines without OSF (SRM) PALcode. The
   appropriate PALcode for handling pre-fetch is present in EV5 and EV6
   OSF PALcode, and most likely in at least some of the EV4s as well.

2. the pre-fetch code is all about attempting to pre-load a TLB entry
   before it actually is needed, so that when it *is* needed, it'll just
   be there. About the slickest code-path in the PALcode is the TLB miss
   code; the pre-fetch instruction allows that code to know it's doing a
   pre-fetch and to (potentially) bail-out of the full processing if it
   looks like it would take too long (as in actually having to walk
   page tables, for example). And by bail-out I mean dismissing the TLB miss
   without delivering it to the kernel as a page fault, which is what would
   happen without the knowledgable PALcode, and where the performance hit
   is incurred.
