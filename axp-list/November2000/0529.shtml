Christian Iseli wrote:
>
> wsb@paralleldata.com said:
> > Check their updates page for info. What comes on the CD you buy is
> > broken.
>
> Ok. I didn't buy a CD. I just grabbed the src.rpm from a local ftp site and
> compiled it.
>
> <Warning "rambling mode on" "and it's getting late here">
>
> I'm under the impression that when you install a new release, you get what you
> asked for (or deserve, depending how you see it).
>

I understand why you might think that but I suspect you
don't run many systems. I pay for a distribution because
I expect them to sort out what works together to save
my time having to do it. That's what I figure I "bought".

> Usually, you have spent quite some time setting up your systems, and you know
> their quirks, and how they run. Just installing a new release is like opening
> a whole new can of worms. I try to avoid it.
>

Yes but I expect those worms to be sorted out
when I spend my $$.

> All my systems still display "RedHat release 5.2" (or even 5.0), but have
> evolved over time, when there was some need. They now tend to run a 2.4.0
> kernel, with glibc 2.2, for example.
>
> I think you are better off, in the long run, to reuse your current setup, and
> update the necessary pieces. This way, it won't be such a black box.

Well, the problem with just updating on your own is
first compatiblity between release levels along with
the time to mess with it. I don't consider what RH
install a black box. I actually try to look at things
they installed to make sure they don't do something
stupid. I consider xinetd really stupid, but then I
know I'm used to the way inetd works and that inetd
is pretty standard for the other systems I use.
(AIX/Solaris/etc) Being different than the other
Unices is not a good plan if you ask me, particularly
if you are interested in the server market. Maybe
RH doesn't give a damn about servers.

Key places to look for me are /etc/inittab, /etc/rc.d,
any /etc/*conf* files, what the heck cron is up to, and
look for unexpected socket/services activity.

>
> Setting up a new system is not a problem either: just boot it using some CD
> that came with one of those colorful Linux magazine, and do a minimal install
> in a small partition that will become the swap partition of the new system.
> Then boot this minimal setup, copy "your" current setup from another of your
> machines, edit some /etc files, and you are all set, with no nasty surprises.

I don't install CD's on my systems. They are a wasted
resource. I boot a floppy and network install. Eventually
I expect to have things do bootp but haven't got that
sorted out yet. Alpha SRM does that and I've actually
tested it but not to the point that I can get a distribution
to install that way. Intel/x86 is behind with respect to
network booting but it seems some new systems have a
version of network bootable-ness.

>
> I guess I'm a bit old-fashioned, but that's the way I like it. Needless to
> say, I tend to want to compile all my stuff from source (and tweak the SRPMS
> too, of course :)
>

I think you just like building things yourself.

I've got much more critical applications SW to worry
about without having to also worry what RH did to my
systems.

For example, the x86 box I just installed currently
has 320GB of disk and 2 tape libraries attached to
it for SW testing. Now I have to go back and install
something else. Downloaded Debian today and will
burn CD's tonight. If that goes well, I'll be using
Debian from now on.
