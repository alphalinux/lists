In my experience, similar problems may occur with all compilers
as soon as the code gets too complicated (holds true for any DBMS) -
I have similar effects ("core dumps and/or execution taking wrong paths)
on Tru64 (with Compaq's compiler) and on AIX (with IBM's).

All you can do is keep track of the options you use in building
and the results of your tests, and maybe have a separate "noopt"
build version in parallel to do regression tests.

My most disturbing experience was that in one case on Tru64,
a module that worked fine with "-O1" suddenly misbehaved
when it got trace output (in "#ifdef" sections) activated, and
I had to switch to "-O0" ("minus oh zero") for the trace case.

So upgrading the compiler might help, but even the most rigorous
testing with one compiler (machine, OS version, compiler version)
will not guarantee it still works after one of these variables
(or some other ...) was changed.
