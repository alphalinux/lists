It seems that the MAGIC value 0xc00fc00f is being sign extended as a
negative number. Changing the printf for the working case and printing out
longs instead of ints shows;

GOOD baseAddr ffffffffc00fc00f == ffffffffc00fc00f

Even casting MAGIC to an unsigned int doesn't make it a postive 64-bit
number. Declaring it as 'const unsigned int' does though.

So gcc is clearing the top 4-bytes of the return value in fn.c and so the
main code compares 0x00000000c00fc00f with 0xffffffffc00fc00f in the if
statement.

Is there a way to define the MAGIC value unsigned ? 0xc00fc00fU doesn't seem
to have any effect.

Addy.

> MAGIC is an unsigned int (32bits) with value 0xc00fc00f. The compiler

> "sign extending" an unsigned int (there is no sign bit or it's
> "inferred" to be zero).

> If MAGIC was defined as int 0xc00fc00f, then the register value
> should be 0xffffffffc00fc00f. int 0xc00fc00f is a negative number.
