Ok, I'll try to tell the whole story...
(At least as far as I have understood the pieces).
a) Every system has some kind of hardware-clock, usually referred to as RTC.
b) The RTC stores the year in one byte and thus needs an extra offset to
   get to 19../20.. On x86 this offset is hardwired to 1900, on alpha this
   offset varies with the os used, some start at 1952, others at 1960/1980...
   but you can't read this offset (officially referred to as "epoch"),
   you can only read the byte for the year.
c) The kernel tries to "guess" on bootup, which of the possible offsets was
   used and thus tries to calculate the current year. Since 2000 it seems
   the kernel initially gets it wrong in some cases. I guess some versions
   now use 1900 as base, but the kernel thinks 1952 is the earliest possible
   epoch.
d) The system-date is set by the kernel by what it read/guessed from the RTC
e) You can change the system date via the "date" command.
f) You can write this changed system-date to the hardware-clock (s.a.)
   After this, the year might appear wrong in bios/console, but if you
   are running linux-only, you will not need to care about this, and the
   kernel will guess the date right on next bootup.

Please correct me where I'm wrong, but I think I'm rather close ;)
