On Tue, 13 Mar 2001 jnix@medi.physik.uni-oldenburg.de wrote:

> The stack size limit seems to be set in
> /etc/security/limits.conf (via PAM). Note that limits are
> inherited from parent processes, i.e. if you restart a
> server process via ssh, you inherit the limit from the sshd process.

Another way to set it is via /etc/initscript as I learned:

#
# initscript Executed by init(8) for every program it
# wants to spawn like this:
#
# /bin/sh /etc/initscript <id> <level> <action> <process>
#

# Increase hard stack size limit (to correct the setting in the Linux/Alpha
# kernel 2.2.14, see /usr/src/linux/include/asm-alpha/resource.h,
# #define INIT_RLIMITS)
ulimit -Hs unlimited

# Execute the program.
eval exec "$4"

> I once had problems with this in combination with XEmacs and
> large regular expressions (already reported to SuSE). The
> XEmacs maintainer commented that
> stack size limits for 64-bit systems seem ridiculous small.
I don't know why, but on Tru64 it's even smaller with 4 MB :-(
The 8 MB stack size limit is indeed set by the kernel, cf.
/usr/src/linux/include/asm-alpha/resource.h. I have no idea why the
alpha kernel developers would want to have such a small stack size.
Compaq's Fortran compiler places F90 automatic arrays on the run-time
stack, so having 8 MB of stack space is a little bit ridiculous...
