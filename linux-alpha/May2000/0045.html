Applications typically handle this by having one
thread act as a dedicated signal handler by having
all other threads mask out all possible signals.

                        -Kip

> | Sending sigterm to a signal handler can cause pthread_cond_wait to
> | return.
>
> as expected. I don't have docs handy (to quote from, so take as many
> grains of salt :), but when a new thread is created it can receive
> any signal that the creating process/thread can. Since a signal will
> only be delivered once, it can only be handled as many times (1).
> Unless you _explicitly_ tell a thread _not_ to handle a signal using
> pthread_sigmask(...), any thread in your process may receive it --
> maybe the one you don't expect :)
>
> I recommend you grab the pthreads book from o'reilly for more info
> on how pthreads and signals interact.
